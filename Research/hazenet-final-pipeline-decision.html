1.1 ★ ช่องว่างที่ต้องซื่อสัตย์ — "ที่สร้างจริง" ยังไม่ตรงกับ "vision เดิม"

เรื่องที่รอบแรกผมประเมินผิด (จากการอ่านโค้ด model_operator.py + build_datacube_m2.py): CLNO ที่รันจริงคือ
PM2.5s(t) = Σg Ksg(mett) · Eg(t) + bs(mett)map "วันเดียวกัน": met+ไฟ ของวัน t → PM2.5 ของวัน t · target = PM2.5 รายสถานี รายวัน · แต่ละวันเป็นตัวอย่างอิสระ
แปลว่ามันคือ เครื่องประเมินแบบ same-day source-receptor / nowcast + attribution — ยังไม่มี 3 อย่างที่ vision เดิม (focused-plan / overview / research-contribution) เคลม:
vision เดิมเคลมโค้ดจริงมีไหมผลต่อการเคลม
Semi-Lagrangian advection (𝒜) แยกชัดไม่มี — transport ฝังหมดใน learned kernel Kไม่มี physics hard-constraint การพัดพา
Autoregressive rollout t→t+1→…→t+48ไม่มี — single-step วันเดียวยังพยากรณ์ล่วงหน้า 24–48h ไม่ได้จริง
Operator-splitting Ct+1=𝒜+ℱθไม่มี — ไม่รับ Ct เป็น inputไม่ใช่ temporal model
Attribution + inversion เชิงเส้นต่อ Eมี ✓นี่คือแกนที่แข็งจริง
ค้นพบนี้เปลี่ยนกรอบการส่งมอบ: สิ่งที่แข็งและพิสูจน์ได้จริงตอนนี้คือ "attribution-native source apportionment" (ไม่ต้องพึ่งการพยากรณ์ล่วงหน้าก็ทำงานได้) ส่วน "forecast 24–48h" เป็นแกนคนละแกนที่ต้องตัดสินใจแยก (ดู D10–D11) — ไม่ใช่สิ่งที่ "เกือบเสร็จ" อย่างที่รอบแรกผมเขียน

2. บทเรียนชี้ขาด — ทำไม "แผนเดิม" ต้องเปลี่ยนทิศ

แผนเก่า (master-execution-plan, best-version) สมมุติว่า คอขวด = expressivity ของโมเดล → จึงเน้น architecture (AFNO, FNO, ablation E2–E5) · แต่ M2 round 2 รันจริงแล้วพบ 3 ข้อค้นพบที่หักล้างสมมุติฐานนั้น:

ค้นพบ 1 — overfitting / ขนาดโมเดล "ไม่ใช่" ปัญหาหลัก

ลดพารามิเตอร์ 24M → 0.4M (LowRank → GlobalV) ผล test MAE แทบไม่ขยับ (78 → 72) · เพิ่ม regularization/dropout ก็ขยับน้อย

ถ้าเพิ่ม/ลดความจุของเน็ตแล้วผลไม่เปลี่ยน แปลว่า error ที่เหลือ ไม่ได้มาจากเน็ตเรียนไม่พอ — มันมาจาก "ข้อมูลเข้า/โครงปัญหา" · การไปไล่ทำ AFNO ใหญ่ ๆ หรือ backbone หรู ๆ จะเสียเวลาฟรี

ค้นพบ 2 — 2023 (ปีวิกฤต) ไม่ใช่ตัวที่ generalize แย่ที่สุด

LOYO CV: ทดสอบ 2023 bias แค่ −2 (เพราะ train เห็นปีฝุ่นหนักช่วยดึง baseline) · ตัวที่พังจริงคือ ปีต่ำ (2021, 2022) โมเดล over-predict +45~50 µg/m³

เหมือนนักเรียนที่ติว "ปีฝุ่นหนัก" มาเยอะ → เจอปีฝุ่นเบากลับ "ทายสูงเกิน" ทุกครั้ง เพราะมันคิดว่าไฟเท่านี้ต้องได้ฝุ่นเยอะแบบที่เคยเห็น · ปัญหาไม่ใช่ "ปีสุดโต่ง" แต่คือ "ความสัมพันธ์ไฟ→ฝุ่นเลื่อนไปมาในแต่ละปี"

ค้นพบ 3 (แกนกลาง) — FRP→PM2.5 เป็น non-stationary ข้ามปี

โมเดล linear-in-emission จับ "ระดับฝุ่นรวมต่อปี" ไม่ได้ · ปีเดียวกัน FRP เท่ากันแต่ PM2.5 ต่างกันมาก เพราะ การระบายอากาศ (ventilation), ความสูงชั้นผสม, ฝน, ชนิดเชื้อเพลิง, ความชื้นดิน เปลี่ยนทุกปี

ข้อสรุปที่เปลี่ยนทุกอย่าง: คอขวดของ HazeNet ไม่ใช่ "เน็ตไม่ฉลาดพอ" แต่คือ (ก) emission proxy (FRP ดิบ) แปลงเป็นการปล่อยจริงได้แย่ และ (ข) ฟิสิกส์การสะสม/ระบายที่ทำให้ไฟเท่ากันได้ฝุ่นไม่เท่ากัน ยังขาด · ดังนั้น งานหลักของ pipeline สุดท้ายต้องอยู่ที่ "ฝั่งซ้ายของสมการ" (อินพุต+ฟิสิกส์) ไม่ใช่ "กล่องเน็ตตรงกลาง"

3. การตัดสินใจหลัก — เก็บ / ทิ้ง / เลื่อน

ทุกตัวเลือกที่เคยคุยกันมา ตัดสินตรงนี้ พร้อมเหตุผลอิงผลทดลอง:

D1
แกนสถาปัตยกรรม = CLNO (conditionally-linear)
เก็บ — ล็อก
เก็บถาวร. CLNO คือสิ่งเดียวที่ให้ 3 พลังจากโมเดลเดียว: forward = พยากรณ์, อ่านแต่ละสาขา = attribution, ย้อน gradient = inversion · และค้นพบ 1 พิสูจน์แล้วว่าเราไม่ได้เสีย accuracy จากการบังคับ linear-in-E (เพราะคอขวดอยู่ที่อื่น)
ถ้าทิ้ง CLNO ไปใช้ black-box (AFNO/GNN ล้วน) เราจะ เสียทั้ง attribution+inversion (= ตัวตนของงาน) โดยที่ accuracy ไม่ได้ดีขึ้น เพราะคอขวดไม่ใช่เน็ต · เป็นการแลกที่ขาดทุนสองต่อ
D2
เป้าหมายงานหลัก = แก้ emission + non-stationarity
เก็บ — เป็นแกน
นี่คืองาน 60% ของ pipeline สุดท้าย. ย้ายแรงจาก "architecture search" (แผนเดิม E2–E5) ไปที่ emission curve + per-year/regime scaling + ventilation physics (BLH) · รายละเอียดเต็มใน §5
อิงค้นพบ 3 โดยตรง · เป็นที่เดียวที่ลงแรงแล้วจะขยับ test MAE ได้จริง
D3
Uncertainty = quantile heads (P10/P50/P90)
เก็บ — แทน ensemble
เก็บ แต่เลือกแบบประหยัด. หัว output คาย 3 quantile พร้อมกัน · ทิ้ง diffusion ensemble 50-member (v2 critique #4: ช้า 50× + ขัด superposition + ทั้ง 50 เส้นอาจ under-predict พร้อมกัน)
ได้แถบความไม่แน่นอนที่เตือนภัยได้ โดยไม่ทุบงบ A100 และไม่ทำลายคุณสมบัติเชิงเส้นที่ใช้ทำ attribution
D4
AOD ดาวเทียม (Himawari/MAIAC) เป็นอินพุตหลัก
เลื่อน → ใช้เป็น "oracle/ตัวปรับ"
เลื่อนออกจาก critical path. ทั้ง 2 บทวิจารณ์ชี้ว่า AOD "ตาบอด" วันวิกฤต (เมฆ/ควันหนา) · แทนที่จะพึ่งมันเป็นอินพุตหลัก เราใช้ AOD เป็น (ก) ตัว constrain emission (E_oracle ใน error budget) และ (ข) target เสริมแบบ masked เฉพาะวันที่มีค่า
หลีกเลี่ยง garbage-in วันวิกฤต · ได้ประโยชน์จาก AOD โดยไม่ผูกชะตาโมเดลกับมัน
D5
เคลม attribution = "first-order primary transport influence"
เก็บ — ปรับถ้อยคำ
เก็บความสามารถ แต่ลด overclaim. ตามทั้ง 2 บทวิจารณ์ — ไม่เคลม "ระบุแหล่งเคมีสมบูรณ์" (secondary aerosol 30–40% เป็น nonlinear) · เคลมเป็น อิทธิพลการพัดพาฝุ่นปฐมภูมิจากการเผา ซึ่งพอดีกับโจทย์รัฐบาล (เป้าคือจัดการ "การเผาชีวมวล")
D6
CTM digital-twin (CMAQ/WRF-Chem) สำหรับ validate attribution
เลื่อน → เวอร์ชันฝัน
ไม่อยู่ใน 8 สัปดาห์. รัน CMAQ/WRF-Chem เองคือโปรเจกต์แยกหลายเดือน + ต้องมี emission inventory · เวอร์ชัน 2 เดือนใช้ synthetic recovery (leave-source-out) + เคสจริงเชิงคุณภาพ (ทิศลม ERA5) แทน · twin เป็นเฟส 2 ของ best-version-plan
D7
Operational GFS eval (train/serve skew)
เลื่อน → ระบุเป็น limitation
ไม่รันใน 8 สัปดาห์ แต่ "พูดถึงตรง ๆ". ดึง GFS archive 10 ปีคือฝันร้าย data-eng (หลาย TB) · เวอร์ชันนี้ train+eval บน reanalysis (ERA5/Open-Meteo) แล้ว เขียน limitation ชัดว่ายังไม่ทดสอบ operational + วาง noise-injection ablation เล็ก ๆ เป็น proxy ถ้าเวลาพอ
D8
Inversion (emission ย้อนรอย)
เก็บแบบ PoC — stretch
เก็บเป็น proof-of-concept. ผล M2 inversion r≈−0.04 (ill-posed: สถานีน้อย, G ใหญ่) · เวอร์ชันนี้โชว์ inversion บน เคสไฟจุดเดียว/สังเคราะห์ที่รู้คำตอบ + regularization · ไม่เคลม inversion ทั้งโดเมน (อันนั้นเวอร์ชันฝัน WS5)
D9
โดเมน/สเกล = คง M2 (SEA 111×101)
เก็บ — เป็น canonical
คง M2 เป็นสเกลมาตรฐาน. ครอบ ไทย+พม่า+ลาว ซึ่งจำเป็นต่อ transboundary attribution · Phase 0 (กล่องเชียงใหม่) เก็บไว้เป็น "unit test เร็ว" สำหรับ debug · ไม่ขยาย ไปทั้ง SEA 5–25°N ในเวอร์ชันนี้ (นั่นคือ benchmark dataset ของเวอร์ชันฝัน)

5. แก้ที่แกนจริง — emission + non-stationarity + ventilation

นี่คือหัวใจของ pipeline สุดท้าย (งาน 60%) · ทั้ง 3 ข้อโจมตี "ค้นพบ 3" จากคนละมุม:

5.1 FRP → emission curve (ไม่ใช้ FRP ดิบ)

FRP (Fire Radiative Power) เป็น proxy ที่ noisy: ดาวเทียมผ่านวันละไม่กี่ครั้ง, เมฆบัง, และความสัมพันธ์ FRP→มวล PM ที่ปล่อยจริง ไม่เป็นเชิงเส้นและอิ่มตัว

ทำไม log/saturating: ฟลักซ์การปล่อย PM ต่อ FRP มีลักษณะ saturate (ไฟใหญ่มากปล่อยต่อหน่วย FRP น้อยลง, plume สูงขึ้นพ้นชั้นผสม) · เราแทน E = emission_factor · FRP ดิบ ด้วย
E = α · log(1 + FRP/FRP₀)  หรือ  E = α · FRP / (1 + FRP/F_sat)α, FRP₀, F_sat = พารามิเตอร์เรียนได้ (fit ร่วมกับโมเดล) · ลดอิทธิพลของ FRP spike สุดโต่งที่ทำให้ over-predict
ค้นพบ 2 บอกว่าโมเดล over-predict ตอนไฟเยอะ · curve ที่อิ่มตัวจะ "เบรก" การแปลง FRP→ฝุ่นในช่วงไฟหนัก → ลด over-prediction โดยตรง

5.2 จัดการ non-stationarity ระดับปี (แกนที่ยากสุด)

ปัญหา: ไฟ FRP เท่ากัน ปีต่างกัน ได้ PM2.5 ไม่เท่ากัน · มี 3 ทางเลือก เรียงตามความ "ซื่อสัตย์ทางวิทยาศาสตร์":

ทางเลือกทำยังไงข้อดี / ข้อเสียตัดสิน
(ก) per-year scalarเรียน background/scale ต่อปี b_yearดูดซับ year-bias ตรง ๆ · แต่ "โกง": ปีใหม่ที่ไม่เคยเห็นก็ไม่มี scalarใช้เป็น diagnostic เท่านั้น
(ข) explain ด้วย metให้ ventilation/BLH/ฝน "อธิบาย" ความต่างรายปี ผ่าน kernel conditioninggeneralize ได้จริง · ปีใหม่ใช้ met ของปีนั้น · ตรงเป้า "physics-structured"★ ทางหลัก
(ค) per-year emission factorเรียน emission factor ต่อปี/ภูมิภาค (ชนิดเชื้อเพลิงต่างกัน)มีฐานฟิสิกส์ (เชื้อเพลิงต่างปล่อยต่างกัน) · เสี่ยง overfit ปีทดลองเป็น ablation
การตัดสิน: ทางหลักคือ (ข) — บังคับให้ met (โดยเฉพาะ BLH + ฝน) เป็นตัวอธิบายความต่างรายปี เพราะมันคือฟิสิกส์จริง (ปีที่ชั้นผสมต่ำ/ลมนิ่ง → ฝุ่นสะสม แม้ไฟเท่าเดิม) และ generalize ไปปีใหม่ได้ · ใช้ (ก) เป็น "เพดานบน" (ถ้า per-year scalar ลด error เยอะกว่า met มาก = met ยังขาด driver) และ (ค) เป็น ablation

5.3 เพิ่ม ventilation physics: BLH = boundary layer height

ฝาชีครอบเมือง: ชั้นผสม (boundary layer) คือ "ความสูงของฝาชี" ที่ควันถูกขังอยู่ข้างใน · ฝาต่ำ (กลางคืน/อากาศนิ่ง) = ควันกระจุกตัว ค่าพุ่ง · ฝาสูง (บ่ายแดดจัด) = ควันเจือจาง · ไฟเท่ากันแต่ "ความสูงฝาชี" ต่างกัน → ฝุ่นต่างกันมาก · นี่คือ driver ที่ทำให้ FRP→PM2.5 ไม่นิ่ง
เข้าโมเดลยังไง: ใส่ BLH เป็น met channel ที่ 7 · ในสมการ ADR ความเข้มข้นใกล้ผิว ~ emission / (BLH · ventilation) · เราให้ kernel/background เป็นฟังก์ชันของ BLH: b, K ∝ fθ(u, v, BLH, precip, rh, temp, dem)เมื่อ BLH ต่ำ → background/kernel gain สูง (ฝุ่นสะสม) · เป็น "ventilation gate" ที่ปรับ scale ตามการระบายอากาศ — ยังคง linear-in-E

5.4 เพิ่ม stations พม่า/ลาว (ลด outlier ที่ลาก mean)

ค้นพบ: mean ถูกลากด้วยไม่กี่สถานีที่พังหนัก (median 45 vs mean 74) · ตอนนี้มีสถานีต้นทาง (พม่า/ลาว) แค่ ~4 · หาเพิ่มจาก OpenAQ + แหล่งภูมิภาค → ทั้งช่วย train และทำให้ attribution ฝั่งต้นทางมี ground-truth ทาบ

ลำดับความสำคัญใน §5 (จากผลที่คาด): 5.3 BLH (ขยับ year-bias ได้เยอะสุด, ลงแรงน้อย) → 5.1 emission curve (แก้ over-predict ตอนไฟหนัก) → 5.2(ข) met-explains-year (ตรวจว่าพอไหม) → 5.4 stations → 5.2(ค) ablation

6. โมเดลสุดท้าย — CLNO v3 (สเปกเต็ม)

ต่อยอดจาก CLNOLowRank/GlobalV ที่รันแล้ว · เปลี่ยนเฉพาะที่จำเป็น:

เหตุผลรวมของ v3: ไม่แตะ backbone (เพราะค้นพบ 1) · ของใหม่ทุกชิ้นโจมตี non-stationarity (BLH, emission curve) หรือเพิ่มความสามารถที่ขายได้ (quantile) · ทุกอย่าง คง linear-in-E → attribution/inversion ไม่พัง
attribution & inversion ยังได้ฟรี: เพราะ ŷ = K·E + b ยังเชิงเส้นใน E · attribution = แยก K·E⁽ˢ⁾ ต่อแหล่ง s (พม่า/ลาว/local จาก geographic mask) · inversion = ∂ŷ/∂E ผ่าน autodiff หรือ dual-form lstsq (มีใน model_operator.py แล้ว, อัปเกรดเป็น S×S solve เมื่อ G≫S)

7.2 Error budget — แยก error รวมว่ามาจากไหน (★ จุดเข้มงวด)

7.3 Cross-validation + significance

  • Temporal: train 2019–21 / val 2022 / test 2023 + LOYO (มีใน loyo_m2.py) — รายงาน per-year bias ตรง ๆ
  • Spatial: leave-station-out (กันบางสถานีไว้ test เชิงพื้นที่ล้วน)
  • Significance: block-bootstrap 95% CI + paired test ข้ามวัน-สถานี · รายงาน median per-station ควบคู่ mean เสมอ (ค้นพบ: mean ถูก outlier ลาก)

7.4 Attribution validation (ไม่มี ground-truth จริง → ใช้ทางอ้อม)

วิธีพิสูจน์ในเวอร์ชัน 2 เดือน:
  1. Synthetic recovery: สร้างสถานการณ์ที่เรา "ตั้งคำตอบเอง" (เปิดไฟเฉพาะแหล่ง A) → วัดว่าโมเดลกู้สัดส่วนกลับได้ไหม
  2. Physical plausibility: ทาบ attribution map กับทิศลม ERA5 ในเคสจริง (วันที่ลมพัดจากพม่าชัด → attribution ต้องชี้พม่า)
  3. Leave-source-out consistency: ปิดแหล่ง → PM ปลายทางต้องลดตามสัดส่วนที่ attribution บอก
CTM digital-twin (CMAQ) = เวอร์ชันฝัน (D6)

10. ขอบเขต — เคลมอะไร / ไม่เคลมอะไร

เคลม (defendable):

  • Attribution-native: อ่านสัดส่วน "อิทธิพลการพัดพาฝุ่นปฐมภูมิจากการเผา" (พม่า/ลาว/local) จาก forward model โดยตรง
  • Inversion proof-of-concept บนเคสที่รู้คำตอบ
  • Error budget ที่บอกได้ว่า error มาจากส่วนไหน — ยกระดับความเข้มงวดเหนือ ML-weather ทั่วไป
  • เร็วระดับวินาที/นาที เทียบ CTM ที่ใช้เวลาเป็นชั่วโมง-วัน
ไม่เคลม (ความซื่อสัตย์):
  • ไม่เคลมระบุแหล่ง เคมีทุติยภูมิ (secondary aerosol nonlinear)
  • ไม่เคลม operational skill บนลม GFS (ทดสอบบน reanalysis เท่านั้น)
  • ไม่เคลม inversion ทั้งโดเมน (ill-posed, สถานี sparse)
  • ไม่เคลมชนะ foundation model (Aurora) บน raw RMSE — เราขายความสามารถใหม่ ไม่ใช่เลขต่ำสุด
  • ไม่มี ground-truth attribution จริง → validate ทางอ้อม (เพดานเชิงโครงสร้าง ไม่ใช่ compute)
