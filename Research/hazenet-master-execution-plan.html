B. ปรัชญาการทดลอง — "ทุกเคลม = การทดลองควบคุม 1 ชุด"

กฎ 4 ข้อที่ทุกการทดลองต้องเคารพ (มาตรฐาน A*):

  1. เทียบ baseline เสมอ — ไม่มีตัวเลขไหนยืนเดี่ยว ต้องมี "ดีกว่าอะไร เท่าไร"
  2. เปลี่ยนทีละตัวแปร — ablation ที่ control ทุกอย่างยกเว้นสิ่งที่ทดสอบ
  3. มีนัยสำคัญทางสถิติ — bootstrap CI / paired test ข้ามวัน-สถานี ไม่ใช่ตัวเลขเดียว
  4. ป้องกัน data leakage — split ตามเวลา (test = ปีที่ไม่เคยเห็น) + spatially-blocked / leave-stations-out CV
กฎทองของ split: Train 2016–2022 · Validation 2024 · Test = 2023 + 2025 (ปีวิกฤตที่โมเดลไม่เคยเห็น) · สถานีบางส่วนกันไว้ test เชิงพื้นที่ล้วน

★ E. Error Analysis — แยก error รวมว่ามาจากส่วนไหน

หัวใจของความเข้มงวด: งาน ML-weather ส่วนใหญ่รายงานแค่ "RMSE = X" แต่ไม่บอกว่า X ประกอบจากอะไร — เราจะแตก error รวมเป็นงบ (error budget) เพื่อตอบว่า "ถ้าจะลด error อีก ต้องไปแก้ตรงไหนคุ้มสุด"

E.1 สมการงบ error

εtotal ≈ εwind + εemission + εstructure + εlearning + εrollout + εinteraction ลม ERA5 หยาบ/ผิด · ไฟ FRP พลาด(เมฆบัง)/แปลงปริมาณ · ฟิสิกส์ที่ขาด(เคมี/3D/linearity gap) · โมเดลเรียนไม่สุด/generalization · สะสมตอน rollout · ส่วนปนกัน

E.2 วิธีแยกแต่ละส่วน — Oracle Substitution Protocol

แทนอินพุต/เงื่อนไขด้วย "ค่าในอุดมคติ" ทีละตัว — error ที่ลดลง = contribution ของ error นั้น

ส่วน errororacle ที่ใส่แทนวัดยังไง
ε_windลมความละเอียดสูงกว่า (เช่น ERA5 + downscale หรือ HRES) แทน ERA5 0.25°error(oracle wind) − error(จริง)
ε_emissionemission ที่ปรับด้วย AOD (constrained) แทน FRP ดิบerror(oracle emission) − error(จริง)
ε_structureเปิดพจน์ nonlinear correction เต็ม (ลด linearity gap) + เทียบ 2D vs missing-physicserror(nonlinear) − error(linear) = expressivity; ส่วนที่เหลือหลังทุก oracle = missing physics
ε_learningtrain error vs test errorgeneralization gap = test − train
ε_rollouterror ที่ t+1 (ไม่มีสะสม) vs t+48error(t+48) − error(t+1) ปรับฐาน
ความซื่อสัตย์: ส่วนเหล่านี้ ไม่ได้บวกกันเป๊ะ (มี interaction) — เราจะรายงานเป็น decomposition โดยประมาณ + interaction residual และระบุลำดับการ substitute (เพราะผลขึ้นกับลำดับ → ใช้ Shapley-style averaging ถ้าเวลาพอ)
