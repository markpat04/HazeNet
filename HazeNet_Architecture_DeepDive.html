FRP คืออะไร (ทฤษฎี)

FRP = Fire Radiative Power (เมกะวัตต์) = พลังงานความร้อนที่ไฟแผ่ออกมา ณ ขณะที่ดาวเทียมบินผ่าน ดาวเทียมวัด "ความสว่างอินฟราเรด" ของแต่ละพิกเซลแล้วแปลงเป็นกำลังความร้อน — FRP สูง = ไฟแรง = ปล่อยควันเยอะ

FRP เหมือน "ความสว่างของเตาไฟเมื่อมองจากบนเครื่องบิน" — เตาที่ลุกโชนสว่างกว่า = ร้อนกว่า = ควันเยอะกว่า แต่มันคือ "ภาพถ่าย ณ วินาทีที่บินผ่าน" เท่านั้น (วันละ ~2 ครั้ง) ไม่ใช่การวัดต่อเนื่อง
จุดที่ต้องระวัง (เป็นแกนปัญหาของทั้งโปรเจกต์): FRP เป็นแค่ proxy ของปริมาณควัน — ความสัมพันธ์ "FRP → กิโลกรัมฝุ่นจริง" ไม่ใช่เส้นตรงและไม่คงที่ (ขึ้นกับชนิดเชื้อเพลิง ความชื้น และที่สำคัญคือดาวเทียมมองไม่เห็นตอนเมฆบัง) → เราจะแก้ด้วย "emission curve" ใน Part E1

A2. 💨 Open-Meteo — ลม + ฝน + ความชื้น + อุณหภูมิ download_openmeteo_met_m2.py

วิธีดึงขอเป็น "จุด" บนกริดหยาบ 1° = 12 (lat) × 11 (lon) = 132 จุด ครอบ SEA
ตัวแปรรายชั่วโมงwind_speed_10m, wind_direction_10m, precipitation, relative_humidity_2m, temperature_2m
ย่อเป็นรายวันลม/rh/temp = ค่าเฉลี่ยต่อวัน · ฝน = ผลรวมต่อวัน (เพราะฝนสนใจปริมาณสะสม)
ผลลัพธ์openmeteo_met_2019_2023.nc (รูป T × 12 × 11 ต่อตัวแปร)

ทฤษฎีสำคัญ: แปลง "ความเร็ว+ทิศลม" → "u, v" (เวกเตอร์)

API ให้ลมมาเป็น ความเร็ว (speed) + ทิศ (direction เป็นองศา) แต่โมเดลต้องใช้เป็น องค์ประกอบ 2 แกน (u = ตะวันออก-ตก, v = เหนือ-ใต้) จึงแปลงด้วยตรีโกณ:

u = −speed · sin(θ)     v = −speed · cos(θ) θ = ทิศที่ลม "พัดมาจาก" (meteorological convention) · เครื่องหมายลบแปลงเป็นทิศที่ลม "พัดไป" · u,v คือสิ่งที่ใช้พัดพาควัน

ทำไมต้องมี precip/rh/temp นอกจากลม? — เพราะการวิเคราะห์ LOYO (leave-one-year-out) พบว่าโมเดลจับ "ระดับฝุ่นต่อปี" ไม่ได้ เพราะ ความสัมพันธ์ไฟ→ฝุ่นขึ้นกับสภาพอากาศที่ยังไม่ได้ใส่: ฝนชะล้างฝุ่น, ความชื้นทำให้ละอองโต, อุณหภูมิคุมเสถียรภาพอากาศ

B1. แนวคิดกริด + ระบบพิกัด

เราสร้าง "ตารางมาตรฐาน" ครอบทั้งภูมิภาค:

LAT = linspace(14.0, 25.0, 111)   LON = linspace(96.0, 106.0, 101)   step = 0.1° H=111 แถว × W=101 คอลัมน์ = G = 11,211 ช่อง · 0.1° ≈ 11 กม. · ครอบ ไทยเหนือ+พม่า(ฉาน/คะฉิ่น)+ลาวเหนือ

การแปลงพิกัดจริง→ช่องกริด (ใช้ทั้ง rasterize ไฟ และวางสถานี):

ilat = round((lat − LAT[0]) / 0.1)    ilon = round((lon − LON[0]) / 0.1) เช่น สถานีที่ lat=18.8 → ilat = round((18.8−14.0)/0.1) = แถวที่ 48 · นี่คือการ "หาว่าจุดนี้ตกช่องไหน"
เหมือนกระดาษกราฟทับแผนที่ — ทุกจุดบนโลก (ละติจูด/ลองจิจูด) ถูก "ปัดเศษ" เข้าช่องสี่เหลี่ยมที่ใกล้ที่สุด เพื่อให้ข้อมูลทุกชนิดพูดภาษาเดียวกัน (ช่องเดียวกัน)

B2. Rasterize ไฟ — แปลงจุด → สนามบนกริด

ไฟเป็น "จุดกระจัดกระจาย" (lat,lon,frp) ต้องแปลงเป็น "สนาม" (grid) วิธีจริง: หาช่องที่จุดไฟตก แล้วบวก FRP สะสมเข้าช่องนั้น

for แต่ละจุดไฟ:
    di   = index ของวัน (acq_date)
    ilat = round((lat − LAT[0]) / 0.1)
    ilon = round((lon − LON[0]) / 0.1)
    emis[di, ilat, ilon] += frp        # หลายจุดในช่องเดียว → บวกกัน

ผลลัพธ์: emission รูป T × H × W = ปริมาณการปล่อยต่อช่องต่อวัน

B3. Interpolation — ขยายข้อมูลหยาบ → ละเอียด

ลม Open-Meteo เป็นกริดหยาบ 1° (12×11) แต่กริดเราละเอียด 0.1° (111×101) → ต้อง "เติมค่าระหว่างจุด" ด้วย bilinear interpolation

ค่า ณ จุดใหม่ = ถ่วงน้ำหนักเชิงเส้นจาก 4 จุดกริดหยาบรอบๆ (ตามระยะ) โค้ดใช้ RegularGridInterpolator(method="linear") · DEM ใช้คนละทาง: coarsen(30×30) ลงมาก่อน แล้วค่อย interp
เหมือนรู้อุณหภูมิแค่ 4 มุมห้อง แล้วเดาอุณหภูมิตรงกลางห้องด้วยการเฉลี่ยถ่วงน้ำหนักตามระยะ — ยิ่งใกล้มุมไหน ค่ายิ่งเหมือนมุมนั้น

B4. ประกอบ Datacube + Normalize + แบ่ง Train/Test

ตัวแปรรูป (shape)คืออะไร
X(T, C, H, W)datacube หลัก · C = [u10, v10, dem, precip, rh, temp, emission] · emission อยู่ช่องสุดท้ายเสมอ
met(T, 6, H, W)= X[:, :−1] → input ของ encoder
emission (E)(T, H, W)= X[:, −1] → ตัวที่ถูกคูณด้วย K

Normalization — และทำไม "เฉพาะ train" (ทฤษฎีกัน leakage)

met_mu  = met[train].mean()      # คำนวณจาก train เท่านั้น
met_std = met[train].std()
met_norm = (met − met_mu) / met_std       # z-score
e_max    = emission[train].max()
emis_norm = emission / e_max              # scale 0..1
y_norm    = y / pm25_max(train)
ทฤษฎี data leakage: ถ้าเราคำนวณค่าเฉลี่ย/max จากทั้งชุด (รวม test) เท่ากับ "โมเดลแอบเห็นสถิติของอนาคต" → คะแนนสอบจะดีเกินจริง · กฎเหล็ก: สถิติทุกตัวคำนวณจาก train แล้วเอาไปใช้กับ test เหมือนสอบจริงที่ห้ามเห็นข้อสอบล่วงหน้า

แบ่ง Train/Test ตามปี (ไม่ใช่สุ่ม)

train = 2019–2022 · test = 2023 (ปีวิกฤต ที่โมเดลไม่เคยเห็น) — แบ่งตามเวลาเพื่อเลียนแบบการใช้งานจริง (ทำนายอนาคต) ไม่ใช่สุ่มวันปนกัน

C1. ภาพรวม forward pass (1 วัน)

C2. Met Encoder — บีบสภาพอากาศเป็น "เวกเตอร์บริบท"

หน้าที่: รับสนามอุตุนิยม (B,6,H,W) → บีบเป็นเวกเตอร์สั้นๆ (B, hidden) ที่สรุป "วันนี้อากาศเป็นยังไง"

encoder = Sequential(
    Conv2d(in_ch=6, 16, kernel=3, padding=1), GELU(),   # จับลวดลายเชิงพื้นที่
    Conv2d(16, 32, 3, padding=1), GELU(),
    AdaptiveAvgPool2d((8,8)),    # ย่อทุกขนาดภาพ → 32×8×8 = 2048
    Flatten(),
    Linear(2048, hidden=64), GELU(),
    Dropout(0.2),
)   # output: h = (B, 64)
ชั้นทำอะไร (ทฤษฎี)
Conv2dเลื่อนตัวกรอง 3×3 ทั่วภาพ จับลวดลายท้องถิ่น (เช่น "ลมแรงปะทะภูเขา") — แชร์น้ำหนักทั้งภาพ จึงประหยัดพารามิเตอร์
GELUฟังก์ชัน activation (ทำให้เน็ตเรียนความสัมพันธ์ไม่เชิงเส้นได้)
AdaptiveAvgPool(8,8)เฉลี่ยลงเหลือ 8×8 ไม่ว่าภาพเข้ามาขนาดไหน → ได้ขนาดคงที่
Linearฉายลงเป็นเวกเตอร์ 64 มิติ = "สรุปอากาศวันนี้"
encoder = "นักอุตุนิยมที่ดูแผนที่อากาศทั้งใบแล้วสรุปเป็นโน้ตสั้นๆ 64 คำ" — โน้ตนี้ (h) คือสิ่งที่หัวต่อไปใช้ตัดสินว่า "วันนี้ลมจะพาควันยังไง"

C3. K_head — สร้าง Transport Kernel K

นี่คือหัวใจ: แปลงเวกเตอร์บริบท h → เมทริกซ์ K (B, S, G) ที่บอกว่า "ไฟที่ช่อง g ส่งฝุ่นถึงสถานี s แรงแค่ไหน"

มี 3 วิธีสร้าง K (จากแพงสุด → ประหยัดสุด):

variantวิธีสร้าง Kparams (M2)เหตุผล
CLNO (full)K = softmax(Linear(h) → S×G)~28Mตรงไปตรงมา แต่ K_head ใหญ่มาก
CLNOLowRankK = softmax(U @ Vᵀ), U:(S,r) V:(G,r), r=32~23Mแตก K เป็นเมทริกซ์ผอม 2 ตัว ประหยัด memory
CLNOGlobalVV เป็น พารามิเตอร์ global (เรียนครั้งเดียว), เน็ตเดาแค่ U(met)~0.56Mลด 40 เท่า กัน overfitting — "โหมดการพัดพาเชิงพื้นที่เสถียร เปลี่ยนแค่น้ำหนักรายวัน"
K_logit = U(h) @ Vᵀ   →   K = softmaxover G(K_logit) U (B,S,r) = "สถานีนี้สนใจโหมดไหน" · V (G,r) = "แต่ละช่องกริดอยู่ในโหมดไหน" · softmax ทำให้ผลรวมน้ำหนักของแต่ละสถานี = 1
คิดถึง K เป็น "ตารางอิทธิพล" ขนาด สถานี×ช่องกริด — แถวของสถานีเชียงใหม่จะมีค่าสูงตรงช่องที่อยู่เหนือลมขึ้นไป (ไฟตรงนั้นลอยมาถึงง่าย) และต่ำตรงช่องใต้ลม · low-rank = แทนที่จะจำทุกช่องตรงๆ ก็อธิบายด้วย "ธีมร่วม r=32 ธีม" (เช่น ธีมลมเหนือ ธีมลมตะวันตก)

C4. การคูณ K @ E + b → พยากรณ์

h    = encoder(met)              # (B, 64)
K    = build_K(h)                # (B, S, G)  softmax
b    = softplus(b_head(h))       # (B, S)     ≥ 0 เสมอ
E    = emission.view(B, G)       # (B, G)
pm25 = bmm(K, E.unsqueeze(-1)).squeeze(-1) + b    # (B, S)
PM2.5s = Σg=1G Ksg · Eg + bs สำหรับแต่ละสถานี s: เอาน้ำหนัก K ของทุกช่องกริดมาคูณ FRP ของช่องนั้น แล้วบวกกัน + ฝุ่นพื้นหลัง

softplus ที่ b รับประกัน b ≥ 0 (ฝุ่นพื้นหลังติดลบไม่ได้) · K จาก softmax ≥ 0 และ E ≥ 0 → PM2.5 ≥ 0 โดยโครงสร้าง ไม่ต้อง clamp

นี่คือเหตุผลที่ "เชิงเส้นใน E": ถ้าเพิ่มไฟที่ช่อง g เป็น 2 เท่า → ส่วนที่ g สมทบก็เป็น 2 เท่าพอดี (K คงที่เพราะ K ขึ้นกับ met ไม่ใช่ E) · คุณสมบัตินี้คือสิ่งที่ปลดล็อก attribution + inversion

C5. Attribution — ระบุแหล่ง (ได้ฟรีจากรูปสมการ)

เพราะ PM2.5 เป็นผลบวกของส่วนสมทบจากแต่ละช่อง เราจึง "แกะ" ส่วนสมทบกลับมาได้ตรงๆ:

contrib  = K * E.unsqueeze(1)               # (B, S, G) = K_sg · E_g
total    = contrib.sum(dim=-1)              # ผลรวมทุกช่อง = PM2.5 (ส่วนไฟ)
fraction = contrib / total                  # สัดส่วน 0..1 ของแต่ละช่อง
สัดส่วนที่สถานี s มาจากช่อง g = (Ksg · Eg) / Σg' (Ksg' · Eg') รวมช่องตามประเทศ/อำเภอ → "เชียงใหม่วันนี้มาจากเมียนมา X% ลาว Y%" · นี่คือ Smoke Fingerprint
เหมือนสูตรค็อกเทลที่รู้ว่าใส่อะไรไปเท่าไร — รสรวม = เหล้า A + เหล้า B + น้ำผลไม้ C ในเมื่อเรารู้ปริมาณแต่ละอย่าง (K·E) ก็บอกได้ทันทีว่า "รสนี้ 40% มาจากเหล้า A"

C6. Inversion — แก้ย้อนหาแหล่ง (ปัญหาผกผัน) ทำแล้ว แต่ยังอ่อน

คำถามกลับด้าน: "รู้ PM2.5 ที่สถานี อยากรู้ว่าไฟ (E) อยู่ตรงไหน" = แก้สมการ K·E ≈ y − b หา E

แก้: minE ‖K·E − (y−b)‖² + α‖E‖²    (Tikhonov regularization) α = ตัวคุมความเรียบ (กันคำตอบระเบิด) · ใช้ dual-form: (KKᵀ+αI)λ = y, E = Kᵀλ → แก้ขนาด S×S แทน G×G (เร็วกว่ามากเมื่อ G≫S)
ทำไมยังได้ผลแย่ (r≈0): มีสถานีแค่ ~12–99 ตัว แต่ช่องกริด G=11,211 → สมการน้อยกว่าตัวแปรมหาศาล (underdetermined) เหมือนมีสมการ 12 ข้อแต่ตัวแปร 11,211 ตัว → คำตอบไม่ซ้ำ · นี่คือเหตุผลที่ inversion ควรลดเป็น proof-of-concept (เคสไฟจุดเดียวที่รู้คำตอบ) ไม่ใช่ฟีเจอร์หลัก

C7. การเทรน — สอนโมเดลยังไง

loss   = masked_MSE(pred, y)     # ข้าม NaN (วันที่สถานีไม่มีข้อมูล)
opt    = Adam(lr=1e-3, weight_decay=1e-4)
sched  = CosineAnnealingLR(200 epochs)
clip_grad_norm(1.0)              # กัน gradient ระเบิด
batch  = 32 วัน/ก้อน             # full T ล้น GPU 6GB
องค์ประกอบทฤษฎี/เหตุผล
masked MSEวัด error เฉพาะจุดที่มีเฉลย (สถานีว่าง = NaN ข้ามไป) ไม่งั้น NaN ทำลาย gradient
Adamoptimizer ปรับ learning rate รายพารามิเตอร์อัตโนมัติ
weight_decayลงโทษน้ำหนักใหญ่ → กัน overfitting
Cosine scheduleค่อยๆ ลด lr แบบโค้ง cosine → ลู่เข้านิ่งช่วงท้าย
grad clip 1.0จำกัดขนาด gradient ไม่ให้ก้าวกระโดดจนพัง

📐 PART D — ทฤษฎีลงลึก (ทำไมออกแบบแบบนี้)

D1. Conditionally-Linear + Superposition

"Conditionally-linear" = เชิงเส้นแบบมีเงื่อนไข:

PM2.5 = K(met) · E met (ลม/ภูมิประเทศ) เข้าผ่านฟังก์ชันไม่เชิงเส้น (เน็ต) → ได้ K · แต่เมื่อ "ตรึง K ไว้" ความสัมพันธ์ระหว่าง E กับ PM2.5 เป็นเส้นตรง

Superposition theorem: เมื่อระบบเชิงเส้น input รวม → output รวม = ผลบวกของ output ย่อย แยกไฟเป็น tracer ตามแหล่ง E = E⁽พม่า⁾ + E⁽ลาว⁾ + … แล้ว:

K·(E⁽พม่า⁾ + E⁽ลาว⁾) = K·E⁽พม่า⁾ + K·E⁽ลาว⁾ → ฝุ่นรวม = ฝุ่นจากพม่า + ฝุ่นจากลาว เป๊ะ (ไม่มี cross-term) → แยกแหล่งได้แม่น + "ดับไฟพม่า" = ลบ K·E⁽พม่า⁾

D2. ทำไม softmax over sources (ไม่ใช่ ReLU เฉยๆ)

K = softmax over G ทำให้ Σg Ksg = 1 — ตีความว่า K เป็น "การกระจายน้ำหนัก" ว่าฝุ่นที่สถานีมาจากช่องไหนในสัดส่วนเท่าไร · ได้ K ≥ 0 อัตโนมัติ (จำเป็นต่อ non-negativity และ attribution ที่ตีความเป็น %)

D3. Low-rank กับ Global-V (พีชคณิตเชิงเส้นเบื้องหลัง)

Low-rank factorization: เมทริกซ์ใหญ่ K (S×G) ถูกประมาณด้วยผลคูณเมทริกซ์ผอม 2 ตัว U(S×r)·Vᵀ(r×G) เมื่อ r≪G

params: S·G  →  r·(S+G)    เช่น 99·11211 ≈ 1.1M → 32·(99+11211) ≈ 0.36M สมมติฐาน: "อิทธิพลการพัดพา" มีโครงสร้างซ้ำ (โหมดร่วม) ไม่ใช่ค่าสุ่มอิสระทุกช่อง → อธิบายด้วย r โหมดได้

GlobalV: ค้นพบว่า "โหมดเชิงพื้นที่ (V) เสถียรทางฟิสิกส์" จึงทำให้ V เป็นพารามิเตอร์เรียนครั้งเดียว (ไม่ต้องเดาใหม่ทุกวัน) เน็ตเดาแค่ U(met) → ลด params 40 เท่า · ผลทดลอง: ลด params 24M→0.4M test แทบไม่ขยับ → ยืนยันว่า "เน็ตไม่ใช่คอขวด"

D4. Inverse problem + Tikhonov (ทำไมต้อง regularize)

ปัญหาผกผันที่ underdetermined มีคำตอบมากมายที่ fit ข้อมูลได้เท่ากัน Tikhonov เพิ่มพจน์ α‖E‖² = "เลือกคำตอบที่เล็ก/เรียบที่สุด" เพื่อให้คำตอบ เสถียรและไม่ซ้ำซ้อน · แต่ถ้าข้อมูล (สถานี) น้อยเกินไป regularize ก็ช่วยได้จำกัด → ต้องเพิ่มสถานี/ข้อมูลถึงจะดีขึ้นจริง

🚀 PART E — ส่วนขยายตามแผน (input→output ของของที่จะเพิ่ม)

หมายเหตุความซื่อสัตย์: โมเดลจริงตอนนี้คือ "per-day linear-in-emission kernel" (Part C) · ส่วน semi-Lagrangian advection + autoregressive rollout + 2 หัว ที่อยู่ในแผนภาพรวม ยังเป็นส่วนขยาย — Part E อธิบายว่ามันจะต่อเข้ากับของเดิมยังไง

E1. Emission Curve + BLH — แก้ "แกนปัญหาจริง" (สำคัญสุดของ 2 เดือน) งานหลัก

(1) Emission curve — แปลง FRP แบบอิ่มตัว

ปัญหา: ตอนนี้ E = FRP ตรงๆ (เชิงเส้น) แต่จริงๆ ไฟแรงมากๆ ฝุ่นไม่ได้เพิ่มเป็นเส้นตรงตาม (อิ่มตัว) → โมเดล over-predict ตอนไฟหนัก

E = f(FRP) = log(1 + FRP)   หรือ   FRP / (FRP + k) เปลี่ยนช่อง emission ใน datacube จาก FRP ดิบ → ผ่านเส้นโค้งอิ่มตัวก่อนป้อนโมเดล · ยังคงเชิงเส้นใน E (ตัวใหม่) → attribution ไม่พัง

(2) BLH (Boundary Layer Height) — ประตูระบายอากาศ

เพิ่ม BLH เป็น met channel ใหม่ — มันคือ "ความสูงของฝาชั้นบรรยากาศที่อากาศผสมกัน" ฝาต่ำ = ฝุ่นถูกอัดเข้มข้น, ฝาสูง = เจือจาง · นี่คือตัวแปรที่ทำให้ "ไฟเท่ากันคนละวันได้ฝุ่นต่างกัน" ที่ขาดหาย

ความเข้มข้น ∝ การปล่อย / BLH    (ฟิสิกส์ box model) ใส่ BLH เข้า encoder → ให้ K/b ปรับตามการระบายอากาศ → คาดว่าจะลด bias ปีต่ำจาก +45 → ≤+25 (gate ตัดสิน)

E2. Semi-Lagrangian Advection + Rollout ส่วนขยาย

ปัจจุบัน K เรียน "การพัดพา" จากข้อมูลโดยปริยาย · ส่วนขยายคือใส่การพัดพาเป็น ฟิสิกส์ตรงๆ ก่อน แล้วให้เน็ตแก้ส่วนที่เหลือ

Ct+1(x) = Ct(x − u·Δt) + [แหล่ง − สูญเสีย] หา "ควันตรงนี้พรุ่งนี้มาจากไหนเมื่อวาน" = เดินถอยทวนลม (x − u·Δt) แล้วหยิบค่ามา · monotone interpolation กันค่าติดลบ/overshoot

Rollout: ป้อน Ct+1 กลับเข้าโมเดลซ้ำ → ได้ t+2, t+3, … t+48 · เทรนแบบ multi-step ให้ทนต่อ error สะสม

E3. สถาปัตยกรรม 2 หัว — ต่อบนโค้ดเดิมยังไง ก้าวข้าม SOTA

เก็บ CLNO (หัว 1 เชิงเส้น) ไว้ทำ attribution + เพิ่มหัว diffusion (หัว 2) มากู้จุดพีค — โดยหัว 2 ทำงานกับ "ค่ารวม" เท่านั้น ไม่แตะ tracer

ความเสี่ยงวิธีกัน (เชิงเทคนิค)
diffusion ทำลาย superpositionหัว 2 รับเฉพาะ PM2.5 รวม (total) → refine เป็นแผนที่เดียว · tracer แยกสียังมาจากหัว 1 ที่เชิงเส้น 100%
ช้า เสียเคลม "วินาที"รัน diffusion เฉพาะ output สุดท้าย + few-step sampler (ไม่ใช่ทุก rollout step)
พิสูจน์ว่าช่วยจริงablation CLNO เปล่า vs CLNO+diffusion บน CSI วันวิกฤต — เลียนแบบตาราง AFNO vs AFNO+StormCast เป๊ะ
ทำไม่ทันหัว 2 ตัดได้ แกน attribution ไม่ตายตาม
สรุป input → output ทั้งระบบ: ข้อมูลดิบ 4 แหล่ง (ไฟ/ลม/ภูเขา/ฝุ่น) → regrid ลงกริด 111×101 → normalize (train เท่านั้น) → met เข้า CNN encoder → ได้ context h → สร้าง kernel K + background b → PM2.5 = K@E + b → จากรูปสมการเชิงเส้นนี้แตกออกเป็น พยากรณ์ + attribution (K·E/total) + inversion (lstsq) ฟรี · งาน 2 เดือนคือเพิ่ม emission curve + BLH แก้แกนปัญหา แล้วต่อ หัว diffusion กู้จุดพีคเพื่อก้าวข้าม SOTA
