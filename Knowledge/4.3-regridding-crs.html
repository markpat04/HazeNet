4.3 — Regridding / CRS / Reprojection 🟡 กลาง

รวมข้อมูล 5 แหล่งคนละความละเอียด/คนละระบบพิกัด ให้มาอยู่บน "กริด 0.1° เดียวกัน"

ต่อจาก 4.2 (เปิดเป็น xarray ได้แล้ว) · ส่งต่อ → 4.4 (ประกอบ datacube) · โยง 2.1 (กริด/discretization) และ 2.4 (mass conservation)

โจทย์ของบทนี้ — ปัญหา "จิ๊กซอว์คนละสเกล": ERA5 มาเป็นช่อง ~31 กม. · Himawari ~5 กม. (ยังเป็นมุมมองดาวเทียมเอียง ๆ) · DEM 30 ม. · FIRMS/Air4Thai เป็น "จุด" · จะซ้อนเป็น datacube ก้อนเดียวได้ ทุกอันต้องมาอยู่บนตารางเดียวกันเป๊ะ ๆ — ช่องที่ [i,j] ของทุกตัวแปรต้องหมายถึงพื้นที่เดียวกันบนโลก

บทนี้สอน 3 การแปลงที่ทำให้สำเร็จ: ① Reproject (เปลี่ยนระบบพิกัด) · ② Regrid (เปลี่ยนความละเอียด) · ③ Rasterize (จุด→กริด) · พร้อม กับดักใหญ่: เลือกวิธีผิดกับ "ปริมาณที่ต้องอนุรักษ์" (เช่น การปล่อย) = มวลรั่ว ผิดทั้งงาน

1. กริดหลัก (master grid) — กำหนด "ตาราง" ของทั้งโปรเจกต์ก่อน

ก่อนแปลงอะไร ต้องนิยาม กริดเป้าหมาย ให้ชัดก่อน 1 อัน แล้วบังคับให้ทุกแหล่ง "ย้ายมาลงตารางนี้" · จากแผน 4.1 เราเลือก 0.1° ทั่วโดเมนไทย/SEA ตอนบน:

import numpy as np, xarray as xr
# กริดหลัก: lat 5–25°N, lon 92–110°E, ระยะห่าง 0.1°
lat = np.arange(25.0, 4.95, -0.1)   # เรียงมาก→น้อย (ให้ตรงกับ ERA5)
lon = np.arange(92.0, 110.05, 0.1)
grid = xr.Dataset(coords={"lat": lat, "lon": lon})   # ~201 × ~181 ช่อง
ทำไมต้องนิยามก่อน: ถ้าไม่มีกริดอ้างอิงกลาง แต่ละแหล่งจะมาคนละขอบ/คนละระยะ → พอซ้อนกันจะ "เหลื่อม" ครึ่งช่อง · กำหนด grid ครั้งเดียวแล้ว regrid ทุกอย่างเข้าหามัน = ทุกตัวแปรตรงช่องกันเป๊ะ (เงื่อนไขจำเป็นของ 4.4)
กับดักลำดับแกน: ERA5 เก็บ latitude เรียง มาก→น้อย (เหนือ→ใต้) แต่หลายชุดเรียงกลับกัน · ต้องบังคับให้ทุกแหล่งเรียงเหมือน grid ไม่งั้นภาพจะ "กลับหัว" แบบเงียบ ๆ (ไม่ error แต่ผลผิด)

2. CRS + Reproject — "พิกัดนี้อยู่ในระบบไหน"

CRS (Coordinate Reference System) = กติกาที่บอกว่าตัวเลขพิกัดแปลเป็นตำแหน่งจริงบนโลกยังไง · 2 ตระกูลหลัก:

  • Geographic — ละติจูด/ลองจิจูดเป็นองศา · มาตรฐานคือ EPSG:4326 (WGS84) · ข้อมูลส่วนใหญ่ของเรา (ERA5, DEM, FIRMS) อยู่ระบบนี้อยู่แล้ว → สบาย
  • Projected/อื่น ๆ — แปลงโลกกลมเป็นระนาบ มีหน่วยเป็นเมตร (เช่น UTM) หรือ geostationary (มุมมองดาวเทียมค้างฟ้า)

Reproject = แปลงจาก CRS หนึ่งไปอีก CRS · ในงานเราเจอกรณีเดียวที่ต้อง reproject จริงจัง คือ Himawari ถ้าผลิตภัณฑ์มาในระบบ geostationary (กริดโค้งตามมุมสแกน) ต้องดัดให้เป็น lat/lon ก่อน:

import rioxarray
# ถ้า DataArray มี CRS ติดมา (rioxarray อ่านได้): reproject + จัดให้ตรงกริดหลักในก้าวเดียว
aod_ll = aod.rio.reproject_match(grid_da)   # grid_da = DataArray ตัวอย่างบนกริดหลัก
ข่าวดี: เพราะ ERA5/DEM/FIRMS อยู่ EPSG:4326 อยู่แล้ว → ส่วนใหญ่ ข้ามขั้น reproject ได้ เหลือแค่ "regrid ความละเอียด" (ข้อ 3) · งานหนักจริงคือ Himawari เท่านั้น

4. Interpolation — หยาบ→ละเอียด (ใช้กับ ERA5)

ERA5 มา 0.25° (~31 กม.) แต่กริดเราคือ 0.1° → ต้อง "เดาค่าตรงช่องที่ไม่มีข้อมูลเดิม" ด้วยการ interpolate · xarray ทำได้ตรง ๆ บนชื่อแกน:

met_on_grid = met.interp(lat=grid.lat, lon=grid.lon, method="linear")  # bilinear
  • nearest — เอาค่าช่องที่ใกล้สุด (เหมาะกับข้อมูลหมวดหมู่ เช่น land cover)
  • linear (bilinear) — เฉลี่ยถ่วงน้ำหนัก 4 ช่องรอบ ๆ → เนียน เหมาะกับสนามต่อเนื่อง (ลม/อุณหภูมิ/AOD)
caveat ความซื่อสัตย์: การ interpolate หยาบ→ละเอียด ไม่ได้สร้างข้อมูลใหม่จริง — มันแค่ทำให้ดูเนียนขึ้น · รายละเอียดระดับ 11 กม. ที่ "ไม่เคยมีใน ERA5" จะยังไม่มี · ห้ามเคลมว่า downscale แล้วได้ความละเอียดจริงเพิ่ม · นี่คือสิ่งที่ทำให้โมเดล (ที่มี DEM/AOD ความละเอียดสูงกว่า) มีคุณค่า — มันเติมรายละเอียดที่ ERA5 ไม่มี

5. Aggregation — ละเอียด→หยาบ (ใช้กับ DEM)

DEM 30 ม. ละเอียดกว่ากริดเรา ~360 เท่า → ต้อง "ยุบหลายพันพิกเซลเป็น 1 ช่อง" ด้วยการรวมทางสถิติ · และตรงนี้มี โอกาสทอง: นอกจากค่าเฉลี่ย เราดึง "สถิติย่อย" ออกมาเป็น channel เสริมได้:

dem_ll  = dem.rio.reproject_match(grid_da)        # วิธีง่าย: ให้ rioxarray จัดการ
# หรือ coarsen เองเพื่อดึงสถิติย่อย:
elev_mean = dem.coarsen(x=360, y=360, boundary="trim").mean()
elev_std  = dem.coarsen(x=360, y=360, boundary="trim").std()   # ความขรุขระภูมิประเทศ!
ทำไม std มีค่า: ค่าเฉลี่ยความสูงบอกว่า "พื้นที่นี้สูงเท่าไร" แต่ ส่วนเบี่ยงเบน (std) บอกว่า "ขรุขระแค่ไหน" — ช่องที่ std สูง = มีทั้งยอดเขาและก้นหุบในช่องเดียว = ภูมิประเทศซับซ้อนที่กัก/ปั่นลม · เป็น channel ที่ช่วยโมเดลเข้าใจการกักฝุ่นในแอ่งภาคเหนือ (โยง 4.1 §7)

6. Conservative vs Bilinear — กับดักที่ทำทั้งงานพังเงียบ ๆ ★ สำคัญสุด

นี่คือหัวใจของบทนี้ และเป็นจุดที่เชื่อมตรงกับ 2.4 (mass conservation) · ปริมาณมี 2 ชนิด และต้อง regrid คนละวิธี:

ชนิดตัวอย่างกฎตอน regridวิธีที่ถูก
Intensive (เข้มข้น/เฉลี่ยได้)อุณหภูมิ, ลม, AOD, ความชื้นค่าเฉลี่ยต้องสมเหตุผลbilinear โอเค
Extensive (ปริมาณรวม/ต้องอนุรักษ์)การปล่อย E, ฝนสะสม, มวลผลรวมทั้งโดเมนต้องไม่เปลี่ยนconservative เท่านั้น
import xesmf as xe
# สนามต่อเนื่อง (ลม/AOD): bilinear
rg_bilin = xe.Regridder(src, grid, "bilinear")
# ปริมาณที่ต้องอนุรักษ์ (การปล่อย/ฝน): conservative — ต้องมีขอบเขตช่อง (bounds)
rg_cons  = xe.Regridder(src, grid, "conservative")
emis_grid = rg_cons(emis_src)
ทำไมเรื่องนี้ "พังเงียบ": ใช้ bilinear กับการปล่อยจะ ไม่ error — โค้ดรันผ่าน ภาพดูดี แต่ปริมาณการปล่อยรวมเพี้ยนไปเรื่อย ๆ · ผลคือ forecast ฝุ่นเบี้ยว และที่ร้ายกว่าคือ source attribution (3.6) ผิด เพราะ "งบมวล" ไม่ตรง · กฎเหล็ก: อะไรที่ต้องอนุรักษ์ → conservative เสมอ (สอดคล้องกับสมบัติที่เราเคลมใน 2.4)

7. Rasterization — จุด→กริด (ใช้กับ FIRMS)

FIRMS เป็นจุดไฟกระจัดกระจาย · ต้อง "นับ/รวม FRP ลงช่อง 0.1°" ให้กลายเป็นกริดการปล่อย E · วิธีตรงไปตรงมาและกันพลาดคือ จับแต่ละจุดเข้า bin แล้ว sum (อนุรักษ์ผลรวมโดยธรรมชาติ):

import numpy as np
# แปลงพิกัดจุดเป็น index ช่องบนกริดหลัก แล้วรวม frp ต่อช่อง
iy = ((25.0 - fire.latitude ) / 0.1).astype(int)
ix = ((fire.longitude - 92.0) / 0.1).astype(int)
E  = np.zeros((len(lat), len(lon)))
np.add.at(E, (iy, ix), fire.frp.values)   # รวม FRP ทุกจุดที่ตกในช่องเดียวกัน
E = xr.DataArray(E, coords={"lat":lat,"lon":lon}, dims=["lat","lon"])
ทำต่อทีละชั่วโมง: ทำ rasterize นี้แยกตามช่วงเวลา (ตาม acq_time) → ได้ E ที่เปลี่ยนตามเวลา · จุดนี้คือที่ที่ "tagged-tracer" (3.6) เริ่ม เพราะเรารู้ว่า FRP แต่ละก้อนมาจากช่อง/ประเทศไหน → ติดป้ายแหล่งได้ตั้งแต่ตรงนี้

ทางเลือกไลบรารี: geocube.make_geocube หรือ rasterio.features.rasterize ถ้าอยากได้ความสามารถเชิงภูมิสารสนเทศเพิ่ม

9. เครื่องมือ + กับดักที่เจอบ่อย

งานเครื่องมือ
interp บนชื่อแกน (เร็ว ง่าย)xarray.interp / .coarsen
reproject + จัดกริดในก้าวเดียวrioxarray .rio.reproject_match
conservative / bilinear แบบมืออาชีพxESMF (xe.Regridder)
จุด→กริดnumpy.add.at / geocube / rasterio
เช็กลิสต์กับดัก (อ่านก่อนรันจริง):
  • ลำดับ lat กลับหัว — ภาพพลิกแบบเงียบ → บังคับเรียงให้ตรง grid
  • เลือก bilinear กับปริมาณอนุรักษ์ — มวลรั่ว → ใช้ conservative
  • NaN จากเมฆ/กลางคืน (Himawari/FIRMS) — ต้องตัดสินใจ: เว้นว่าง / เติม / มาสก์ ก่อน regrid ไม่งั้น NaN ลามทั้งช่อง
  • ขอบโดเมนเหลื่อมครึ่งช่อง — regrid เข้าหา grid เดียวเสมอ อย่าให้แต่ละแหล่งนิยามขอบเอง
  • หน่วยไม่ตรง (ฝน m vs mm, อุณหภูมิ K vs °C) — แปลงก่อน merge

10. Cheat sheet + แบบฝึกหัด

Cheat sheet

  • master grid ก่อน → regrid ทุกอย่างเข้าหามัน
  • CRS: ส่วนใหญ่ EPSG:4326 อยู่แล้ว · reproject แค่ Himawari (geostationary)
  • 3 โหมด: ① interp (หยาบ→ละเอียด) · ② aggregate (ละเอียด→หยาบ) · ③ rasterize (จุด→กริด)
  • กฎเหล็ก: intensive (ลม/T/AOD)→bilinear · extensive (E/ฝน)→conservative
  • interp หยาบ→ละเอียด ไม่สร้างข้อมูลจริงใหม่ (เขียน caveat)

❓ แบบฝึกหัด

  1. ทำไมต้องนิยาม master grid ก่อนแปลงข้อมูลอื่น?
  2. แหล่งไหนต้อง reproject จริง ๆ และเพราะอะไร?
  3. การปล่อย E ควร regrid ด้วย bilinear หรือ conservative? ถ้าเลือกผิดเกิดอะไร?
  4. DEM 30ม.→0.1° ใช้โหมดไหน และทำไม std ถึงเป็น channel ที่มีประโยชน์?
  5. "พังเงียบ" หมายความว่าอะไร ยกตัวอย่าง 1 อย่าง
เฉลย
  1. เพื่อให้ทุกตัวแปรลงช่อง [i,j] ที่หมายถึงพื้นที่เดียวกันเป๊ะ — เงื่อนไขจำเป็นของการ merge เป็น datacube
  2. Himawari (ถ้ามาเป็น geostationary) — กริดโค้งตามมุมสแกน ต้องดัดเป็น lat/lon ก่อน · ส่วน ERA5/DEM/FIRMS เป็น EPSG:4326 อยู่แล้ว
  3. conservative · ถ้าใช้ bilinear ผลรวมการปล่อยจะเพี้ยน (มวลรั่ว) → forecast เบี้ยว + attribution ผิด (โยง 2.4)
  4. ② aggregate (mean) · std บอก "ความขรุขระ" = ภูมิประเทศซับซ้อนที่กัก/ปั่นลม ช่วยอธิบายการสะสมฝุ่นในแอ่ง
  5. โค้ดรันผ่าน ไม่ error แต่ผลผิด เช่น lat กลับหัว (ภาพพลิก) หรือ bilinear ทำมวลรั่ว — อันตรายเพราะไม่มีสัญญาณเตือน

✅ เช็กลิสต์ก่อนไป 4.4

  • ☐ นิยาม master grid 0.1° เรียบร้อย (ลำดับ lat ถูก)
  • ☐ regrid ERA5 (bilinear) ลงกริดหลักได้
  • ☐ rasterize FIRMS เป็น E (sum) ได้
  • ☐ aggregate DEM (mean+std) ได้
  • ☐ แยกออกว่าตัวแปรไหน intensive/extensive และเลือก method ถูก
