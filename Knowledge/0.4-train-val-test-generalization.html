0.4 — Train / Val / Test & Generalization

แบ่งข้อมูลให้ถูก ไม่ให้โมเดล "โกงตัวเอง" — ถ้าพลาดตรงนี้ ตัวเลขทุกอย่างเชื่อไม่ได้

ปิดท้ายโมดูล 0 · เน้นเรื่อง data leakage ในข้อมูลเชิงพื้นที่-เวลา (จุดที่งานเราพังง่ายสุด) · เชื่อม 5.2 · ผูกกับ HazeNet ตลอด

หัวใจของบทนี้: เป้าหมายจริงไม่ใช่ "โมเดลเก่งกับข้อมูลที่เคยเห็น" แต่คือ เก่งกับข้อมูลใหม่ที่ไม่เคยเห็น (generalization) · วิธีพิสูจน์คือกันข้อมูลส่วนหนึ่งไว้ "สอบ" · แต่สำหรับข้อมูลแผนที่ฝุ่นรายชั่วโมง การแบ่งแบบสุ่มธรรมดา ทำให้โมเดลแอบเห็นคำตอบ — บทนี้สอนแบ่งให้ถูกจนตัวเลขเชื่อถือได้จริง

1. ทำไมต้องแบ่งข้อมูล — เรื่อง "นักเรียนท่องข้อสอบ"

นึกภาพนักเรียน 2 คน:

  • คน A ท่องเฉลยข้อสอบเก่าทุกข้อ → ทำข้อสอบเก่าได้ 100% แต่เจอข้อสอบใหม่ตก
  • คน B เข้าใจหลักการ → ทำข้อสอบเก่าได้ 85% แต่เจอข้อสอบใหม่ก็ยังได้ 85%

เราอยากได้ คน B (generalize ได้) แต่ถ้าเราวัดผลด้วย "ข้อสอบเก่า" คน A จะดูเก่งกว่า — นี่คือกับดัก · ทางแก้: เก็บ "ข้อสอบใหม่" ที่โมเดลไม่เคยเห็นไว้วัดผล

กฎเหล็กข้อ 1: ห้ามวัดผลโมเดลด้วยข้อมูลที่มันเคยเห็นตอนเทรน — มันจะดูเก่งหลอก ๆ (เรียกว่า "memorization") · การวัดผลที่มีความหมายต้องทำบนข้อมูลที่ ไม่เคยเห็น เท่านั้น

2. 3 ก้อน: Train / Validation / Test ★

เราแบ่งข้อมูลเป็น 3 ส่วน แต่ละส่วนมีหน้าที่ต่างกันชัดเจน:

ก้อนหน้าที่โมเดล "เห็น" ไหม
Trainปรับพารามิเตอร์ (เรียนรู้) — ที่ใช้ใน gradient descentเห็น (เรียนจากมัน)
Validationจูน hyperparameter (lr, ขนาดโมเดล), ตัดสินใจหยุดเทรน (early stopping), เลือกโมเดลที่ดีสุดเห็นทางอ้อม (เราตัดสินใจจากมัน)
Testวัดผลสุดท้าย "ครั้งเดียว" เพื่อรายงานตัวเลขที่ซื่อสัตย์ห้ามเห็นจนกว่าจะจบ
กฎเหล็กข้อ 2 — "แตะ test ครั้งเดียว": ถ้าคุณดู test แล้วกลับไปปรับโมเดล ดู test อีก ปรับอีก... = test กลายเป็น validation กลาย ๆ ตัวเลขจะดีหลอก · test มีไว้รายงานผลตอนจบเท่านั้น ทุกการตัดสินใจระหว่างทางใช้ val
ทำไมต้องมี val แยกจาก test: ถ้าใช้ test ทั้งจูนทั้งวัด เราจะ "overfit ต่อ test" โดยไม่รู้ตัว (เลือก setting ที่บังเอิญดีกับ test ชุดนั้น) · val ทำหน้าที่ "สนามซ้อม" ให้จูนได้เต็มที่ ส่วน test เก็บไว้เป็น "สนามจริง"

3. Overfitting vs Underfitting (ลึกกว่าใน 0.3)

2 โรคของโมเดล อยู่คนละขั้ว — เป้าหมายคือจุดสมดุลตรงกลาง

UnderfitOverfit
อาการtrain แย่, val แย่train ดีมาก, val แย่ (ช่องว่างกว้าง)
สาเหตุโมเดลง่ายไป / เทรนน้อยไป / lr เล็กโมเดลซับซ้อนไป / ข้อมูลน้อย / เทรนนานไป
แก้เพิ่ม capacity, เทรนนานขึ้น, เพิ่ม featureเพิ่มข้อมูล, regularization/dropout, early stopping, ลด capacity

ศัพท์เชิงทฤษฎี: underfit = "bias สูง" (อคติ มองง่ายไป) · overfit = "variance สูง" (ไวต่อ noise เกินไป) · เป้าหมายคือสมดุล bias-variance

4. Data Leakage — นักฆ่าเงียบที่ทำตัวเลขสวยหลอก

Data leakage = ข้อมูล/ความรู้จาก test แอบรั่วเข้ามาตอนเทรน ทำให้ผลดูดีเกินจริง แล้วพอใช้งานจริงพัง · มันเงียบเพราะ ไม่มี error เด้ง แค่ตัวเลขดีผิดปกติ

รูปแบบ leakageเกิดยังไง
Preprocessing leakคำนวณ mean/std (normalize) จากข้อมูลทั้งหมดก่อนแบ่ง → สถิติของ test รั่วเข้า train
Temporal leakใช้ข้อมูลอนาคตมาทำนายอดีต (เช่น สุ่มแบ่งเวลา → บางชั่วโมงอนาคตไปอยู่ train)
Spatial leakจุดที่อยู่ติดกันไปอยู่ทั้ง train และ test → โมเดล "ลอกเพื่อนข้าง ๆ"
Duplicate leakตัวอย่างซ้ำ/เกือบซ้ำ กระจายอยู่ทั้ง 2 ฝั่ง
ทำไมเรื่องนี้สำคัญกับงานประกวด/ตีพิมพ์: กรรมการ/reviewer ที่เก่งจะโจมตี evaluation ก่อนเลย — ถ้า split รั่ว ตัวเลขที่สวยจะถูกตีตกทันที · "ระวัง leakage" คือสิ่งที่แยกงานมืออาชีพออกจากงานมือใหม่

5. ★ ทำไม "random split" ผิดสำหรับข้อมูลของเรา

สำหรับข้อมูลทั่วไป (เช่นรูปแมว/หมาที่เป็นอิสระต่อกัน) การสุ่มแบ่งใช้ได้ · แต่ข้อมูลเชิงพื้นที่-เวลาของเรา จุดที่อยู่ใกล้กัน (ทั้งในเวลาและพื้นที่) เหมือนกันมาก (autocorrelation) — สุ่มแบ่งจะทำให้ test "เกือบเหมือน" train

★ จุดที่งานเราพังง่ายสุด: ถ้าสุ่มแบ่งแผนที่ฝุ่นรายชั่วโมง โมเดลจะได้คะแนนสูงปลอม ๆ เพราะมัน "เห็นชั่วโมงข้างเคียง" หรือ "จุดข้างเคียง" ในชุดเทรนอยู่แล้ว · พอเอาไปใช้จริง (พยากรณ์อนาคต/พื้นที่ใหม่) จะพังทันที · นี่คือเหตุผลที่เราต้องแบ่งตามเวลาและตามพื้นที่ ไม่ใช่สุ่ม

6. แบ่งให้ถูก — Temporal split + Spatial blocking

① Temporal split — เทรนอดีต ทดสอบอนาคต

เลียนแบบสถานการณ์จริง: เรามีแค่อดีต ต้องทำนายอนาคต · ห้ามให้อนาคตหลุดเข้า train เด็ดขาด

② Spatial blocking / Leave-stations-out — กันพื้นที่/สถานีเป็นก้อน

กันบางสถานี/บางภูมิภาคออกทั้งหมด ไม่ให้โมเดลเห็นตอนเทรน → วัดว่าโมเดลทำนาย "ที่ใหม่" ได้ไหม (สำคัญถ้าจะ deploy ในพื้นที่ที่ไม่มีสถานี)

เชื่อมกับงานเรา: เราจะใช้ ทั้งสองพร้อมกัน — แบ่งเวลา (test = ปีวิกฤตที่ไม่เคยเห็น) + กันบางสถานีไว้ test เชิงพื้นที่ · ทำให้เคลมได้ว่าโมเดล generalize ทั้ง "เวลาใหม่" และ "พื้นที่ใหม่" (เนื้อหาเต็มของวิธี CV อยู่ในหัวข้อ 5.2)

7. Cross-Validation — และทำไม k-fold ปกติใช้ไม่ได้

k-fold CV = แบ่งข้อมูลเป็น k ก้อน เทรน k ครั้ง สลับให้แต่ละก้อนได้เป็น test ครั้งหนึ่ง → ได้ค่าเฉลี่ย + ความแปรปรวน (น่าเชื่อถือกว่าแบ่งครั้งเดียว)

แต่ k-fold แบบสุ่มผิดสำหรับงานเรา (ด้วยเหตุผลเดียวกับส่วน 5) → ต้องใช้ blocked CV: แบ่งก้อนตามบล็อกเวลา/พื้นที่ที่ต่อเนื่องกัน ไม่สุ่มข้ามกัน · เช่น "spatially-blocked CV" และ "leave-one-year-out" · นี่คือมาตรฐานของงาน geospatial และเป็นสิ่งที่เราจะใช้ (รายละเอียด 5.2)

8. ป้องกัน Leakage ในทางปฏิบัติ (โค้ดจริง)

กฎทอง: "แบ่งก่อน แล้วค่อยทำอย่างอื่น"

# ❌ ผิด — normalize ก่อนแบ่ง (สถิติ test รั่วเข้า train)
mean, std = data.mean(), data.std()
data = (data - mean) / std
train, val, test = split(data)

# ✅ ถูก — แบ่งก่อน คำนวณสถิติจาก train เท่านั้น
train, val, test = split(data)
mean, std = train.mean(), train.std()        # จาก train เท่านั้น!
train = (train - mean) / std
val   = (val   - mean) / std                  # ใช้สถิติของ train
test  = (test  - mean) / std                  # ใช้สถิติของ train

หลักการเดียวกันกับทุก preprocessing ที่ "เรียนรู้จากข้อมูล" (normalize, fill missing, feature scaling) — fit บน train เท่านั้น แล้ว apply ไป val/test

Early stopping — ใช้ val หยุดให้ถูกเวลา

best_val = float("inf"); patience = 10; wait = 0
for epoch in range(max_epochs):
    train_one_epoch(...)
    val_loss = evaluate(val)
    if val_loss < best_val:
        best_val = val_loss; save(model); wait = 0   # ดีขึ้น → เซฟ
    else:
        wait += 1
        if wait >= patience: break                     # val ไม่ดีขึ้นนาน → หยุด (กัน overfit)
เชื่อมกับ 0.3: early stopping คือวิธีจับ "จุดที่ val เริ่มเด้งขึ้น" (overfit) จาก loss curve แล้วหยุดที่จุดดีสุด — ใช้ val ตัดสิน ไม่ใช่ test

9. เชื่อมกับ HazeNet — แผน split จริงของเรา

ก้อนช่วงเหตุผล
Trainฤดูแล้ง 2016–2022อดีตที่หลากหลาย ให้โมเดลเรียน pattern
Validation2024จูน lr/โมเดล + early stopping
Test2023 + 2025 (ปีวิกฤตหนัก)วัดบนปีที่ไม่เคยเห็น + เป็นปีที่ pain สูงสุด (มีค่าเทียบ WRF-Chem/PMF ด้วย)
+ กันสถานีบางสถานีกันไว้ testวัด generalization เชิงพื้นที่ (leave-stations-out)
  • ทำไมปีวิกฤตเป็น test: ถ้าโมเดลทำนายปีวิกฤตที่ไม่เคยเห็นได้ = พิสูจน์ว่า generalize จริง ไม่ใช่จำ · และตรงกับ pain ที่เราแคร์ที่สุด
  • normalize: คำนวณ mean/std จาก train (2016–2022) เท่านั้น แล้ว apply ไป val/test
  • กราฟ val ที่เราดูตอนเทรน ต้องมาจาก split กันรั่วนี้ ไม่งั้นกราฟสวยหลอก
  • ทั้งหมดนี้คือฐานของ systematic experiment (แผน master) — split ที่ถูกต้องคือเงื่อนไขที่ทำให้ benchmark/error analysis ทุกอย่างเชื่อถือได้

แบบฝึกหัด

  1. โมเดลได้ test accuracy 99% แต่ใช้งานจริงพัง — สงสัยอะไรเป็นอันดับแรก?
  2. ทำไมห้าม normalize ข้อมูลทั้งหมดก่อนแบ่ง train/test?
  3. เพื่อนสุ่มแบ่งแผนที่ฝุ่นรายชั่วโมง 80/20 แล้วได้ RMSE ต่ำมาก ควรเชื่อไหม เพราะอะไร?
  4. ทำไมเราเอา "ปีวิกฤต (2023/2025)" เป็น test ไม่ใช่ train?
  5. Val กับ Test ต่างกันยังไง ทำไมต้องมีทั้งคู่?
▶ เฉลย
  1. Data leakage — test รั่วเข้า train (ตัวเลขสวยหลอก) ต้องตรวจวิธีแบ่ง/preprocessing
  2. สถิติ (mean/std) ของ test จะรั่วเข้า train = leakage → fit บน train เท่านั้น
  3. ไม่ควรเชื่อ — random split ทำให้ชั่วโมง/จุดข้างเคียงอยู่ทั้ง train และ test (autocorrelation) โมเดลลอกเพื่อน RMSE ต่ำปลอม → ต้อง temporal + spatial split
  4. เพื่อพิสูจน์ว่าโมเดล generalize กับปีที่ไม่เคยเห็น + ตรงกับ pain ที่แคร์สุด + มี ground-truth (WRF-Chem/PMF) ให้เทียบ
  5. Val = สนามซ้อม (ใช้จูน/ตัดสินใจได้เต็มที่) · Test = สนามจริง (แตะครั้งเดียว) · ถ้าใช้ test จูนด้วย จะ overfit ต่อ test ตัวเลขดีหลอก
จบ 0.4 และจบโมดูล 0 ทั้งหมดแล้ว! 🎉 ตอนนี้คุณมีพื้นฐาน ML ครบ: คิดเป็น tensor (0.1) · สร้าง+เทรนโมเดล (0.2) · จูนให้เทรนลง (0.3) · แบ่งข้อมูลให้ผลเชื่อถือได้ (0.4)
