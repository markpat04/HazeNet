2. gradient คืออะไร — สัญชาตญาณก่อนสมการ

gradient = "ถ้าขยับ input นิดนึง output จะเปลี่ยนไปเท่าไรและทิศไหน" (เชิงคณิตคือ ความชัน/อนุพันธ์)

gradient descent = เดิน "สวนทาง" gradient ทีละก้าวเล็ก ๆ เพื่อลง "หุบเขา" ของ loss → โมเดลค่อย ๆ ดีขึ้น · autograd ทำหน้าที่คำนวณ gradient ตัวนี้ให้เราอัตโนมัติ ไม่ว่าโมเดลจะซับซ้อนแค่ไหน

(รายละเอียดเชิงลึกของ gradient descent อยู่ในหัวข้อ 0.3 — ที่นี่รู้แค่สัญชาตญาณก็พอ)

3. Autograd — หัวใจของ PyTorch ★

เมื่อ tensor มี requires_grad=True PyTorch จะ จดทุกการคำนวณ ที่เกิดกับมันไว้เป็น "กราฟ" พอเราสั่ง .backward() มันจะไล่กราฟย้อนกลับคำนวณ gradient ให้ทุกตัว

x = torch.tensor(2.0, requires_grad=True)

y = x**2 + 3*x          # y = x² + 3x   (PyTorch แอบจดสูตรไว้)

y.backward()              # ★ คำนวณ dy/dx ย้อนกลับ

print(x.grad)            # → tensor(7.)   # dy/dx = 2x+3 = 2(2)+3 = 7 ✓
ส่วนประกอบคืออะไร
requires_grad=True"ติดตาม gradient ของตัวนี้" (พารามิเตอร์โมเดลทุกตัวเป็นแบบนี้)
.backward()สั่งคำนวณ gradient ย้อนกลับจาก loss
.gradที่เก็บผล gradient ที่คำนวณได้
ข้อควรระวังใหญ่: .grad จะ บวกสะสม ทุกครั้งที่ backward() ไม่ลบให้เอง → ก่อน backward รอบใหม่ ต้อง เคลียร์ก่อนเสมอ (ในการเทรนจริงใช้ optimizer.zero_grad() — ดูส่วน 7) · ลืมข้อนี้ = โมเดลเทรนเพี้ยน เป็น bug คลาสสิก

4. Hello World ของการเรียนรู้ — ลดค่าฟังก์ชันด้วยมือ

ก่อนเทรนโมเดลจริง มาดู "หัวใจการเรียนรู้" แบบเปลือย ๆ: หาค่า x ที่ทำให้ (x−3)² น้อยสุด (คำตอบคือ x=3) โดยให้ autograd พาเราไป

x = torch.tensor(0.0, requires_grad=True)   # เริ่มมั่ว ๆ ที่ 0
lr = 0.1                                       # ขนาดก้าวเดิน (learning rate)

for step in range(30):
    loss = (x - 3)**2            # ความผิด (อยากให้น้อยสุด)
    loss.backward()                # คำนวณ d(loss)/dx
    with torch.no_grad():        # ปรับค่าโดยไม่ให้ autograd จด
        x -= lr * x.grad           # เดินสวนทาง gradient
    x.grad.zero_()                 # เคลียร์ gradient ก่อนรอบหน้า

print(x.item())     # → ~2.999   # เข้าใกล้ 3 ✓ โมเดล"เรียนรู้"แล้ว!
นี่คือทั้งหมดของ deep learning ในรูปย่อ: มี loss → หา gradient → เดินสวนทาง → ทำซ้ำ · ส่วนที่เหลือ (nn.Module, optimizer) แค่ทำให้ pattern นี้ สะดวกและขยายไปพารามิเตอร์ล้านตัวได้ เท่านั้นเอง

5. nn.Module — วิธี "ห่อ" โมเดล

โมเดลจริงมีพารามิเตอร์เป็นล้าน เราไม่จัดการเองทีละตัว แต่ห่อไว้ในคลาส nn.Module

import torch.nn as nn

class TinyNet(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(10, 32)    # ชั้นเชิงเส้น: 10 เข้า → 32 ออก
        self.fc2 = nn.Linear(32, 1)     # 32 → 1

    def forward(self, x):              # บอกว่าข้อมูลไหลยังไง
        x = torch.relu(self.fc1(x))    # ผ่านชั้น 1 + activation
        x = self.fc2(x)                # ผ่านชั้น 2
        return x

model = TinyNet()
print(sum(p.numel() for p in model.parameters()))   # นับพารามิเตอร์ทั้งหมด
ชิ้นส่วนหน้าที่
__init__ประกาศ "ชั้น" ที่โมเดลมี (เก็บพารามิเตอร์)
forwardนิยามว่าข้อมูลไหลผ่านชั้นยังไง (เรียกตอน model(x))
nn.Linear(a, b)ชั้นเชิงเส้น y = Wx + b (W, b เป็นพารามิเตอร์ที่เรียนรู้)
torch.reluactivation — ใส่ความไม่เชิงเส้นให้โมเดลเรียน pattern ซับซ้อนได้
เชื่อมกับงานเรา: โมเดล HazeNet ทั้งตัว (conditionally-linear operator) จะเป็น nn.Module หนึ่งตัว · forward ของมันจะรับ (C_t, ลม, ไฟ) → คืน C_{t+1} · ส่วน AFNO ที่ทำนายสัมประสิทธิ์ก็เป็น Module ย่อยข้างใน · ทุกอย่างประกอบจาก building block พวกนี้

6. Loss function — วัด "ความผิด"

loss_fn = nn.MSELoss()             # Mean Squared Error = เฉลี่ย (pred−true)²

pred = torch.tensor([2.0, 4.0])
true = torch.tensor([3.0, 3.0])
loss = loss_fn(pred, true)         # = mean([1, 1]) = 1.0
เชื่อมกับงานเรา: MSELoss คือพื้นฐานของการวัดว่าฝุ่นที่พยากรณ์ห่างจากของจริงแค่ไหน — มันคือ RMSE² นั่นเอง (RMSE = √MSE ที่คุณเจอใน 0.1) · เราจะใช้ตัวนี้ (อาจดัดแปลงเพิ่ม physics) เป็น loss หลักตอนเทรน

7. Optimizer + Training Loop ศักดิ์สิทธิ์ ★ จำให้ขึ้นใจ

optimizer คือตัวจัดการ "เดินสวนทาง gradient" ให้พารามิเตอร์ทุกตัว (แทนที่จะเขียน x -= lr*x.grad เองเหมือนส่วน 4)

opt = torch.optim.Adam(model.parameters(), lr=1e-3)
loss_fn = nn.MSELoss()

for epoch in range(num_epochs):
    for x, y in dataloader:          # ดึงข้อมูลทีละ batch
        opt.zero_grad()              ①  เคลียร์ gradient เก่า
        pred = model(x)              ②  forward: คำนวณคำทำนาย
        loss = loss_fn(pred, y)      ③  วัดความผิด
        loss.backward()             ④  หา gradient ของทุกพารามิเตอร์
        opt.step()                   ⑤  ปรับพารามิเตอร์ (เดินลงหุบเขา)
5 ขั้นนี้คือ "วงจรการเรียนรู้" ที่ทุกโมเดลในโลกใช้ — ท่องให้ได้:

1zero_grad เคลียร์ของเก่า → 2model(x) ทำนาย → 3loss_fn วัดผิด → 4backward หา gradient → 5step ปรับ

ลำดับห้ามสลับ + ห้ามลืม zero_grad: ถ้าลืม zero_grad() gradient จะสะสมข้ามรอบ (ดูคำเตือนส่วน 3) โมเดลจะเทรนพัง · Adam เป็น optimizer ที่นิยมสุด ปรับ learning rate ให้อัตโนมัติ ใช้ lr=1e-3 เป็นจุดเริ่มมาตรฐาน

9. ตัวอย่างเต็ม — เทรนโมเดลจริง (รันได้)

import torch, torch.nn as nn

# ข้อมูลของเล่น: เรียนฟังก์ชัน y = 2x + 1
X = torch.randn(200, 1)
Y = 2*X + 1 + 0.1*torch.randn(200, 1)   # มี noise นิดหน่อย

model   = nn.Linear(1, 1)                       # โมเดลง่ายสุด: หา w, b
opt     = torch.optim.Adam(model.parameters(), lr=0.05)
loss_fn = nn.MSELoss()

for epoch in range(200):
    opt.zero_grad()                     ①
    pred = model(X)                     ②
    loss = loss_fn(pred, Y)             ③
    loss.backward()                     ④
    opt.step()                          ⑤
    if epoch % 50 == 0:
        print(epoch, loss.item())

# ดูว่าโมเดลเรียนได้ w≈2, b≈1 ไหม
print(model.weight.item(), model.bias.item())   # → ~2.0, ~1.0 ✓

โมเดลนี้เริ่มจากค่าสุ่ม แล้ว "เรียนรู้" ว่า y = 2x+1 ด้วยตัวเอง ผ่านวงจร 5 ขั้น · เปลี่ยน nn.Linear เป็นโมเดลอะไรก็ได้ โครงเดิมไม่เปลี่ยน — นี่คือ template ที่คุณจะใช้เทรน HazeNet

12. .backward() ดิฟอะไร? — ตอบชัดๆ

หลายคนงงว่า .backward() "ดิฟอะไร" — คำตอบคือมันดิฟ loss ตาม weight เสมอ

สูตรที่ backward หา: 
∂loss/∂w   ← อ่านว่า "เดล loss เดล w"

แปลตรงๆ: "ถ้าขยับ w นิดเดียว → loss จะเปลี่ยนเท่าไร"

ถ้าสมการยาวหลายชั้น PyTorch ใช้ chain rule ไล่คูณทีละขั้น:

# สมมติมีโซ่:
y    = x * w        # ขั้น 1
z    = y + b        # ขั้น 2
loss = z ** 2       # ขั้น 3

# chain rule ไล่ย้อน:
# ∂loss/∂w = ∂loss/∂z × ∂z/∂y × ∂y/∂w
#           =   2z    ×    1   ×   x
# PyTorch คำนวณทั้งหมดนี้ให้อัตโนมัติ

loss.backward()
print(w.grad)   # = 2z × x  ← ผลจาก chain rule
สรุปสั้น: .backward() = สั่งให้ PyTorch คำนวณ chain rule ทั้งหมดแทนเรา · .grad = กล่องเก็บผลลัพธ์ · optimizer เปิดกล่องนี้แล้วปรับ weight ด้วย opt.step()
วงจรการเรียนรู้ทั้งหมดในภาพเดียว: 
opt.zero_grad()          ① เคลียร์กล่อง .grad เก่า
y_pred = model(x)        ② forward — คำนวณคำทำนาย + จด graph
loss = loss_fn(y_pred, y)③ วัดความผิด
loss.backward()          ④ หา ∂loss/∂w ของ weight ทุกตัว → เก็บใน .grad
opt.step()               ⑤ w = w − lr × w.grad  (เดินลงเขา)

13. โครงสร้าง TinyNet ละเอียด — nn.Module ให้อะไรฟรีบ้าง

เมื่อเขียน class TinyNet(nn.Module) = "ขอสืบทอดความสามารถจากแม่แบบ" — ได้ของฟรีทันที:

ฟังก์ชันทำอะไร
model.parameters()คืน weights ทั้งหมด (ให้ optimizer ใช้)
model.train()โหมดเทรน (เปิด dropout ฯลฯ)
model.eval()โหมดทดสอบ (ปิด dropout)
model.to(device)ย้ายทั้งโมเดลไป GPU/CPU
model(x)เรียก forward() อัตโนมัติ

ทำไม self. ต้องมีก่อนทุก layer?

# ถ้าไม่มี self. → เครื่องจักรวางบนพื้น แล้วหายหลัง __init__ จบ
def __init__(self):
    super().__init__()
    fc1 = nn.Linear(10, 32)   # ❌ ไม่มี self. → หาย → parameters() มองไม่เห็น
    self.fc1 = nn.Linear(10, 32) # ✅ เก็บในโมเดล → อยู่ตลอด

ทำไม model(x) ไม่ใช่ model.forward(x)?

PyTorch ไม่ได้เรียก forward() ตรงๆ — มันเรียกผ่าน __call__ ของ nn.Module ซึ่งมีระบบเพิ่มเติม (hooks สำหรับ debug ฯลฯ) · ให้เรียก model(x) เสมอ

14. nn.Linear(10, 32) — รับ 10 ออก 32 ยังไง ทำไม 352 ตัว

เริ่มจากกรณีง่ายสุด แล้วค่อยขยาย

กรณีที่ 1: 1 input → 1 output

y = x × w + b
# x = ตัวเลขเข้า, w = weight 1 ตัว, b = bias 1 ตัว
# พารามิเตอร์ = 1 + 1 = 2 ตัว

กรณีที่ 2: 3 input → 1 output

y = x1×w1 + x2×w2 + x3×w3 + b
# w = 3 ตัว, b = 1 ตัว → รวม 4 ตัว
# สูตร: (input × output) + output = 3×1 + 1 = 4

กรณีที่ 3: 3 input → 4 output (แก่นสำคัญ)

แต่ละ output ใช้ ชุด weight ของตัวเอง — คำนวณแยกกัน 4 รอบ:

output1 = x1×w11 + x2×w21 + x3×w31 + b1
output2 = x1×w12 + x2×w22 + x3×w32 + b2
output3 = x1×w13 + x2×w23 + x3×w33 + b3
output4 = x1×w14 + x2×w24 + x3×w34 + b4
# weights = 3×4 = 12, bias = 4 → รวม 16 ตัว
กลับมาที่ nn.Linear(10, 32): 
weights = 10 × 32 = 320 ตัว
bias    =      32 =  32 ตัว
              รวม = 352 ตัว  ✓

สูตรทั่วไป: nn.Linear(a, b) → พารามิเตอร์ = (a × b) + b

15. bias b — สุ่มมาแล้วปรับด้วยมั้ย?

ปรับด้วยครับ — b เป็นพารามิเตอร์เหมือน weight ทุกอย่าง มี requires_grad=True อัตโนมัติ

# backward() หา gradient ทั้ง W และ b พร้อมกัน
loss.backward()
# → ∂loss/∂W เก็บใน W.grad
# → ∂loss/∂b เก็บใน b.grad

# step() ปรับทั้งคู่พร้อมกัน
opt.step()
# → W = W − lr × W.grad
# → b = b − lr × b.grad

ทำไมต้องมี bias? — ถ้าไม่มี b เส้นต้องผ่านจุดศูนย์กำเนิดเสมอ ยืดหยุ่นน้อย

# ไม่มี b:  y = 2x     → ถ้า x=0 แล้ว y=0 เสมอ
# มี b:     y = 2x + 3 → ถ้า x=0 แล้ว y=3  (ขยับได้)
# เหมือน y = mx + c ของเส้นตรง — ถ้าไม่มี c เส้นต้องผ่านจุด (0,0)

16. ทำไมต้อง 32 output? ตัวเดียวไม่ได้หรอ?

ได้ — แต่โมเดลจะโง่มาก เพราะ output แต่ละตัวจับ pattern คนละชนิด

อุปมา: ทีมนักสืบ 
output=1  → นักสืบคนเดียว ดูข้อมูลทุกอย่างแล้วสรุปเลย
           → มองแค่ "รายได้สูง = ซื้อ" ตื้นๆ

output=32 → ทีม 32 คน แต่ละคนเชี่ยวชาญคนละด้าน
           → คนที่ 1: ดูหนี้สิน
           → คนที่ 2: ดูอายุ+รายได้
           → คนที่ 3: ดูภูมิภาค+ราคาย่าน
           → ...
           → ชั้นสุดท้ายรวบความเห็น 32 ชิ้น → สรุป

ทำไมแต่ละ output ถึงจับ pattern ต่างกัน?

เพราะ สุ่ม weight ต่างกันตอนเริ่ม + relu ทำให้แยกตัวกัน

# weight สุ่มต่างกัน → แต่ละ output "มองข้อมูลคนละมุม"
y1 = x1×0.23 + x2×0.87 + ...   # ชุดนี้สนใจ feature แบบนึง
y2 = x1×0.95 + x2×0.02 + ...   # ชุดนี้สนใจ feature อีกแบบ

# หลัง relu บางตัวดับ (=0) บางตัวสว่าง (>0) ต่างกันต่อข้อมูลแต่ละชุด
# → แต่ละ output เชี่ยวชาญ pattern ต่างกันโดยธรรมชาติ ไม่ใช่เราสั่ง
ถ้าไม่มี relu: ชั้น 2 ชั้น (หรือ N ชั้น) รวมกันได้เป็นชั้นเดียวเสมอ → มี 32 output ก็ไม่ต่างจาก 1 output · relu คือสิ่งที่ทำให้หลายชั้นมีความหมาย

จะรู้ได้ยังไงว่าควรใช้กี่ output?

หลักคิดรายละเอียด
ดูงานคล้ายกันคนอื่นใช้เท่าไร → เริ่มตรงนั้น
ดูปริมาณข้อมูลข้อมูลน้อย = โมเดลเล็ก, ข้อมูลเยอะ = โมเดลใหญ่ได้
วัดบน validation setลอง 16 → 32 → 64 → เลือกที่ val loss ต่ำสุด
ใช้เลขยกกำลัง 232, 64, 128, 256 — GPU คำนวณเร็วกว่า

ไม่มีสูตรตายตัว — ทุกคนในวงการก็ลองแล้ววัด

17. โมเดลใหญ่ ≠ ฉลาดขึ้นเสมอ — Overfitting + Vanishing Gradient

ปัญหา 1: Overfitting — ท่องจำแทนที่จะเข้าใจ

# เด็กที่จำเฉลยข้อสอบเก่า:
ข้อสอบเก่า → ทำได้ 100%  ✓
ข้อสอบใหม่ → ทำได้ 20%   ✗  ← ท่องจำ ไม่ได้เข้าใจ

# โมเดลใหญ่เกิน + ข้อมูลน้อย:
training loss   = 0.001  (จำทุก data point)
validation loss = 2.500  (ข้อมูลใหม่ทำนายพลาด)

ปัญหา 2: Vanishing Gradient — gradient หายกลางทาง

ยิ่งชั้นลึก gradient ที่ไล่ย้อนกลับยิ่งเล็กลงเรื่อยๆ จนชั้นแรกๆ ไม่ได้รับ gradient และไม่เรียนอะไรเลย

# เหมือนโทรศัพท์สายยาว 100 คน:
คนที่ 100 → 99 → 98 → ... → 1
ข่าวจางหายและเพี้ยนก่อนถึงต้น

# gradient ก็เหมือนกัน:
loss → ชั้น 10 (เต็ม) → ชั้น 5 (เล็กลง) → ชั้น 1 (แทบศูนย์)
→ ชั้นแรกไม่ได้เรียน
เชื่อมกับ HazeNet: เราอยู่บน A100 เครื่องเดียว 15 ชั่วโมง + สถานีวัด PM2.5 ไทยมีไม่มาก → โมเดลใหญ่เกินทำให้ overfitting กับสถานีไม่กี่สิบจุด · เป้าหมายคือหา "จุดพอดี" ไม่ใช่ใหญ่ที่สุด

18. Adam Optimizer — ทำงานยังไงเบื้องหลัง

ปัญหาของ SGD ที่ Adam แก้

ปัญหา SGDอาการ
ก้าวขนาดเดิมทุก weightweight บางตัวเรียนช้า บางตัวกระโดดข้ามหุบ
ทิศสั่นไปมาgradient สลับบวก/ลบ → เดินซ้ายขวา ไปช้า

Adam แก้ด้วยการจำ "ประวัติ" 2 อย่าง

# m = ค่าเฉลี่ยทิศทาง (Momentum)  → แก้การสั่น
# v = ค่าเฉลี่ยขนาด²  (Variance)  → แก้ขนาดก้าว

# ทั้งสองเริ่มที่ 0 และอัปเดตทุก step:
m = 0.9  × m_เก่า + 0.1  × gradient   # 90% จำอดีต + 10% ปัจจุบัน
v = 0.999 × v_เก่า + 0.001 × gradient² # จำขนาด gradient ที่ผ่านมา
ทำไม m แก้การสั่น? 
รอบ 1: gradient = +10  → m =  0 + 1    =  1.0
รอบ 2: gradient = -10  → m = 0.9 - 1   = -0.1  ← เกือบศูนย์ ไม่สั่น!
รอบ 3: gradient = +10  → m = -0.09 + 1 =  0.91

# m เฉลี่ยออก → ทิศทางนิ่ง เดินตรงกว่า SGD มาก
ทำไม / √v แก้ขนาดก้าว? 
gradient ใหญ่มาตลอด → v ใหญ่ → √v ใหญ่ → หารแล้วก้าวเล็กลง ✓
gradient เล็กมาตลอด → v เล็ก → √v เล็ก → หารแล้วก้าวใหญ่ขึ้น ✓
# ปรับขนาดก้าวอัตโนมัติต่อ weight แต่ละตัว!

สูตรอัปเดต weight ของ Adam (ครบ)

# ขั้น 1: gradient ปกติจาก backward()
g = ∂loss/∂w

# ขั้น 2: อัปเดต m และ v
m = 0.9   × m + 0.1   × g
v = 0.999 × v + 0.001 × g²

# ขั้น 3: แก้ bias (รอบแรกๆ m,v เริ่มที่ 0 จึงเล็กเกิน)
m̂ = m / (1 - 0.9^t)    # t = รอบที่เท่าไร
v̂ = v / (1 - 0.999^t)

# ขั้น 4: ปรับ weight
w = w - lr × m̂ / (√v̂ + 1e-8)
# 1e-8 กันหารศูนย์
ชื่อ Adam มาจาก: Adaptive Moment Estimation · Adaptive = ปรับขนาดก้าวเองได้ · Moment = m (moment 1) และ v (moment 2)
SGDAdam
ทิศทางgradient ล้วนๆ สั่นเฉลี่ยจากอดีต นิ่งกว่า
ขนาดก้าวคงที่ทุก weightปรับอัตโนมัติต่อ weight
จำอดีต❌✅ (m และ v)
RAMน้อยมากกว่า (เก็บ m,v เพิ่ม)
ความนิยมใช้เฉพาะบางงานdefault สำหรับงานส่วนใหญ่
แต่ละ weight มี m และ v ของตัวเอง — Adam จำประวัติแยกกันทุกตัว · โมเดลที่มี 1 ล้าน weight ก็มี m และ v 1 ล้านคู่ → RAM เพิ่ม 2 เท่าของ weight

19. ภาพรวม PyTorch เบื้องหลังทั้งหมด — จบในที่เดียว

3 ระบบใหญ่ที่ทำงานเบื้องหลัง:

ระบบ 1: Computation Graph — สมุดจดการคำนวณ

# ทุก operation บน tensor ที่มี requires_grad=True
# PyTorch แอบสร้าง node และเชื่อมเป็น graph

x ──┐
    ├──[MulBackward]──→ y ──┐
w ──┘                        ├──[AddBackward]──→ loss
b ───────────────────────────┘

# แต่ละ node จำ: input มาจากไหน + สูตรอนุพันธ์ของตัวเอง

ระบบ 2: Backward Pass — ไล่ย้อนด้วย chain rule

# loss.backward() → เดินย้อนกราฟจากขวาไปซ้าย
loss → [AddBackward] → y.grad, b.grad
     → [MulBackward] → w.grad

# ผลสุดท้ายเก็บใน .grad ของทุก parameter

ระบบ 3: Optimizer — ปรับ weight ตาม .grad

# opt.step() วนไปทุก parameter:
# SGD:  w = w - lr × w.grad
# Adam: w = w - lr × m̂ / (√v̂ + ε)  (ปรับขนาดก้าวด้วย)
