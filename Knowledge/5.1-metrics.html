1. ทำไม metric เดียวไม่พอ — "ค่าเฉลี่ยกลบหายนะ" ⭐

ลองดูตัวอย่างที่อันตราย: โมเดล 2 ตัวมี RMSE เท่ากัน แต่คุณภาพต่างกันคนละโลก

สถานการณ์วันปกติ (PM2.5≈30)วันวิกฤต (PM2.5จริง=180)RMSE รวมใช้ได้จริง?
โมเดล Aแม่นมากทำนาย 60 (พลาดวันอันตราย!)ต่ำ ✓อันตราย — พลาดวันที่สำคัญสุด
โมเดล Bแกว่งเล็กน้อยทำนาย 165 (จับได้!)ใกล้เคียง Aดี — จับวันวิกฤตได้
ประเด็น: วันวิกฤตมีไม่กี่วันต่อปี → มันแทบไม่ขยับ RMSE รวม (ถูกค่าเฉลี่ยของวันปกติกลบ) · แต่มัน คือวันที่โรงเรียนต้องสั่งหยุด คนต้องใส่หน้ากาก · ถ้าวัดด้วย RMSE อย่างเดียว เราจะเลือกโมเดล A ที่อันตราย · นี่คือเหตุผลที่ต้องมี event metric แยกต่างหาก

2. ตระกูล ① — Regression: ค่าตรงไหม

วัด "ตัวเลข PM2.5 ที่ทำนายห่างจากจริงแค่ไหน" · 3 ตัวพื้นฐาน (หน่วยเดียวกับ PM2.5 = μg/m³ ตีความง่าย):

MAE — ค่าผิดพลาดเฉลี่ย (เข้าใจง่ายสุด)

MAE = (1/N) Σ |ŷ − y|

เฉลี่ยขนาดความผิดพลาด · "โดยเฉลี่ยพลาดไป ±X μg/m³" · ทนต่อ outlier (ไม่ลงโทษค่าพลาดใหญ่เป็นพิเศษ)

RMSE — รากของกำลังสองเฉลี่ย (ลงโทษพลาดใหญ่)

RMSE = √[ (1/N) Σ (ŷ − y)² ]

ยกกำลังสองก่อน → พลาดมาก ๆ ถูกลงโทษหนักกว่า · เหมาะกับงานเราเพราะ "พลาดวันวิกฤต 100 หน่วย" ควรเจ็บกว่า "พลาดวันปกติ 10 หน่วย ×10 ครั้ง"

Bias — เอนเอียงเชิงระบบ

Bias = (1/N) Σ (ŷ − y)

ไม่ใส่ค่าสัมบูรณ์ → บอกว่าโมเดล ทำนายสูงไปหรือต่ำไปเป็นระบบ · bias เป็นบวก = ทำนายเกินจริงตลอด · สำคัญเพราะถ้า "ต่ำไปตลอด" จะอันตราย (ประเมินอันตรายต่ำเกิน)

ใช้ 3 ตัวคู่กัน: MAE (ภาพรวมเข้าใจง่าย) + RMSE (จับพลาดใหญ่) + Bias (เอนเอียงทิศไหน) · รายงานทั้งสามเสมอ — ตัวเดียวเล่าเรื่องไม่ครบ

3. R² และ ACC — วัด "ทักษะเชิงรูปแบบ"

R² (สัมประสิทธิ์การตัดสินใจ): โมเดลอธิบายความแปรปรวนของข้อมูลได้กี่ % · R²=1 สมบูรณ์แบบ, R²=0 = เท่ากับเดาค่าเฉลี่ย · (FastCTM รายงาน PM2.5 R²≈0.80 — จำจากไฟล์วิเคราะห์ได้ไหม)

ACC (Anomaly Correlation Coefficient): มาตรฐานวงการพยากรณ์อากาศ · วัด correlation ของ "ส่วนต่างจากค่าปกติ (anomaly)" ไม่ใช่ค่าดิบ:

ACC = corr( ŷ − climatology , y − climatology )
ทำไม ACC ดีกว่า correlation ดิบ: ถ้าวัด correlation ของค่าดิบ โมเดลจะได้คะแนนสูงฟรี ๆ แค่เพราะ "ทายตามฤดูกาล" (หน้าแล้งฝุ่นเยอะ หน้าฝนฝุ่นน้อย — ใครก็รู้) · ACC หักส่วนที่เป็น climatology ออกก่อน → วัดเฉพาะ "ทักษะที่เกินกว่าการรู้ฤดูกาล" ซึ่งคือสิ่งที่โมเดลต้องพิสูจน์จริง ๆ

4. ตระกูล ② — Event/Categorical: จับวันวิกฤตได้ไหม ⭐

เปลี่ยนจาก "ค่าตรงไหม" เป็น "เตือนถูกไหม" · ตั้ง เกณฑ์ ก่อน (เช่น PM2.5 > 75 μg/m³ = อันตราย — ค่าจริงให้อิงมาตรฐาน คพ./WHO) แล้วแปลงทั้งทำนายและจริงเป็น "เกิน/ไม่เกิน" → ได้ 4 กรณี:

จริง: เกินเกณฑ์จริง: ไม่เกิน
ทำนาย: เกินHit
เตือนถูก ✓		False Alarm
เตือนเก้อ
ทำนาย: ไม่เกินMiss
พลาดวันอันตราย ✗✗		Correct Neg
ปกติก็บอกปกติ
ไม่ใช่ทุก error เท่ากัน: Miss (ฝุ่นมาแต่ไม่เตือน) อันตรายกว่า False Alarm (เตือนแต่ไม่มา) มาก · การแยก 4 กรณีทำให้เรา "เห็นชนิดของความผิดพลาด" ไม่ใช่แค่จำนวนรวม — และเลือกโมเดลที่ Miss น้อยได้

5. POD / FAR / CSI — สรุปตาราง contingency เป็นตัวเลข

Metricสูตรตอบคำถามดี=
POD (Probability of Detection / hit rate)Hits / (Hits + Misses)"วันอันตรายจริง เราจับได้กี่ %"สูง (→1)
FAR (False Alarm Ratio)FA / (Hits + FA)"ที่เราเตือน ผิดกี่ %"ต่ำ (→0)
CSI (Critical Success Index / Threat Score)Hits / (Hits + Misses + FA)"ภาพรวมการจับ event (ไม่นับ correct neg)"สูง (→1)
ทำไม CSI เหมาะกับเหตุการณ์หายาก: วันวิกฤตมีน้อย → "correct negative" (ปกติก็บอกปกติ) มีเยอะมหาศาล · ถ้าใช้ accuracy ธรรมดา โมเดลที่ "บอกปกติตลอด" จะได้คะแนน 95%+ ทั้งที่ไร้ประโยชน์ · CSI ตัด correct negative ทิ้ง → วัดเฉพาะความสามารถจับ event จริง · นี่คือ metric หลักสำหรับ "การเตือนภัย"
POD กับ FAR ต้องดูคู่กัน: ทำให้ POD=1 ง่ายมาก — แค่ "เตือนทุกวัน" · แต่ FAR จะพุ่ง (เตือนเก้อตลอด คนเลิกเชื่อ) · เกณฑ์ที่ดีคือ POD สูง + FAR ต่ำ พร้อมกัน = CSI สูง · การเลือก threshold คือการ trade ระหว่างสองตัวนี้

6. ตระกูล ③ — Onset-Lead Time: จุดขายของ HazeNet ⭐

นี่คือ metric ที่ทำให้งานเรา "มีค่าเชิงปฏิบัติ" จริง · ไม่ใช่แค่ "ทายวันนี้ถูกไหม" แต่ "เตือนการมาถึงของ episode ฝุ่นได้ล่วงหน้ากี่ชั่วโมง"

วิธีวัด: หา "เวลาที่ค่าจริงข้ามเกณฑ์ครั้งแรกของ episode" และ "เวลาที่โมเดลทำนายว่าจะข้าม" → ผลต่าง = lead time · รายงานเป็น lead time เฉลี่ยของ episode ที่จับได้ (เช่น "เตือนล่วงหน้าเฉลี่ย 18 ชม.")

ทำไมนี่คือจุดขาย: RMSE ต่ำไม่ได้แปลว่ามีประโยชน์ — แต่ "เตือนล่วงหน้า 1 วันได้แม่น" แปลว่า โรงเรียนวางแผนหยุดได้ คนเตรียมหน้ากากทัน เกษตรกรเลื่อนเผาได้ · onset-lead เปลี่ยนงานจาก "exercise วิชาการ" เป็น "เครื่องมือเตือนภัยจริง" · ต้องรายงานคู่กับ "จับ episode ได้กี่ % (POD)" เสมอ — เตือนเร็วแต่พลาดครึ่งหนึ่งก็ไม่ดี

7. Skill vs Lead Time — ยิ่งทำนายไกล ยิ่งยาก

ความแม่นไม่ใช่ตัวเลขเดียว — มันลดลงตามระยะพยากรณ์ · ต้องรายงาน metric เป็น "เส้นโค้งตามชั่วโมงข้างหน้า" ไม่ใช่เลขเดียว:

สิ่งที่ต้องเคลม: ไม่ใช่ "เราแม่น" แต่คือ "เราแม่นกว่า baseline จนถึง t+H ชั่วโมง" · จุดที่เส้นเราตัดกับ baseline = ขีดจำกัดความมีประโยชน์ · นี่คือตัวเลขที่ตอบกรรมการได้ว่า "ใช้พยากรณ์ได้ถึงกี่ชั่วโมง"

8. Skill Score — ตัวเลขดิบไม่มีความหมายถ้าไม่เทียบ baseline

"RMSE = 15 μg/m³" ดีหรือแย่? ตอบไม่ได้ถ้าไม่มีตัวเทียบ · ต้องเทียบกับ baseline ง่าย ๆ แล้วรายงานเป็น "ดีขึ้นกี่ %":

Skill Score = 1 − (errorโมเดล / errorbaseline)
  • Persistence ("พรุ่งนี้เหมือนวันนี้") — baseline ที่แกร่งน่ากลัวสำหรับระยะสั้น
  • Climatology ("ใช้ค่าเฉลี่ยตามฤดูกาล") — baseline สำหรับระยะยาว
กับดักที่กรรมการจับได้ทันที: เคลม "R²=0.8 เจ๋ง!" แต่ถ้า persistence ได้ R²=0.82 แปลว่าโมเดลเรา แย่กว่าการทายว่า "เหมือนเมื่อวาน" · ตัวเลขดิบสวยไม่พอ — ต้องชนะ baseline · (รายละเอียด baseline ทั้งหมดอยู่ใน 5.3) · เลือก baseline ให้ honest อย่าเลือกตัวอ่อน ๆ มาชนะ

9. กับดักการวัด + ตะกร้า metric ของ HazeNet

กับดักที่ต้องเลี่ยง

  • คิด metric บนจุดที่ target หาย: PM2.5 มีเฉพาะสถานี + บางชั่วโมงขาด → คิดเฉพาะจุดที่มีค่าจริง (masked, จาก 4.4)
  • เฉลี่ยข้ามสถานีมั่ว ๆ: สถานีกระจุกในเมือง → เฉลี่ยตรง ๆ จะเอนเข้าเมือง · พิจารณาเฉลี่ยถ่วงพื้นที่/รายงานรายสถานี
  • double penalty (สำหรับภาพ): ถ้าทำนายกลุ่มฝุ่น "ถูกแต่เลื่อนไป 1 ช่อง" จะโดนลงโทษ 2 เด้ง (ทั้งที่ทำนาย + ที่ควรมี) · บรรเทาด้วย neighborhood/CRPS (โยง 3.9)
  • class imbalance: วันวิกฤตน้อย → อย่าใช้ accuracy ดิบ ใช้ CSI

ตะกร้า metric ที่ HazeNet จะรายงาน (ครบ 3 ตระกูล)

ตระกูลmetricตอบอะไรกรรมการ
① regressionMAE, RMSE, Bias, R², ACCค่าทำนายแม่นและไม่เอนเอียง
② eventPOD, FAR, CSI (ที่เกณฑ์อันตราย)จับวันวิกฤตได้ เตือนไม่เก้อ
③ timingonset-lead time (+ skill-vs-leadtime)เตือนล่วงหน้าได้นานแค่ไหน (จุดขาย)
เทียบskill score เทียบ persistence/climatologyดีกว่าของง่ายจริง (5.3)
uncertaintyCRPS (จาก 3.9)ความไม่แน่นอน calibrated
สรุปจุดยืน: "แม่น (regression) + จับวันอันตรายได้ (event) + เตือนล่วงหน้าได้ (onset-lead) + ชนะ baseline (skill score) + บอกความมั่นใจได้ (CRPS)" — 5 ด้านนี้คือคำตอบครบชุดเวลากรรมการถามว่า "ดีจริงไหม พิสูจน์มา"

10. Cheat sheet + แบบฝึกหัด

Cheat sheet

  • MAE เฉลี่ย|พลาด| · RMSE ลงโทษพลาดใหญ่ · Bias เอนสูง/ต่ำ — รายงานทั้งสาม
  • ACC = skill ที่เกินกว่ารู้ฤดูกาล (หัก climatology)
  • POD จับได้กี่% · FAR เตือนเก้อกี่% · CSI ภาพรวม event (ตัด correct neg) — ดูคู่กัน
  • onset-lead = เตือนล่วงหน้ากี่ ชม. — จุดขาย · คู่กับ POD
  • skill score = ต้องชนะ persistence/climatology ถึงจะมีความหมาย
  • วัดเฉพาะจุด target จริง (masked) · ระวัง double penalty/imbalance

❓ แบบฝึกหัด

  1. โมเดล 2 ตัว RMSE เท่ากัน จะแยกว่าตัวไหนดีกว่าสำหรับ "เตือนภัย" ด้วย metric ไหน?
  2. ทำไม accuracy ดิบใช้ไม่ได้กับ "วันวิกฤต" และต้องใช้ CSI แทน?
  3. POD=1.0 อย่างเดียวพิสูจน์ว่าโมเดลดีไหม? เพราะอะไร?
  4. ACC ต่างจาก correlation ของค่าดิบยังไง ทำไมถึงยุติธรรมกว่า?
  5. เคลม "R²=0.80" พอไหมที่จะบอกว่าโมเดลเก่ง? ต้องเทียบกับอะไร?
เฉลย
  1. event metric (CSI/POD ที่เกณฑ์อันตราย) + onset-lead — RMSE กลบความต่างที่วันวิกฤต
  2. วันวิกฤตหายาก → correct negative ท่วม → "บอกปกติตลอด" ได้ accuracy สูงปลอม · CSI ตัด correct neg ทิ้ง
  3. ไม่ — แค่ "เตือนทุกวัน" ก็ได้ POD=1 แต่ FAR พุ่ง · ต้องดู POD+FAR (หรือ CSI) คู่กัน
  4. ACC หัก climatology ออกก่อน → ไม่ให้คะแนนฟรีจากการทายตามฤดูกาล วัดเฉพาะทักษะที่เกินกว่านั้น
  5. ไม่พอ — ต้องเทียบ persistence/climatology · ถ้า persistence ได้ 0.82 แปลว่าเราแย่กว่าทายว่า "เหมือนเมื่อวาน" (skill score ติดลบ)

✅ เช็กลิสต์ก่อนไป 5.2

  • ☐ คำนวณ MAE/RMSE/Bias/R²/ACC บน test ได้ (masked)
  • ☐ สร้างตาราง contingency + POD/FAR/CSI ที่เกณฑ์อันตราย
  • ☐ วัด onset-lead time ของ episode + เส้น skill-vs-leadtime
  • ☐ รายงานทุก metric เป็น skill score เทียบ persistence
  • ☐ เข้าใจกับดัก double penalty / imbalance / target ที่หาย
