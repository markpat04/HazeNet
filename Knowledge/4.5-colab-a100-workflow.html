1. งบ A100 — ทำเลขก่อนกดรัน ★

ก่อนเทรนจริงต้องประเมิน "รันนี้กินกี่ชั่วโมง" ให้เป็น ไม่งั้นจะรู้ตัวอีกทีตอนงบหมดกลางคัน · สูตรคร่าว ๆ:

เวลา ≈ (จำนวน sample ÷ batch) × วินาที/step × จำนวน epoch ÷ 3600

ตัวอย่าง: 8,000 sample, batch 4 → 2,000 step/epoch · ถ้า ~0.5 วิ/step → 1,000 วิ/epoch ≈ 0.28 ชม./epoch · เทรน 30 epoch ≈ 8.3 ชม. · เหลือ buffer ~7 ชม. สำหรับ debug/ลองใหม่

วินัยงบ: วัด "วินาที/step" จากการรันสั้น ๆ 50 step ก่อน → คูณกลับเป็นชั่วโมง → ตัดสินใจว่าคุ้มไหมก่อนปล่อยยาว · อย่าเดา · 1 รันยาวพลาด = เสียงบไป 1/3 ของทั้งโปรเจกต์

2. VRAM เต็มด้วยอะไร — รู้ศัตรูก่อนสู้

A100 มี VRAM ~40GB · เวลา OOM ต้องรู้ว่าอะไรกิน · 4 ก้อนหลัก:

params
grads
optimizer
activations
ก้อนคืออะไรขนาดขึ้นกับ
Parametersน้ำหนักโมเดลจำนวนพารามิเตอร์
Gradientsเกรเดียนต์ (เท่า params)จำนวนพารามิเตอร์
Optimizer statesAdam เก็บ ~2 เท่า params (m, v)จำนวนพารามิเตอร์
Activationsค่ากลางทุกชั้น (เก็บไว้ทำ backward)batch × ความละเอียด × ความลึก × ความยาว rollout
กุญแจ: สำหรับโมเดลภาพ/spatiotemporal อย่างเรา activations คือก้อนที่ใหญ่ที่สุดและคุมได้ · มันโตตาม batch, ขนาดภาพ, และ ความยาว rollout (3.4) · ทุกเทคนิคลด VRAM ในบทนี้ส่วนใหญ่คือการ "ลด activations"

3. Mixed Precision (AMP) — ของฟรีที่ต้องเปิดเสมอ ★

ปกติเลขเป็น fp32 (32-bit) · mixed precision คำนวณบางส่วนด้วย 16-bit → ใช้ VRAM ~ครึ่งเดียว + เร็วขึ้น (A100 มี tensor core เร่ง 16-bit) · แทบไม่เสียความแม่น

fp32 — activations เต็ม ๆ
bf16 — ครึ่งเดียว
ว่าง

บน A100 แนะนำ bf16 (bfloat16) — ช่วงเลขกว้างเท่า fp32 จึงเสถียร ไม่ต้องใช้ GradScaler ให้ยุ่ง:

for X, y in loader:
    X, y = X.cuda(), y.cuda()
    with torch.autocast("cuda", dtype=torch.bfloat16):   # ← เปิด AMP
        pred = model(X)
        loss = loss_fn(pred, y)
    opt.zero_grad()
    loss.backward()
    opt.step()
ทำก่อนเลย: เปิด bf16 เป็นค่าเริ่มต้นของทุกการเทรน · เป็นวิธีลด VRAM + เพิ่มความเร็วที่ "แทบไม่มีข้อเสีย" → ได้ทั้งประหยัด VRAM และประหยัด A100-ชม.พร้อมกัน
หมายเหตุ fp16 vs bf16: ถ้าใช้ fp16 (เช่นบน GPU เก่า) ต้องมี GradScaler กัน gradient underflow · บน A100 ใช้ bf16 ตัดปัญหานี้ทิ้งได้เลย

4. Gradient Checkpointing — แลกเวลาเพื่อหน่วยความจำ ★ สำคัญต่อ rollout

ปกติ backward ต้องเก็บ activations ของ ทุกชั้น ไว้ → กิน VRAM มหาศาลเมื่อ rollout ยาว · gradient checkpointing = ไม่เก็บ activations กลางทาง แต่ คำนวณใหม่ตอน backward → ลด VRAM ลงมาก แลกกับเวลาเพิ่ม ~20–30%

from torch.utils.checkpoint import checkpoint
# ใน rollout ยาว: ห่อแต่ละ step ด้วย checkpoint → เก็บแค่ state ไม่เก็บ activations
state = init_state
for t in range(rollout_len):
    state = checkpoint(model.step, state, forcings[t], use_reentrant=False)
เชื่อมกับงานวิจัยที่อ่านมา: ทั้งไฟล์ adjoint chemistry emulator และ NeuralGCM ชี้ว่า backprop ผ่าน rollout ยาว = ระเบิดหน่วยความจำ · gradient checkpointing คือคำตอบมาตรฐาน — ทำให้เราเทรน rollout หลาย step บน A100 เดียวได้ · นี่คือเทคนิคที่ทำให้ rollout curriculum (3.4) เป็นไปได้จริงในงบเรา

5. Gradient Accumulation — แกล้งว่า batch ใหญ่

บางทีอยากได้ batch ใหญ่ (เทรนนิ่งกว่า) แต่ใส่ทีเดียวไม่พอ VRAM · accumulation = สะสม gradient จากหลาย batch ย่อยก่อนค่อย step() ครั้งเดียว → ได้ผลเท่า batch ใหญ่ โดยใช้ VRAM เท่า batch เล็ก:

accum = 4                         # batch จริง 2 → เสมือน 8
opt.zero_grad()
for i, (X, y) in enumerate(loader):
    with torch.autocast("cuda", dtype=torch.bfloat16):
        loss = loss_fn(model(X.cuda()), y.cuda()) / accum
    loss.backward()
    if (i+1) % accum == 0:
        opt.step(); opt.zero_grad()
เมื่อไรใช้: เมื่อ batch ที่พอ VRAM เล็กจน training ไม่นิ่ง (loss กระโดด) · accumulation ให้ effective batch ใหญ่ขึ้นโดยไม่เพิ่ม VRAM · แลกกับเวลาต่อ optimizer step ที่นานขึ้น

③ เดบั๊กบน "ของถูก" ก่อน: เขียน/ทดสอบโค้ดบน CPU หรือ GPU ฟรี (T4) ด้วย datacube จิ๋ว · เปลี่ยนมา A100 เฉพาะตอนเทรนจริง เท่านั้น

บาปที่ทำงบหมดเร็วสุด: เปิด A100 ทิ้งไว้ตอนนั่งแก้ bug / โหลดข้อมูล / นั่งคิด · A100-ชม.เดินตลอดที่ runtime เปิด ไม่ว่าจะคำนวณหรือไม่ · เปิด A100 ตอนพร้อมเทรนแล้วเท่านั้น แล้วปิดทันทีที่เสร็จ

8. DataLoader ไม่ให้ GPU "หิว"

ถ้าโหลดข้อมูลช้ากว่าที่ GPU คำนวณ → GPU นั่งรอข้อมูล = เสีย A100-ชม.ฟรี ๆ · จูน DataLoader (ต่อจาก 4.4):

loader = torch.utils.data.DataLoader(
    ds, batch_size=4, shuffle=True,
    num_workers=2,            # โหลดขนานหลาย process
    pin_memory=True,          # ส่งเข้า GPU เร็วขึ้น
    persistent_workers=True,  # ไม่ตั้ง worker ใหม่ทุก epoch
    prefetch_factor=2)         # เตรียม batch ถัดไปล่วงหน้า
วิธีรู้ว่าใคร bottleneck: ดู GPU utilization (nvidia-smi) — ถ้าต่ำ ๆ แกว่ง = GPU หิว (ติดที่ data) → เพิ่ม workers / ปรับ chunk Zarr (4.2) · ถ้าเต็ม ~100% = ติดที่ compute (ถูกแล้ว) · วัดเสมอ อย่าเดา
# เช็ค VRAM ที่ใช้สูงสุด (หา OOM ก่อนมันเกิด)
print(torch.cuda.max_memory_allocated()/1e9, "GB")

10. Cheat sheet + แบบฝึกหัด

Cheat sheet — ลด VRAM / ประหยัดงบ

  • เปิด bf16 เสมอ (autocast) → VRAM ครึ่ง + เร็วขึ้น ฟรี ๆ
  • gradient checkpointing สำหรับ rollout ยาว → แลกเวลา ลด VRAM มาก
  • accumulation → effective batch ใหญ่ขึ้นโดย VRAM เท่าเดิม
  • ลดจริง: batch → tile → rollout → res/channel (ตามลำดับกระทบ)
  • checkpoint ลง Drive + resume — กัน session หลุด
  • เดบั๊กบน CPU/T4 · A100 ตอนเทรนจริงเท่านั้น · ปิดทันทีที่เสร็จ
  • วัดก่อนรันยาว: วินาที/step → ประเมินชั่วโมง · GPU util เช็ก bottleneck

❓ แบบฝึกหัด

  1. ใน VRAM ก้อนไหนใหญ่สุดและคุมได้สำหรับโมเดล spatiotemporal? ขึ้นกับอะไร?
  2. ทำไมบน A100 แนะนำ bf16 มากกว่า fp16?
  3. gradient checkpointing แลกอะไรกับอะไร และสำคัญต่อ rollout ยังไง?
  4. อยาก effective batch = 16 แต่ VRAM พอแค่ 4 — ทำยังไง?
  5. ทำไม "เดบั๊กบน A100" ถึงเป็นการผลาญงบ?
เฉลย
  1. Activations — ขึ้นกับ batch × ความละเอียด × ความลึก × ความยาว rollout
  2. bf16 มีช่วงเลขกว้างเท่า fp32 → เสถียร ไม่ต้องใช้ GradScaler (fp16 ต้องมีกัน underflow)
  3. แลก "เวลา (คำนวณ activations ใหม่)" เพื่อ "ลด VRAM" → ทำให้ backprop ผ่าน rollout ยาวได้บน A100 เดียว (โยง adjoint/NeuralGCM/3.4)
  4. gradient accumulation: batch 4 สะสม gradient 4 ครั้งก่อน step (4×4=16)
  5. A100-ชม.เดินตลอดที่ runtime เปิด ไม่ว่าจะคำนวณหรือนั่งแก้ bug → ควรเดบั๊กบน CPU/T4 แล้วเปิด A100 เฉพาะตอนเทรน

✅ เช็กลิสต์ก่อนกดเทรนจริง (เฟส 3)

  • ☐ ผ่าน overfit test (loss ลงใกล้ 0 บน 1 batch)
  • ☐ เปิด bf16 + (ถ้าจำเป็น) checkpointing/accumulation
  • ☐ วัดวินาที/step แล้ว → ประเมินชั่วโมงรวม ≤ งบที่เหลือ
  • ☐ checkpoint→Drive + resume ทำงานจริง (ทดสอบ kill แล้วต่อได้)
  • ☐ GPU util สูง (ไม่หิวข้อมูล)
  • ☐ บันทึก hyperparams/config ของรันนี้ไว้
