3.3 — U-Net

สถาปัตยกรรม encoder-decoder ที่เห็นทั้งภาพรวมและรายละเอียด — ทางเลือก/ส่วนผสมของ backbone

ต่อจาก 3.2 (AFNO) · บทนี้เทียบ U-Net กับ AFNO และดูว่าผสมกันได้ยังไง · ผูกกับ HazeNet ตลอด

ทำไมต้องรู้จัก U-Net ทั้งที่เลือก AFNO แล้ว: AFNO (3.2) เก่งเรื่อง global (เห็นทั้งแผนที่) แต่เพราะมันเก็บแค่ความถี่ต่ำ (low modes) มันอ่อนเรื่องรายละเอียดเล็ก/ขอบคม · U-Net เก่งตรงข้าม — จับรายละเอียดเชิงพื้นที่ละเอียดได้ดี (ขอบกลุ่มควัน, ความต่างระดับจังหวัด) · ในงานจริง backbone ที่ดีมักเป็นลูกผสม: AFNO ดึงความสัมพันธ์ระยะไกล + U-Net เก็บรายละเอียด · บทนี้ทำให้คุณเข้าใจ U-Net เพื่อตัดสินใจว่าจะใช้เดี่ยว ผสม หรือเป็น decoder · เป็นบท 🟡 (ไม่ใช่หัวใจ) — เข้าใจภาพรวมพอ

1. ปัญหาที่ U-Net แก้

งานเราคือ "สนาม → สนาม" (อากาศ → g,D) ขนาดเท่ากัน · นี่เรียกงาน dense prediction (ทำนายทุกพิกเซล) · ความท้าทายคือต้องใช้ทั้ง:

  • บริบทใหญ่ (context): เพื่อรู้ว่า "บริเวณนี้มีลมพัดมาจากไหน" ต้องเห็นกว้าง
  • รายละเอียดเล็ก (detail): เพื่อให้ขอบ/ตำแหน่งคมชัด ต้องเห็นละเอียด

CNN ธรรมดา (0.5) ถ้าทำให้เห็นกว้าง (ลดขนาดภาพ/pool) จะเสียรายละเอียด · U-Net คือสถาปัตยกรรมที่ออกแบบมาให้ได้ทั้งสองอย่างพร้อมกัน

2. โครงสร้างตัว "U" — encoder + decoder ★

ชื่อ U-Net มาจากรูปร่างเหมือนตัว U: ฝั่งซ้ายบีบลง (encoder) แล้วฝั่งขวาขยายขึ้น (decoder)

  • Encoder (ลง): ค่อย ๆ ย่อภาพ (pool) + เพิ่ม channel → ทิ้งรายละเอียดเชิงตำแหน่ง แต่เห็นบริบทกว้างขึ้น (receptive field โต)
  • Bottleneck (คอขวด): เล็กสุด = เห็นกว้างสุด = เข้าใจ "ภาพรวม"
  • Decoder (ขึ้น): ค่อย ๆ ขยายกลับ (upsample) → คืนขนาดเดิม เพื่อออกสนาม g,D ที่ละเอียดเท่าอินพุต

3. Skip connection — กุญแจที่ทำให้ U-Net เก่ง ★★

ปัญหา: พอ encoder ย่อภาพลงไปจนคอขวด รายละเอียดเชิงตำแหน่งหายไปแล้ว · decoder ขยายกลับขึ้นมาเองจะได้ภาพเบลอ (ไม่รู้ขอบคมอยู่ตรงไหน)

Skip connection แก้ตรงนี้: ส่งสำเนาของ feature จากชั้น encoder ตรง ๆ ข้ามไปต่อกับชั้น decoder ที่ขนาดเดียวกัน (เส้นประเหลืองในรูป) · decoder จึงได้ทั้ง "ภาพรวมจากคอขวด" + "รายละเอียดคมจาก encoder"

อุปมา: เหมือนวาดรูปคน — คุณร่างโครงใหญ่ก่อน (เห็นภาพรวมว่าคนยืนท่าไหน = คอขวด) แล้วกลับไปดูภาพถ่ายต้นฉบับ (skip connection) เพื่อเติมรายละเอียดใบหน้าให้ตรงตำแหน่ง · ถ้าไม่มีภาพต้นฉบับอ้างอิง รายละเอียดจะมั่ว
ทำไมสำคัญกับฝุ่น: เราต้องการ g,D ที่คมตรงตำแหน่ง — เช่น อัตราการปล่อย g ต้องสูงเฉพาะ "ตรงจุดที่มีไฟ" ไม่ใช่เบลอทั่วไปหมด · skip connection ทำให้ตำแหน่งคม ในขณะที่คอขวดให้บริบทลม · จับทั้งสองสเกล

4. เดินตามข้อมูลทีละชั้น

  1. อินพุต: สนามอากาศ (ลม/ความชื้น/PBL/FRP) ขนาด เช่น 256×256, หลาย channel
  2. Encoder: conv+pool ลง 256→128→64→32 · ทุกครั้งที่ลง เห็นกว้างขึ้น (RF โต) แต่ละเอียดน้อยลง
  3. Bottleneck 32×32: เข้าใจ "ลมพัดทิศไหนทั่วทั้งภาค"
  4. Decoder: upsample 32→64→128→256 · แต่ละชั้นต่อ skipจาก encoder ขนาดเดียวกัน → เติมรายละเอียด
  5. หัวสุดท้าย: conv → softplus(g≥0), sigmoid(D∈0..1) (จาก 2.4)

6. ผสมกัน — ทางเลือกของ HazeNet ★

ในงานเรา backbone ℱθ (ตัวทำนาย g,D) มี 3 ทางเลือก:

  • AFNO เดี่ยว: ง่าย เร็ว global · เริ่มต้นด้วยอันนี้ (baseline แรกของ backbone)
  • U-Net เดี่ยว: เป็น baseline เปรียบเทียบ (ในการ ablation — ดู E-experiments ของ master plan)
  • ลูกผสม AFNO+U-Net: เช่น AFNO blocks สำหรับ global + โครง U-Net (encoder-decoder+skip) สำหรับรายละเอียด → ได้ทั้งสองด้าน (มีงานอากาศหลายชิ้นทำแนวนี้)
แนวปฏิบัติที่ผมแนะนำ (เข้ากับ 2 เดือน): เริ่มจาก AFNO เดี่ยวให้ pipeline ครบก่อน (global RF สำคัญสุดสำหรับการพัดพาไกล) · แล้วค่อยทดลอง U-Net และลูกผสมเป็น ablation เพื่อวัดว่ารายละเอียดช่วยแค่ไหน · อย่าเริ่มจากลูกผสมซับซ้อน — over-engineering ขัดข้อจำกัดโครงการ · ให้ข้อมูล (การทดลอง) เป็นตัวตัดสินว่าต้องผสมไหม
กฎทองยังคุมอยู่ (จาก 2.5/3.2): ไม่ว่าเลือก backbone ไหน — U-Net, AFNO, หรือลูกผสม — มันทำงานกับอากาศเพื่อออก g,D เท่านั้น ห้ามแตะ C โดยตรง · U-Net ก็ไม่เชิงเส้น (มี activation/pool) เหมือน AFNO → ถ้าเอาไปประมวล C จะพัง superposition · backbone เป็นแค่ "ตัวทำนายสัมประสิทธิ์" เสมอ

7. โค้ดโครงสร้าง U-Net (ย่อ)

import torch, torch.nn as nn

class UNet(nn.Module):
    def __init__(self, in_ch, out_ch):
        super().__init__()
        self.enc1 = conv_block(in_ch, 64)
        self.enc2 = conv_block(64, 128)
        self.bottleneck = conv_block(128, 256)
        self.up2 = up_block(256, 128)
        self.up1 = up_block(128, 64)
        self.head = nn.Conv2d(64, out_ch, 1)   # out_ch = g,D

    def forward(self, x):
        e1 = self.enc1(x)                 # เก็บไว้ทำ skip
        e2 = self.enc2(self.pool(e1))     # เก็บไว้ทำ skip
        b  = self.bottleneck(self.pool(e2))   # คอขวด (กว้างสุด)
        d2 = self.up2(b,  skip=e2)        # ขยาย + ต่อ skip e2 ★
        d1 = self.up1(d2, skip=e1)        # ขยาย + ต่อ skip e1 ★
        return self.head(d1)           # สนาม g,D (ตามด้วย softplus/sigmoid)

สังเกต skip=e2, skip=e1 — นั่นคือ skip connection ที่ส่ง feature จาก encoder ไป decoder · หัวใจทั้งหมดอยู่ตรงนั้น

8. Cheat sheet + แบบฝึกหัด

สรุปจำ:
  • U-Net: สถาปัตยกรรม encoder-decoder รูปตัว U สำหรับงาน "สนาม→สนาม" (dense prediction)
  • Encoder (ลง): ย่อภาพ → เห็นกว้าง แต่เสียรายละเอียด · Bottleneck: เล็กสุด=ภาพรวม
  • Decoder (ขึ้น): ขยายกลับเป็นขนาดเดิม → ออกสนาม g,D
  • ★★ Skip connection: ส่ง feature จาก encoder ข้ามไป decoder → คืนรายละเอียด/ขอบคมตรงตำแหน่ง
  • U-Net vs AFNO: U-Net=รายละเอียด/ขอบคม · AFNO=ระยะไกล/global → เก่งคนละด้าน
  • HazeNet: เริ่ม AFNO เดี่ยว → ทดลอง U-Net/ลูกผสมเป็น ablation (ให้ข้อมูลตัดสิน)
  • กฎทอง: ทุก backbone ทำกับอากาศออก g,D · ห้ามแตะ C (จะพัง superposition)

แบบฝึกหัด 1 — ถ้าตัด skip connection ออกจาก U-Net ผลลัพธ์จะเป็นยังไง และทำไม?

เฉลย

ผลลัพธ์จะเบลอ ขอบไม่คม ตำแหน่งมั่ว · เพราะ encoder ย่อภาพลงจนคอขวดทำให้รายละเอียดเชิงตำแหน่งหายไป · decoder ที่ขยายกลับขึ้นมาเองไม่มีข้อมูลว่าขอบคม/ไฟอยู่พิกัดไหน → เดาเอาเบลอ ๆ · skip connection คือสิ่งที่ส่งรายละเอียดต้นฉบับข้ามมาให้ decoder อ้างอิง → ขอบคมตรงตำแหน่ง

แบบฝึกหัด 2 — งานเราเน้นควันพัดข้ามพรมแดน (ระยะไกล) · ทำไมจึงเริ่มจาก AFNO ไม่ใช่ U-Net?

เฉลย

การพัดพาข้ามพรมแดนเป็นความสัมพันธ์ระยะไกล (แหล่งพม่า→ผู้รับเชียงใหม่) · AFNO มี global receptive fieldตั้งแต่ชั้นแรก (Fourier) → จับความสัมพันธ์ไกลได้โดยตรง · U-Net receptive field โตตามความลึกแต่ยังจำกัดกว่า · เริ่มจาก AFNO จึงตรงกับฟิสิกส์หลักของปัญหาที่สุด · แล้วค่อยเพิ่ม U-Net/ลูกผสมถ้าพบว่ารายละเอียดเล็กสำคัญ (ตัดสินด้วย ablation)

แบบฝึกหัด 3 — เพื่อนเสนอใช้ U-Net รับสนามควัน C โดยตรงเพื่อ "ทำให้ควันคมขึ้น" · ทำไมนี่ผิดหลักการเรา?

เฉลย

U-Net มี pool/activation = ไม่เชิงเส้น · ถ้าเอาไปประมวล C โดยตรง → UNet(C⁽¹⁾+C⁽²⁾) ≠ UNet(C⁽¹⁾)+UNet(C⁽²⁾) → ทำลาย superposition (2.5) → แยกแหล่งไม่ได้ · ตามกฎทอง: backbone (U-Net/AFNO) ทำได้เฉพาะกับอากาศเพื่อออกสัมประสิทธิ์ g,D เท่านั้น · ถ้าอยากให้ควันคม ต้องทำผ่านโครงสร้างเชิงเส้น (เช่น 𝒜 ที่ละเอียดขึ้น) ไม่ใช่เอา nonlinear net ไปแตะ C

จบ 3.3 แล้วคุณควร: เข้าใจ U-Net (encoder บีบลง → bottleneck เห็นกว้าง → decoder ขยายกลับ) และskip connectionที่คืนรายละเอียดคม, รู้ว่า U-Net เก่งรายละเอียด/AFNO เก่งระยะไกล (เก่งคนละด้าน), และแนวทาง HazeNet = เริ่ม AFNO เดี่ยวแล้วทดลองผสมเป็น ablation · กฎทองยังคุม: backbone แตะแค่อากาศ ไม่แตะ C
