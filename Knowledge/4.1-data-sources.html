4.1 — แหล่งข้อมูล + วิธีดึง 🟡 กลาง

FIRMS · ERA5 · Himawari · Air4Thai · DEM — "วัตถุดิบ" ทั้งหมดของ HazeNet และวิธีเอามาจริง ๆ

เริ่มโมดูล 4 (ข้อมูล+วิศวกรรม) · ต่อจาก 1.1 (PM2.5/AOD/FRP คืออะไร) และ 1.3 (อุตุนิยม/ERA5) ที่ "เลื่อนวิธีดึงมาบทนี้" · ส่งต่อ → 4.2 (รูปแบบไฟล์) → 4.3 (regrid) → 4.4 (สร้าง datacube)

บทนี้ตอบคำถามเดียว: "ข้อมูลที่ใช้เทรน HazeNet มาจากไหน และเราดึงมายังไงด้วยมือเราเอง?"

1. ภาพรวม — 5 แหล่ง รวมเป็นก้อนเดียว แล้วป้อนโมเดล

ก่อนลงรายละเอียดทีละแหล่ง ต้องเห็นภาพใหญ่ก่อนว่าแต่ละอัน "ไหลไปไหน" ในระบบ · ทุกแหล่งจะถูกแปลงให้อยู่บน กริดเดียวกัน (0.1° ทั่วโดเมนไทย/SEA) แล้วซ้อนเป็น datacube ก้อนเดียว (เรื่องวิธีรวมอยู่ใน 4.3–4.4):

จุดที่คนพลาดบ่อย: Air4Thai (target) ไม่ใช่ input — มันคือ "เฉลย" ที่เอาไว้เทียบกับสิ่งที่โมเดลทำนายเพื่อคำนวณ loss · ถ้าเผลอเอา PM2.5 จริงไปใส่เป็น input โมเดลจะ "โกง" (data leakage) แล้วผลที่ได้จะดีปลอม ๆ (ดู 0.4)

1.1 ทำไมต้องหลายแหล่ง — ไม่มีแหล่งเดียวที่ครบ

คำถามที่ดี: "ทำไมไม่ใช้ PM2.5 จากสถานีอย่างเดียว?" คำตอบคือ ไม่มีแหล่งไหนสมบูรณ์ในตัวเอง — แต่ละอันเก่งคนละด้าน เราต้องเอามา "เติมจุดอ่อนกันและกัน":

แหล่งเก่งเรื่องจุดอ่อนใครมาเติม
Air4Thaiวัด PM2.5 จริงที่พื้น แม่นที่สุดเป็น "จุด" ห่าง ๆ ครอบคลุมพื้นที่ไม่ทั่วHimawari/แบบจำลองเติมช่องว่างเชิงพื้นที่
Himawari AODเห็นควันทั่วทั้งภาพ ทุก 10 นาทีเป็นค่าทั้งคอลัมน์อากาศ ไม่ใช่ที่พื้น + กลางคืน/เมฆใช้ไม่ได้Air4Thai สอนให้แปลงเป็นค่าพื้น
ERA5อากาศครบทุกจุดทุก ชม. (ตัวขับ)ไม่ได้บอกฝุ่นโดยตรงเป็นเชื้อเพลิงให้ฟิสิกส์เคลื่อนฝุ่น
FIRMSบอกว่าไฟอยู่ตรงไหน แรงแค่ไหนเป็นจุด + เมฆบัง + เห็นแค่ตอนดาวเทียมผ่านFINN/กระจายเชิงเวลามาเติม
DEMรูปร่างภูมิประเทศละเอียดมากนิ่ง ไม่เปลี่ยนตามเวลาให้ "ฉากหลัง" ที่อากาศไหลผ่าน

2. 4 เรื่องที่ต้องเข้าใจก่อนดึงข้อมูลทุกแหล่ง

ก่อนแตะ API ใด ๆ ต้องเข้าใจ 4 คำนี้ก่อน เพราะมันโผล่ในทุกแหล่ง:

2.1 ความละเอียด 2 แกน: เชิงพื้นที่ vs เชิงเวลา

ข้อมูลภูมิสารสนเทศมี "ความละเอียด" 2 ทิศ และมัก ได้อย่างเสียอย่าง:

  • เชิงพื้นที่ (spatial): 1 พิกเซลแทนพื้นที่กว้างเท่าไร — ERA5 ~31 กม. (หยาบ), VIIRS 375 ม. (ละเอียด)
  • เชิงเวลา (temporal): ได้ข้อมูลใหม่บ่อยแค่ไหน — Himawari ทุก 10 นาที (ถี่), MODIS วันละ ~2 ครั้ง (ห่าง)
นี่คือเหตุผลเชิงลึกว่าทำไม HazeNet ต้องเป็น GeoAI: เราต้องหลอมข้อมูลที่ความละเอียดต่างกันมากให้อยู่บนกริดเดียว แล้วให้โมเดล "เติมช่องว่าง" ทั้งเชิงพื้นที่ (ระหว่างสถานี) และเชิงเวลา (ระหว่างที่ดาวเทียมผ่าน) — งานนี้ทำด้วยสถิติธรรมดาไม่ได้ ต้องใช้ฟิสิกส์+AI

2.2 ข้อมูลจุด (point) vs ข้อมูลกริด (gridded)

แหล่งข้อมูลมา 2 รูปทรง — และต้องแปลงให้เป็นกริดเหมือนกันหมดก่อนเทรน:

  • กริด (raster): ค่ากระจายเต็มทุกช่องตาราง → ERA5, Himawari, DEM · ใช้ได้เกือบทันที (แค่ regrid ความละเอียด)
  • จุด (point/vector): ค่าอยู่เฉพาะพิกัด → FIRMS (จุดไฟ), Air4Thai (จุดสถานี) · ต้อง "วางลงกริด" (rasterize) ก่อน

2.3 ลิขสิทธิ์ + บัญชี (license & auth)

ทุกแหล่งในบทนี้ ฟรีและเปิด (เงื่อนไขโครงการเรา) แต่เกือบทุกอันต้อง "สมัครบัญชีฟรี + ขอ key" ก่อน:

ข้อควรระวังด้านความซื่อสัตย์ทางวิชาการ: เก็บ "ที่มา + เวอร์ชัน + วันที่ดึง" ของทุกชุดข้อมูลไว้ในไฟล์ data_sources.md ตั้งแต่วันแรก · ตอนเขียนรายงาน/ตอบกรรมการ ต้องอ้างอิงได้ว่า PM2.5 มาจาก Air4Thai, อากาศจาก ERA5 ฯลฯ — ไม่งั้นงานจะ reproduce ไม่ได้

2.4 Latency — ข้อมูลมาช้าแค่ไหน (NRT vs archive)

มี 2 โหมด: NRT (near-real-time) มาเร็ว (ชั่วโมง) แต่ยังไม่ผ่านการตรวจสอบเต็ม · archive/reanalysis มาช้า (วัน–สัปดาห์) แต่ผ่านการตรวจคุณภาพแล้ว · โปรเจกต์เราเทรนกับอดีต → ใช้ archive เป็นหลัก (คุณภาพสำคัญกว่าความเร็ว) · NRT จะสำคัญตอนทำ "เดโมพยากรณ์สด" ทีหลังเท่านั้น

วิธีดึง — FIRMS Area API (ฟรี, ง่ายสุด)

ขอ MAP_KEY ฟรีที่ firms.modaps.eosdis.nasa.gov/api/area แล้วเรียกเป็น URL ได้เลย คืนค่าเป็น CSV:

import pandas as pd

MAP_KEY = "<ใส่ MAP_KEY ของคุณ>"
SOURCE  = "VIIRS_SNPP_SP"      # SP = archive · NRT = near-real-time
AREA    = "92,5,110,25"        # west,south,east,north = โดเมนไทย/SEA ตอนบน
DATE    = "2024-03-15"          # วันเริ่ม
DAYS    = 10                    # สูงสุด 10 วัน/คำขอ

url = (f"https://firms.modaps.eosdis.nasa.gov/api/area/csv/"
       f"{MAP_KEY}/{SOURCE}/{AREA}/{DAYS}/{DATE}")
fires = pd.read_csv(url)
print(fires[["latitude","longitude","acq_date","acq_time","frp"]].head())

→ ไล่ลูปทีละ 10 วันครอบฤดูเผา แล้วต่อกัน · สำหรับย้อนหลังหลายปีใช้ FIRMS Archive Download Tool (ขอเป็นไฟล์ก้อนใหญ่ทางอีเมลได้)

บทบาทใน HazeNet: FIRMS คือ ต้นทางของพจน์ E (emission) ในสมการ ADR (1.2) และในโมดูล Emission ของ FastCTM · แต่มันเป็น จุด → ต้อง rasterize ลงกริด 0.1° (2.2) ก่อน · และเพราะ "tagged-tracer" (3.6) ติดป้ายแหล่งจากตำแหน่งไฟ → FIRMS คือข้อมูลที่ทำให้ source attribution เป็นจริงได้
caveat ต้องบอกกรรมการ: (1) เมฆบังไฟ — วันที่เมฆเยอะจะ "เห็นไฟน้อยกว่าจริง" · (2) เห็นแค่ ตอนดาวเทียมผ่าน (~2–4 ภาพ/วัน) ไฟกลางคืน/ช่วงไม่มีดาวเทียมหาย · (3) FRP เป็น "ความแรง ณ ขณะถ่าย" ไม่ใช่มวลควันโดยตรง — การแปลง FRP→มวลต่อสารต้องใช้ FINN (ดูข้อ 8)

1.3 อธิบาย "ทำไม" ครบแล้ว (อากาศ = ตัวขับทุกพจน์ ADR) · ตรงนี้คือ "ทำยังไง" · ERA5 เป็น reanalysis = ค่าประมาณอากาศอดีตที่ดีที่สุด เต็มทุกจุดทุกชั่วโมง · มี 2 กลุ่มผลิตภัณฑ์:

  • single levels (ผิวพื้น): 10m_u/v_wind (ลม→advection), boundary_layer_height (PBL→กักควัน), total_precipitation (ฝน→ชะล้าง), 2m_temperature, 2m_dewpoint (→ความชื้น/เคมี), surface_pressure
  • pressure levels (แนวตั้ง): ลม/อุณหภูมิ/ความชื้นที่ระดับ 1000/925/850 hPa — ใช้ดูลมชั้นบนที่หอบควันข้ามแดน

วิธีดึง — cdsapi (Python)

1) สมัครบัญชี cds.climate.copernicus.eu · 2) เอา token ใส่ไฟล์ ~/.cdsapirc · 3) กดยอมรับ licence ของชุดข้อมูลบนเว็บก่อน (ไม่งั้น error) · 4) รัน:

import cdsapi
c = cdsapi.Client()

c.retrieve("reanalysis-era5-single-levels", {
    "product_type": "reanalysis",
    "variable": ["10m_u_component_of_wind", "10m_v_component_of_wind",
                 "boundary_layer_height", "total_precipitation",
                 "2m_temperature", "2m_dewpoint_temperature"],
    "year": "2024", "month": "03",
    "day":  [f"{d:02d}" for d in range(1,32)],
    "time": [f"{h:02d}:00" for h in range(24)],
    "area": [25, 92, 5, 110],   # N, W, S, E (ระวังลำดับ! ต่างจาก FIRMS)
    "format": "netcdf",
}, "era5_2024_03.nc")
เรื่องล่าสุดที่ทำคนติดบ่อย (2024+): CDS ย้ายไประบบใหม่และ ปิด API เก่า แล้ว · ต้องใช้ cdsapi เวอร์ชันใหม่, endpoint https://cds.climate.copernicus.eu/api, และ key รูปแบบใหม่ (เป็น token เดี่ยว ไม่มี UID: นำหน้าแบบเก่า) · ชื่อชุดข้อมูลบางตัวก็เปลี่ยน — ถ้า error ให้ก๊อปโค้ดจากปุ่ม "Show API request" บนหน้าเว็บชุดข้อมูลนั้นโดยตรง
บทบาทใน HazeNet: ERA5 คือ อินพุตหลักที่ป้อนเข้าตัวฟิสิกส์ — ลม u,v เข้าตัว advection (2.2), PBL/อุณหภูมิ→สัมประสิทธิ์การกระจาย K, ฝน/ความชื้น→การสูญเสีย · ในภาษา 3.5 (conditionally-linear) อากาศพวกนี้คือ "condition" ที่ใช้คำนวณสัมประสิทธิ์ของตัวดำเนินการเชิงเส้น · ต้อง regrid 0.25°→0.1° (4.3)

10. ชุดข้อมูลขั้นต่ำสำหรับโปรเจกต์ 2 เดือน ★ ทำจริง

นี่คือข้อเสนอ "ทำได้จริงในเวลา + งบ A100 จำกัด" — เลือกให้พอครบเพื่อพิสูจน์แนวคิด ไม่บานปลาย:

โดเมน + เวลา

มิติเลือกเหตุผล
พื้นที่lat 5–25°N, lon 92–110°Eคลุมไทยตอนบน + เมียนมา(ฉาน) + ลาวเหนือ = แหล่งควันข้ามแดนหลัก
กริด0.1° (~11 กม.) → ~180×200 ช่องละเอียดพอเห็นแอ่ง แต่เบาพอเทรนบน A100 เดียว
ฤดูกาลเน้นฤดูเผา ธ.ค.–เม.ย.ช่วงที่ปัญหาเกิดจริง + signal แรง (เทรนคุ้มกว่าเอาทั้งปี)
ปี~5 ฤดู เช่น 2019–2023 (เทรน) + 2024 (ทดสอบ)แบ่งตามเวลา กัน data leakage (5.2)
ความถี่เวลาเริ่มที่ราย 3 ชม. (ขยับเป็นรายชม.ได้)ลดขนาดข้อมูล 3 เท่า → เทรนเร็ว/ประหยัดงบตอน prototype

ช่องข้อมูล (channels) ใน datacube

  • Input ไดนามิก: u, v, PBL, precip, T, RH (ERA5) · AOD (Himawari) · E การปล่อย (FIRMS→กริด หรือ FINN)
  • Input static: ความสูง + ความลาดชัน (DEM) · (ทางเลือก) land cover
  • Target: PM2.5 ที่ตำแหน่งสถานี (Air4Thai/OpenAQ)

ประเมินขนาด: ~12 ช่อง × 180×200 × (5 ฤดู×~5 เดือน×ราย3ชม.) ≈ หลัก 10 GB เมื่อบีบอัดเป็น Zarr/float16 → พอดีกับ Colab (จัดการ memory ใน 4.5)

กฎกันงานบานปลาย: เฟสแรกใช้ 5 แกน เท่านั้น · อย่าเพิ่งแตะ TROPOMI/MAIAC/WorldCover จนกว่า pipeline พื้นฐาน (4.4) จะวิ่งครบรอบและเทรนได้ผลแรก · "ให้มันวิ่งก่อน แล้วค่อยทำให้ดี"

11. เชื่อม HazeNet + สิ่งที่จะทำต่อ + caveat

หลังดึง 5 แหล่งดิบมาแล้ว ขั้นต่อไปคือทำให้มัน "คุยกันรู้เรื่อง" บนกริดเดียว:

สรุปการแมป "แหล่ง → พจน์ในฟิสิกส์": FIRMS→ E (การปล่อย) · ERA5 ลม→ advection, PBL/T→ K กระจาย, ฝน/RH→ การสูญเสีย/เคมี · Himawari AOD→ ตัวสังเกตการณ์ตรวจทาน · DEM→ ฉากหลังเชิงพื้นที่ · Air4Thai→ เฉลยที่ใช้คำนวณ loss · นี่คือเหตุผลว่าทำไม "ข้อมูล" กับ "ฟิสิกส์ใน 1.2" เป็นเรื่องเดียวกัน
3 caveat ที่ต้องเขียนในรายงานตั้งแต่ต้น (ความซื่อสัตย์):
1) ช่องว่างเชิงพื้นที่ฝั่ง target: สถานีกระจุกในเมืองไทย ฝั่งเมียนมา/ลาวแทบไม่มี → validate ต้นทางข้ามแดนตรง ๆ ไม่ได้ ต้องพึ่ง AOD
2) ช่องว่างจากเมฆ/กลางคืน: ทั้ง FIRMS และ Himawari หายตอนเมฆ/มืด → ต้องมีวิธีจัดการ missing data (4.4) และระบุผลกระทบ
3) error สะสมจากการแปลงจุด→กริดและ regrid: ทุกการแปลงความละเอียดมี error → ควรทำ ablation ว่าผลไวต่อ choice เหล่านี้แค่ไหน (5.4)

12. Cheat sheet + แบบฝึกหัด

Cheat sheet — จำ 1 บรรทัดต่อแหล่ง

  • 🔥 FIRMS = ไฟอยู่ไหน แรงแค่ไหน (FRP) → ต้นทาง E · CSV จุด · MAP_KEY
  • 🌬️ ERA5 = อากาศครบทุกชม. → ตัวขับฟิสิกส์ · NetCDF กริด · cdsapi
  • 🛰️ Himawari = ควันจากบน ทุก 10 นาที → สังเกตการณ์ · กริด · P-Tree/AWS
  • 📍 Air4Thai = PM2.5 จริงที่พื้น → เฉลย (ไม่ใช่ input!) · JSON จุด · +OpenAQ
  • ⛰️ DEM = รูปร่างภูเขา/แอ่ง → ฉากหลัง static · GeoTIFF · AWS เปิด

❓ แบบฝึกหัด (ตอบในใจ — เช็กความเข้าใจ)

  1. ทำไม Air4Thai ห้ามใส่เป็น input ของโมเดล?
  2. Himawari ได้เปรียบ MODIS เรื่องอะไร และเสียเปรียบเรื่องอะไร?
  3. FRP จาก FIRMS เป็นข้อมูล "จุด" — ต้องทำอะไรก่อนป้อนโมเดล?
  4. ทำไม DEM (ข้อมูลนิ่ง) ถึงสำคัญกับฝุ่นเชียงใหม่เป็นพิเศษ?
  5. ถ้าจะเพิ่มแหล่งเสริมได้ 1 อย่าง ควรเพิ่มอะไร เพราะอะไร?
เฉลย
  1. เพราะมันคือ "เฉลย" (target) — ถ้าใส่เป็น input = โกง (data leakage) ผลดีปลอม (0.4)
  2. ได้เปรียบ: ความถี่ทุก 10 นาที (จับพลวัตรายชม.) · เสียเปรียบ: กลางคืน/เมฆใช้ไม่ได้ + ความละเอียดเชิงพื้นที่หยาบกว่า (5กม. vs 375ม.-1กม.)
  3. ต้อง rasterize = นับ/รวม FRP ลงช่องกริด 0.1° ให้กลายเป็นกริดการปล่อย E (2.2)
  4. ภาคเหนือเป็นแอ่งกระทะ → เกิด inversion กักควัน · DEM บอกโมเดลว่าจุดไหนเป็นแอ่ง (ฟิสิกส์ที่มองไม่เห็นจากค่าฝุ่นอย่างเดียว)
  5. FINN v2.5 — แปลง FRP→มวลปล่อยจริงแยกตามสาร ทำให้พจน์ E ถูกต้องเชิงฟิสิกส์ (สำคัญต่อทั้ง forecast + attribution)
