เป้าหมายการส่ง: เดโม + repo โค้ด ที่รันได้จริงและโชว์ "กลุ่มควันมาจากไหน" ได้

ทำหลังผ่าน feasibility 3 ประตู · เน้น "ของที่ใช้งานได้จริง + ทำซ้ำได้" มากกว่าความเข้มงวดแบบเปเปอร์ · A100 แตะเฉพาะ main run

นิยาม "เสร็จ" สำหรับเดโม + repo:
คนอื่น clone repo → ติดตั้งตาม README → รันเดโม → เห็นพยากรณ์ PM2.5 + แผนที่กลุ่มควันว่ามาจากแหล่งไหน + ความไม่แน่นอน — โดยไม่ต้องถามคุณ

ไม่ใช่ "โมเดลแม่นที่สุดในโลก" แต่คือ "ระบบที่ทำงานครบวงจร โชว์จุดขายได้ และทำซ้ำได้" · ความน่าเชื่อถือมาจากการมี baseline เทียบ + caveat ซื่อสัตย์ ไม่ใช่จากตัวเลขสวยอย่างเดียว

0. "เสร็จ" หน้าตาเป็นยังไง — สำหรับเดโม + repo

เป้าการส่งกำหนดว่า main run ต้องผลิตอะไร · เดโม+repo ต้องการ 5 ชิ้นนี้:

ชิ้นคืออะไรทำไมต้องมี
checkpoint 1 ตัวโมเดลที่เทรนเสร็จ (.pt) จาก run ที่ดีที่สุดหัวใจของเดโม — โหลดมาใช้ inference
เดโมรันได้app/notebook: ใส่วันที่ → ออกพยากรณ์ + แผนที่กลุ่มควัน + uncertainty= ตัวส่งหลัก โชว์จุดขาย (3.6)
หลักฐานว่าไม่ได้มั่วตาราง RMSE vs persistence + กราฟ skill-vs-leadtimeความน่าเชื่อถือ (5.1/5.3)
repo สะอาดโครงโฟลเดอร์ + README + requirements + configคนอื่นรันต่อได้
caveat ซื่อสัตย์ส่วนข้อจำกัดใน READMEงานวิจัยที่ดีต้องบอกขอบเขต
เทียบเป้าอื่น: ถ้าเป็นเปเปอร์ต้องมี CV ครบ + significance + baseline หลายตัว · แต่ เดโม+repo เน้น "ใช้งานได้ + ทำซ้ำได้ + มี baseline ขั้นต่ำ 1 ตัวกันมั่ว" → เบากว่ามาก เหมาะกับงบ 15 ชม.

1. Pre-flight 7 ข้อ — ต้องล็อกครบก่อนกดรันใหญ่

1
ผ่าน feasibility 3 ประตูแล้ว
ข้อมูล (ชั้น 0–1) → ท่อ/Lab 0 (ชั้น 2–3) → วิทย์/Lab 1 (ชั้น 4) เขียวครบ · นี่คือใบเบิกทาง ห้ามข้าม
เช็ก: ทั้ง 3 ประตูในไฟล์ feasibility-checklist ติ๊กครบ
2
แช่แข็ง config การทดลอง 4.4
เขียน config.yaml แล้วห้ามขยับระหว่างรัน: 
domain:   {lat: [5,25], lon: [92,110], step: 0.1}
time:     {seasons: ["2021-2025 burning"], freq: "3H"}
channels: [u10, v10, dem, emission, aod, pbl, precip, rh, ...]
split:    {train: "...→2024", val: "2024", test: "2025"}   # บล็อกเวลา
model:    {type: cond_linear, ...}
train:    {epochs: ?, lr: ?, batch: ?, rollout: ?}
seed:     42
เช็ก: มี config.yaml ที่ครอบทุก hyperparam · เปลี่ยน config = เปลี่ยนผล = ต้องรันใหม่ → จึงล็อกก่อน
3
ประมาณงบ A100 จากของจริง สำคัญสุด 4.5
กับดักใหญ่สุดของโปรเจกต์: เผางบ 15 ชม.หมดโดยยังไม่ได้ checkpoint ที่ใช้ได้ · ต้องประมาณ ก่อน ไม่ใช่ลุ้น
  1. วัด "วินาที/epoch" ตอนสเกลกลาง (1 ฤดู, โดเมนจริง, T4 หรือ A100 สั้น ๆ)
  2. คูณ: เวลา/epoch × epochs × (main + baseline หนัก + ablation)
  3. ยืนยัน รวม ≤ ~12 ชม. (เผื่อพัง/รันซ้ำ 3 ชม.)
  4. ถ้าเกิน → ตัด scope ตอนนี้: ลดฤดู / ลด resolution / ลด rollout / ตัด baseline หนัก
เช็ก: มีตารางงบที่บวกแล้ว ≤ 12 ชม. มีเผื่อ · รู้ว่าจะตัดอะไรถ้าไม่พอ
4
ข้อมูลโหลดครบ + datacube cache เป็น Zarr 4.1/4.4
ห้ามดึงข้อมูล/ประมวลผล regrid ระหว่างเทรน — ทำให้เสร็จก่อน เก็บเป็น Zarr พร้อมใช้
A100 ที่เสียไปกับการรอ download/regrid = งบที่หายไปเปล่า · ทุก byte ที่โมเดลอ่านควรมาจาก Zarr ที่ทำเสร็จแล้ว
เช็ก: มี cube.zarr ของโดเมน/ช่วงเต็ม + norm_stats.json (train-only) · เปิดเทสได้
5
Checkpoint/resume + logging ใช้ได้จริง 4.5
Colab หลุดบ่อย → ต้อง resume ได้ + log ทุก epoch (ไม่งั้นรันเสร็จแต่ไม่มีกราฟไปโชว์) 
# เซฟทุก epoch ไป Drive + log metric
torch.save({"model":model.state_dict(),"opt":opt.state_dict(),
  "epoch":ep,"val_rmse":val_rmse}, "/drive/ckpt_last.pt")
log.append({"epoch":ep,"train_loss":..,"val_rmse":..})  # → csv
เช็ก: kill แล้ว resume ต่อได้ตรงจุด · มี log csv ออกมาทุก epoch
6
ตัดสินใจ "ผลที่ต้องผลิต" ก่อนรัน 5.1/3.6/3.9
รู้ก่อนว่าเดโมต้องโชว์อะไร จะได้ไม่ต้องรันซ้ำเพื่อเก็บผลเพิ่ม:
  • พยากรณ์ PM2.5 field (แผนที่ + เลื่อน lead time)
  • แผนที่/กราฟ attribution — กลุ่มควันจากแต่ละแหล่ง (จุดขาย, 3.6)
  • uncertainty (3.9) — แถบความไม่แน่นอน
  • กราฟ skill-vs-leadtime + ตาราง RMSE vs persistence (5.1/5.3)
เช็ก: มีลิสต์รูป/ตารางที่ต้องได้ + โค้ด generate มันจาก checkpoint
7
Reproducibility package พร้อม
seed คงที่ · config.yaml + manifest (เวอร์ชันข้อมูล/วันที่ดึง) · requirements.txt freeze
เช็ก: รันด้วย config+seed เดิม แล้วได้ผลใกล้เดิม

2. แผนงบ A100 ของ main run (ตัวอย่างกรอบ ~12 ชม.)

ตัวเลขจริงต้องมาจากการวัด (ข้อ 3) — นี่คือ "กรอบให้เห็นภาพ" ปรับตามที่วัดได้

กิจกรรมเวลาโดยประมาณหมายเหตุ
เทรน main model (cond-linear)~5–7 ชม.หัวใจ — ได้ checkpoint หลัก
baseline ConvLSTM (ถ้าทำ)~2–3 ชม.ตัดได้ถ้างบตึง (persistence/climatology ฟรี)
ablation 1 ตัว (ตัด 3.5 → CNN)~2 ชม.พิสูจน์ contribution หลัก (5.4)
inference + generate รูป/attribution<0.5 ชม.ถูกมาก (ไม่ใช่เทรน)
เผื่อพัง/รันซ้ำ~2–3 ชม.ห้ามลืมกันไว้
ลำดับความสำคัญถ้างบไม่พอ: main model > ablation(3.5) > ConvLSTM · baseline persistence/climatology ฟรี ทำได้เสมอ · เดโม+repo ไม่จำเป็นต้องมี baseline หนักครบ

3. ลำดับการรันจริง

  1. ไต่บันไดจน config นิ่ง — ขยายทีละแกนบนของฟรี/T4 จับเวลา/VRAM ทุกขั้น (อย่ากระโดดเต็ม)
  2. main run บน A100 — เทรน cond-linear เต็ม resume ได้ เซฟ checkpoint ที่ดีสุด (จาก val)
  3. ประเมิน — test holdout: RMSE/skill-vs-leadtime + persistence/climatology (ฟรี) + ablation 3.5
  4. generate artifacts — แผนที่พยากรณ์ + attribution + uncertainty + กราฟ (inference ราคาถูก)
  5. ประกอบ demo + repo — โหลด checkpoint ในเดโม + เขียน README + caveat → ส่ง

5. ตัวเดโม — ต้องโชว์อะไร (= จุดขาย)

เดโมคือสิ่งที่กรรมการ/ผู้ใช้เห็นก่อน · ต้องเล่า "HazeNet ทำอะไรได้" ใน 1 หน้าจอ:

โชว์คืออะไรโยง
เลือกวันที่/เวลาinput ให้ผู้ใช้เลือกสถานการณ์—
พยากรณ์ PM2.5 fieldแผนที่ + slider เลื่อน lead time3.4/5.1
กลุ่มควันจากแต่ละแหล่งแผนที่/กราฟ stacked: ที่จุดนี้ ควันมาจากพม่า X% / local Y%3.6 (จุดขาย)
ความไม่แน่นอนแถบ/แผนที่ uncertainty3.9
เทียบ baselineตัวเลข RMSE โมเดล vs persistence5.3
# เดโมแบบเบาสุด: Streamlit โหลด checkpoint แล้ว inference
import streamlit as st, torch
model = load_model("checkpoints/best.pt")
date = st.date_input("เลือกวันที่")
lead = st.slider("lead time (ชม.)", 6, 72, 24)
pred, attribution, unc = forecast(model, date, lead)
st.pyplot(plot_field(pred))            # แผนที่พยากรณ์
st.pyplot(plot_attribution(attribution)) # กลุ่มควันจากแต่ละแหล่ง
ทางเลือกเดโม: Streamlit/Gradio = interactive สวย · notebook = ง่ายสุด commit ได้เลย · เลือกตามเวลาที่เหลือ — notebook ที่รันผ่านครบก็ "ส่งได้" แล้ว

6. ทำซ้ำได้ (reproducibility) — เงื่อนไขของ repo ที่ดี

  • seed คงที่ ทุกที่ (torch/numpy/random) — ผลรันซ้ำใกล้เดิม
  • config.yaml เดียว คุมทุกอย่าง — เปลี่ยนทดลองที่ไฟล์เดียว
  • manifest — บันทึกว่าใช้ข้อมูลเวอร์ชันไหน ดึงวันไหน โดเมน/ช่วงอะไร
  • requirements freeze — pip freeze > requirements.txt
  • README สั่งรันได้จริง — python data/download.py → python src/train.py → streamlit run demo/app.py
เทสขั้นสุดท้าย: ลอง clone repo ลงเครื่อง/Colab ใหม่ที่ไม่มี context อะไรเลย แล้วรันตาม README → ถ้าเดโมขึ้นได้ = ส่งได้จริง

7. caveat ที่ต้องเขียนใน README (ความซื่อสัตย์ = คุณภาพงานวิจัย)

เขียนส่วน "Limitations" ให้ชัด — กรรมการที่เก่งจะเชื่อถืองานที่รู้ขอบเขตตัวเองมากกว่างานที่อวด:
  • Compute: เทรนบน A100 เดี่ยว ~ไม่กี่ชม. → สเกล/ความละเอียดจำกัด
  • สถานี sparse: target จากสถานีภาคพื้นจำกัด · แหล่งต้นทาง (พม่า/ลาว) แทบไม่มีสถานี → attribution validate ตรง ๆ ไม่ได้เต็มที่ (5.2)
  • ข้อมูลหาย: AOD ดาวเทียมหายช่วงเมฆ/กลางคืน → ใช้ mask
  • ขอบเขตเวลา: เทรน/เทสแค่ฤดูเผาบางปี → ไม่ extrapolate นอกช่วง
  • attribution เป็นค่าประมาณ จากโครงสร้าง superposition (2.5) ไม่ใช่ ground truth ของแหล่ง

8. Definition of Done — "ส่งได้" เมื่อติ๊กครบ

🚪 ประตูก่อนรันใหญ่: Pre-flight ข้อ 1–7 ครบ → ค่อยกด main run บน A100

ก่อนรันใหญ่ (Pre-flight)

  • feasibility 3 ประตูเขียวครบ
  • config.yaml แช่แข็งครบทุก hyperparam
  • งบ A100 ประมาณแล้ว ≤ ~12 ชม. มีเผื่อ + รู้จะตัดอะไรถ้าไม่พอ
  • cube.zarr + norm_stats.json พร้อม (ไม่ดึงข้อมูลระหว่างเทรน)
  • checkpoint/resume + log csv ใช้ได้จริง
  • ลิสต์รูป/ตารางที่ต้องผลิต + โค้ด generate พร้อม
  • seed + manifest + requirements freeze

หลังรัน (Artifacts)

  • checkpoint best.pt ที่ใช้ได้
  • ตาราง RMSE โมเดล vs persistence + กราฟ skill-vs-leadtime
  • ablation 3.5 (delta) อย่างน้อย 1 ตัว
  • แผนที่ attribution/กลุ่มควัน + uncertainty

ส่ง (demo + repo)

  • เดโม (app/notebook) รันได้ โชว์พยากรณ์ + กลุ่มควัน + uncertainty
  • repo โครงสร้างสะอาด + .gitignore (ไม่มีไฟล์ใหญ่)
  • README รันตามได้จริง (เทส clone ใหม่)
  • ส่วน Limitations ซื่อสัตย์ครบ
