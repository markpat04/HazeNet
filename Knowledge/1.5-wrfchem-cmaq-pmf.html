1.5 — WRF-Chem / CMAQ / PMF เจาะลึก

"มาตรฐานทอง" ที่เราจะแข่งด้วย และยืมมาเป็นเฉลย — ปิด Module 1

ต่อจาก 1.4 (เอ่ยถึงเครื่องมือพวกนี้) · บทนี้ทำให้รู้จักจริง ๆ ว่ามันคืออะไร ทำงานยังไง ทำไมช้า · ผูกกับ HazeNet ตลอด

ทำไมต้องรู้จักศัตรู: HazeNet ไม่ได้สร้างมาแทน "ไม่มีอะไรเลย" — มันสร้างมาแทน WRF-Chem / CMAQ (โมเดลฟิสิกส์เคมีบรรยากาศ) และเสริม PMF (สถิติแยกแหล่ง) · ถ้าไม่เข้าใจว่าเครื่องมือเดิมเก่งตรงไหน ช้าตรงไหน เราจะ (1) เคลมจุดเด่นผิด (2) ตั้ง benchmark ผิด (3) ใช้ ground truth ผิด · บทนี้คือ "รู้เขารู้เรา" ก่อนลงสนาม

1. CTM คืออะไร — ภาพใหญ่ก่อน

CTM (Chemical Transport Model) = แบบจำลองที่จำลอง "ชะตากรรมของสารในอากาศ" — มันเกิดตรงไหน เคลื่อนไปไหน ทำปฏิกิริยาอะไร หายไปยังไง · พูดอีกอย่าง: CTM คือโปรแกรมที่แก้สมการ ADR (จาก 1.2) ทั้งกริด ทั้งเวลา

WRF-Chem และ CMAQ คือ CTM ที่ดังที่สุดสองตัว · ต่างกันที่ "เชื่อมกับโมเดลอากาศยังไง" (ข้อ 2–3)

เชื่อม HazeNet: เราเรียก HazeNet ว่า "neural CTM" — มันทำงานเดียวกับ CTM (แก้ ADR แล้วได้ PM2.5) แต่แทนที่จะแก้สมการทีละ step ด้วยตัวเลข เราให้นิวรอลเรียนรู้ตัวดำเนินการ · นี่คือเหตุผลที่เราต้องเข้าใจ CTM จริงก่อน — เพราะเราเลียนแบบมัน

2. WRF-Chem — CTM แบบ "ออนไลน์" ★ ground truth หลักของเรา

WRF = โมเดลพยากรณ์อากาศ (ลม อุณหภูมิ ความชื้น) · WRF-Chem = WRF + โมดูลเคมี (Chem) ที่จำลองฝุ่น/ก๊าซไปพร้อมกันในการรันเดียว

"ออนไลน์ (online-coupled)" = อากาศกับเคมีคุยกันทุก time-step · เช่น ควันบังแดด → อุณหภูมิเปลี่ยน → ลมเปลี่ยน → ควันเคลื่อนต่าง · ป้อนกลับกันแบบเรียลไทม์

อุปมา: WRF-Chem เหมือนทำกับข้าวที่ "ปรุงรสไปชิมไป" — สภาพอากาศกับฝุ่นปรับตัวเข้าหากันตลอด · สมจริงสุด แต่ก็หนักสุด
บทบาทในงานเรา: งานวิจัยไทย (PMC11280843, จาก 1.4) ใช้ WRF-Chem + tagged tracer ให้ตัวเลข "เชียงราย ~45% ข้ามแดน" · นี่คือ ground truth หลักที่เราจะเทียบ attribution (C2) · WRF-Chem จึงเป็นทั้ง "คู่แข่งที่เราจะเร็วกว่า" และ "ครูที่เราจะเลียนแบบ+ตรวจตัวเอง"

3. CMAQ — CTM แบบ "ออฟไลน์"

CMAQ (ของ US EPA) ก็เป็น CTM แต่ทำงานแบบ "ออฟไลน์ (offline-coupled)": รันโมเดลอากาศ (WRF) ให้เสร็จก่อน → เซฟผลอากาศ → แล้วค่อยป้อนให้ CMAQ คำนวณเคมีทีหลัง · อากาศไม่รับ feedback จากฝุ่น

4. ทำไม CTM ถึงช้า — pain ที่ HazeNet จะแก้ ★

CTM แก้สมการ ADR ด้วยวิธีเชิงตัวเลข (numerical solver) ทุกช่องกริด ทุก time-step · เหตุผลที่ช้า:

  • กริดละเอียด × หลายชั้นความสูง × ราย step เล็ก ๆ → จำนวนการคำนวณมหาศาล
  • เคมีมีสมการหลายร้อยปฏิกิริยา (stiff equations) → ต้อง step เวลาเล็กมากเพื่อเสถียร
  • รันบน CPU cluster ไม่ได้ออกแบบมาเพื่อ GPU

ผลคือ: พยากรณ์ 1 เหตุการณ์อาจใช้หลายชั่วโมงถึงหลายวันบนเครื่องใหญ่ · ใช้ตอบ "วันนี้ฝุ่นมาจากไหน" แบบทันใช้งานไม่ได้

นี่คือ pain ที่ทั้งโปรเจกต์เกาะ: CTM แม่นและน่าเชื่อถือ แต่ช้าเกินจะใช้รายวัน และหาอนุพันธ์ไม่ได้ (counterfactual ต้องรันใหม่) · HazeNet เรียน "พฤติกรรม" ของ CTM ลงในนิวรอล → รันเป็นวินาทีบน GPU + หาอนุพันธ์ได้ (adjoint ฟรี, จาก 0.2) · เร็วขึ้นหลักหมื่นเท่า โดยพยายามรักษาความแม่น
เชื่อม emulator concept: สิ่งที่เราทำเรียกว่า "neural emulator / surrogate" — เลียนแบบโมเดลฟิสิกส์ที่แพงด้วยนิวรอลที่ถูก · เหมือน FourCastNet/GraphCast เลียนแบบโมเดลพยากรณ์อากาศ · เราทำกับ CTM

5. Tagged Tracer / ISAM — กลไกแยกแหล่งใน CTM

จาก 1.4 เราพูดถึง "ติดป้ายสีให้ควัน" แล้ว · ในทางเทคนิคมันคือการเพิ่มตัวแปรติดตาม (tracer) หนึ่งชุดต่อแหล่ง แล้วให้ CTM ขนส่งมันไปพร้อมฝุ่นจริง:

  • WRF-Chem: ใช้ "tagged tracer" — กำหนดโซนต้นทาง แล้วติดตามควันจากแต่ละโซน
  • CMAQ: ใช้โมดูล "ISAM" (Integrated Source Apportionment Method) — ทำหน้าที่เดียวกัน

เพราะ 4 พจน์แรกของ ADR เป็นเชิงเส้น (จาก 1.2) ผลรวมของทุก tracer = ฝุ่นรวมพอดี (superposition) · นี่คือหลักการเดียวกันเป๊ะที่ HazeNet ใช้ — เราแค่ทำให้มันเร็วและหาอนุพันธ์ได้

จุดเจ็บของ tagged tracer ใน CTM: ยิ่งติดป้ายหลายโซน → ยิ่งเพิ่มตัวแปรที่ต้องขนส่ง → ยิ่งช้าและกินหน่วยความจำทวีคูณ · ใน HazeNet การเพิ่มแหล่งแทบไม่เพิ่มต้นทุน เพราะเป็น batch ของ tensor บน GPU (จาก 0.1) → นี่เป็นข้อได้เปรียบที่จับต้องได้

6. PMF — Receptor Model เจาะลึก

ทบทวนจาก 1.4: PMF เป็นคนละตระกูล (receptor — ปลายทางย้อนต้นทาง) · เจาะลึกกลไก:

  1. เก็บตัวอย่างฝุ่นจริงหลายวัน วัดองค์ประกอบเคมีแต่ละตัว (โพแทสเซียม, ซัลเฟต, EC/OC, โลหะ…)
  2. PMF แยกเมทริกซ์ข้อมูลออกเป็น "factor" = กลุ่มสารที่ขึ้น-ลงพร้อมกัน → แต่ละ factor = "ลายเซ็นแหล่งหนึ่งประเภท"
  3. ตีความ factor ด้วยความรู้: factor ที่มีโพแทสเซียมสูง = การเผาชีวมวล, วาเนเดียม/นิกเกิล = น้ำมันเตา, ฯลฯ

ชื่อเต็ม Positive Matrix Factorization — "Positive" เพราะบังคับให้สัดส่วน ≥ 0 (มวลเป็นลบไม่ได้) · คล้ายแนวคิด non-negativity ที่ HazeNet ก็บังคับ (แต่คนละบริบท)

ความต่างที่ต้องจำ: PMF บอก "ประเภทแหล่ง" (ชีวมวล/รถ/อุตสาหกรรม) · HazeNet/tagged tracer บอก "ที่ตั้งแหล่ง" (เมียนมา/ลาว/ไทย) · สองอย่างนี้เสริมกัน ไม่ใช่แทนกัน — PMF ตอบ "เผาอะไร" เราตอบ "เผาที่ไหน"
บทบาทในงานเรา: PMF เชียงใหม่ (ชีวมวล ~51%) เป็น ground truth เสริม — ยืนยันว่า "ส่วนใหญ่เป็นการเผา" สอดคล้องกับสมมติฐานที่เราเน้น primary smoke จากไฟ · ใช้ cross-check ความสมเหตุสมผล ไม่ใช่ตัวเทียบสัดส่วนภูมิศาสตร์โดยตรง

8. บทบาทในงานเรา — คู่แข่ง + ครู + เฉลย

เครื่องมือเหล่านี้เล่น 3 บทบาทพร้อมกันในโปรเจกต์เรา:

  • คู่แข่ง (baseline): เราต้องแสดงว่า HazeNet เร็วกว่ามาก โดยความแม่นยังแข่งได้ — นี่คือ selling point หลัก
  • ครู (reference physics): โครงสร้าง HazeNet เลียนแบบ CTM (operator splitting, advection, tagged tracer) → ทำให้เราได้คุณสมบัติฟิสิกส์มาฟรี ไม่ต้องให้นิวรอลเรียนเองทั้งหมด
  • เฉลย (ground truth): ผล tagged tracer (~45%/~30%) และ PMF (~51% ชีวมวล) ใช้ตรวจว่า attribution ของเราตรงฟิสิกส์ (validation C2)
ทำไมต้องชัดเรื่องบทบาท: ในเปเปอร์ A* reviewer จะถามทันทีว่า "แล้วต่างจาก WRF-Chem ยังไง" · คำตอบที่ดีไม่ใช่ "เราดีกว่าทุกอย่าง" (ไม่จริง) แต่คือ "เราเป็น emulator ที่เร็วหลักหมื่นเท่า + หาอนุพันธ์ได้ ทำให้ทำสิ่งที่ CTM ทำไม่ไหวรายวัน (attribution + counterfactual + inversion) ได้จริง โดย validate กับ CTM แล้วใกล้เคียง" — บทนี้ให้คุณตอบแบบนั้นได้

9. caveat การใช้ ground truth (ความซื่อสัตย์)

  • CTM ก็ไม่ใช่ความจริงสัมบูรณ์ — มันเป็นแบบจำลอง มี error ของมันเอง (จากบัญชีการปล่อย, ความละเอียด) · เราใช้มันเป็น "reference" ไม่ใช่ "ความจริง 100%"
  • คนละความละเอียด/ปี — ผลไทยส่วนใหญ่ ~20 กม. และคนละปีกับเรา → เทียบ "ระดับสัดส่วน" ได้ แต่ "รายวันรายช่องเป๊ะ" ต้องระวัง (ย้ำจาก 1.4)
  • การวัดจริงคือเฉลยสูงสุด — สำหรับ "PM2.5 พยากรณ์แม่นไหม" เฉลยที่ดีที่สุดคือสถานี Air4Thai จริง ไม่ใช่ CTM · ใช้ CTM เป็นเฉลยเฉพาะ "การแยกแหล่ง" ที่วัดตรงไม่ได้
หลักการเลือกเฉลย: "PM2.5 แม่นไหม?" → เทียบสถานีจริง · "แยกแหล่งถูกไหม?" → เทียบCTM (เพราะวัดตรงไม่ได้) · "เผาอะไรเป็นหลัก?" → cross-check PMF · เลือกเฉลยให้ตรงกับคำถาม = หัวใจของ error analysis ที่ดี (จะลงลึกใน Module 5)

10. Cheat sheet + แบบฝึกหัด

สรุปจำ:
  • CTM = โปรแกรมแก้สมการ ADR ทั้งกริด → ได้ PM2.5 ทุกจุด
  • WRF-Chem = CTM online (อากาศ+เคมีคุยกัน) · ground truth หลักของเรา (~45% ข้ามแดน)
  • CMAQ = CTM offline (รันอากาศก่อน) · แยกแหล่งด้วย ISAM
  • ทำไมช้า = numerical solver ทุก step ทุกช่อง + เคมี stiff + CPU → ชม.–วัน
  • tagged tracer/ISAM = ติดป้ายควันตามโซน → แยกที่ตั้ง (อาศัย superposition)
  • PMF = receptor, แยก "ประเภท" (ชีวมวล/น้ำมัน) จากเคมี · ไม่บอกที่ตั้ง
  • HazeNet = neural CTM: เร็ว(วินาที) + พยากรณ์ + adjoint ฟรี · validate กับ CTM/PMF
  • เลือกเฉลย: PM2.5→สถานีจริง · แยกแหล่ง→CTM · เผาอะไร→PMF

แบบฝึกหัด 1 — เพื่อนบอก "ถ้า WRF-Chem แม่นกว่า ก็ใช้ WRF-Chem สิ จะสร้าง HazeNet ทำไม" · ตอบยังไงใน 2 ประโยค?

เฉลย

(1) WRF-Chem ช้าหลักชั่วโมง–วันต่อเหตุการณ์ ใช้พยากรณ์+แยกแหล่งรายวันทันใช้งานไม่ได้ และหาอนุพันธ์ไม่ได้จึงทำ counterfactual/inversion ไม่ไหว · (2) HazeNet เป็น emulator ที่เรียนพฤติกรรม WRF-Chem มา → เร็วเป็นวินาทีบน GPU เดียว + หาอนุพันธ์ได้ ทำให้ทำสิ่งที่ CTM ทำไม่ไหวรายวันได้จริง โดยความแม่นยัง validate กับ WRF-Chem แล้วใกล้เคียง

แบบฝึกหัด 2 — เราถาม "ฝุ่นเชียงใหม่วันนี้กี่ % มาจากเมียนมา" · ควรใช้ PMF หรือ tagged tracer เป็นเฉลย? เพราะอะไร?

เฉลย

tagged tracer (CTM) · เพราะคำถามถาม "ที่ตั้ง" (เมียนมา) ซึ่งเป็นเรื่องภูมิศาสตร์+ลม · PMF บอกได้แค่ "ประเภท" (ชีวมวล) แยกเมียนมา-vs-ไทยไม่ได้เพราะเคมีเหมือนกัน · เลือกเฉลยให้ตรงคำถาม

แบบฝึกหัด 3 — ทำไมการ "เพิ่มแหล่งให้แยก" ถึงแพงใน WRF-Chem แต่แทบฟรีใน HazeNet?

เฉลย

WRF-Chem ต้องเพิ่มตัวแปร tracer ใหม่ที่ต้องขนส่งเชิงตัวเลขทุก step → คำนวณ+หน่วยความจำเพิ่มทวีคูณ · HazeNet ทำ tracer เป็นมิติ batch ของ tensor บน GPU (จาก 0.1) → ประมวลผลขนานทีเดียว เพิ่มแหล่งแทบไม่เพิ่มเวลา · ทั้งคู่อาศัย superposition เหมือนกัน แต่ฮาร์ดแวร์+การ implement ต่างกัน
