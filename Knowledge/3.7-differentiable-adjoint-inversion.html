1. ทบทวน 3.6 และตั้งโจทย์ใหม่

ใน 3.6 เราแก้ Forward Problem: รู้แหล่งไฟแล้ว → ตามว่าฝุ่นไปถึงไหน

3.6 Forward

รู้: ไฟอยู่ที่พม่า ลาว ไทย

หา: ฝุ่นจะไปถึงสถานีไหนเท่าไหร่

Q (emissions) → HazeNet → C (concentrations)

3.7 Inverse

รู้: สถานีวัด PM2.5 ได้ค่านี้

หา: ไฟต้องอยู่ที่ไหน ปริมาณเท่าไหร่?

C_obs (measured) → ??? → Q* (estimated)

ตัวอย่างโจทย์จริง:

  • สถานีวัด 8 แห่งทั่วภาคเหนือไทยรายงาน PM2.5 ณ เวลา t
  • ดาวเทียม MODIS มีเมฆปกคลุม 60% → ไม่เห็น FRP ตรงๆ
  • คำถาม: ไฟต้องมีอยู่ที่พิกัดไหน ปล่อยควันเท่าไหร่ จึงทำให้ค่าสถานีเป็นแบบนี้?

→ Source Inversion: ย้อนกลับจากการวัดไปหาแหล่ง

2. Inverse Problem คืออะไร — นิยามอย่างเป็นทางการ

2.1 สัญลักษณ์

สัญลักษณ์ความหมายขนาด (ตัวอย่าง)
Q(x,t)Emission field — ปริมาณฝุ่นที่ปล่อยจากแต่ละจุด(H×W) = 128×128
C(x,t)Concentration field — ฝุ่นสะสมทั่ว domain(H×W×T)
MObservation operator — เลือกค่าที่ตำแหน่งสถานีวัด(n_stations × H×W)
y_obsค่าที่วัดได้จริง + noise(n_stations × T)
HazeNet(Q)Forward model — Q → Cฟังก์ชันที่ frozen

2.2 สูตรปัญหา

Forward: y_obs = M · HazeNet(Q) + ε
Inverse: หา Q* = argminQ J(Q)

โดย cost function J(Q) คือ:

J(Q) = ||M · HazeNet(Q) - y_obs||²  +  λ·R(Q)
          ↑                                ↑
       Data fit term                  Regularization term
    (prediction ใกล้ observation)     (Q ต้องสมเหตุสมผล)
ทำไม minimize แทนที่จะ solve ตรงๆ: เพราะ y_obs มี noise + ระบบ underdetermined (สถานีน้อยกว่า pixel มาก) → ไม่มี unique solution → แทนที่หาคำตอบเดียว เราหาคำตอบที่ "ดีที่สุด" ตาม criteria ที่กำหนด

3. ทำไมถึงยาก — Ill-posed Problem

3.1 Ill-posed คืออะไร

ปัญหาเรียกว่า well-posed ถ้า (1) มีคำตอบ (2) คำตอบ unique (3) stable ต่อ noise
Inverse problem ฝุ่นมักเป็น ill-posed — ล้มเหลวทั้ง 3 ข้อ:

เงื่อนไขปัญหาในงานเราผลที่ตามมา
มีคำตอบถ้า y_obs มี measurement error มาก → อาจไม่มี Q ใดที่ fit ได้พอดีต้องผ่อนจาก = เป็น ≈
Unique solution128×128 = 16,384 unknowns · แต่สถานีแค่ 8 แห่ง → severely underdeterminedหลาย Q ให้ y_obs เหมือนกัน
Stablenoise เล็กน้อยใน y_obs → Q* เปลี่ยนมาก ("butterfly effect ย้อนกลับ")ต้องการ regularization

3.2 อุปมา: ถอดเสียงจากการสะท้อน

จินตนาการห้องมีลำโพง 5 ตัว (แหล่ง Q) และไมค์ 3 ตัว (สถานีวัด)
ไมค์ได้ยินเสียงผสมของลำโพงทุกตัว — แต่จะ decode ได้ไหมว่าลำโพงแต่ละตัวเล่นอะไร?

  • ลำโพง 5 ตัว > ไมค์ 3 ตัว → underdetermined → หลายคำตอบเป็นไปได้
  • เพิ่มข้อสมมติ: "เสียงต้องเป็นคำพูดภาษาไทย" → ลด solution space ลง (regularization!)
  • ยิ่งมีไมค์มาก (สถานีมาก) → ยิ่งแก้ได้แม่นขึ้น

3.3 ตัวเลขจริงที่ทำให้เห็นภาพ

# ปัญหาของเรา
n_unknowns = 128 * 128   # = 16,384 pixel emissions ที่ต้องหา
n_stations = 8            # สถานีวัดจริงในภาคเหนือไทย
n_timesteps = 24          # 24 ชั่วโมงที่มีข้อมูล

n_observations = n_stations * n_timesteps  # = 192

4. Differentiable Programming — โปรแกรมที่ differentiate ได้ทั้งหมด

4.1 แนวคิด

ปกติ เวลาเราเขียนโปรแกรม เราคิดว่า: "รับ input → ประมวลผล → ได้ output"
Differentiable Programming เพิ่มความสามารถ: "หาว่า output เปลี่ยนยังไงถ้า input เปลี่ยนนิดเดียว"

กล่าวได้ว่า โปรแกรมทุกขั้นตอนถูก build เป็น computation graph อัตโนมัติ (จาก 0.2!)
ดังนั้นเราสามารถ .backward() ผ่าน ทุกอย่าง ที่โปรแกรมทำ — ไม่ใช่แค่ neural network แต่รวม:

  • สมการฟิสิกส์ (advection, diffusion)
  • การ interpolate ข้อมูล
  • การ select ค่าที่สถานีวัด
  • การคำนวณ loss

ทุกขั้นตอนรวมกันเป็น "ท่อ gradient" ต่อเนื่อง

4.2 เทียบกับ Traditional Approach

วิธีเก่า (Classical Adjoint)

ขั้นตอน:

  1. เขียน forward model (เช่น WRF-Chem)
  2. เขียน adjoint code แยก (transpose ของ forward)
  3. ตรวจสอบ adjoint ด้วยมือ
  4. Compile และ test

ใช้เวลา 1–3 ปี ต่อ 1 โมเดล · ทีม 5+ คน

Differentiable Programming

ขั้นตอน:

  1. เขียน forward model ใน PyTorch
  2. เรียก .backward()

 

 

ใช้เวลา 0 วัน เพิ่ม · ได้ adjoint ฟรีทันที

ทำไม HazeNet ถึงทำได้: เราเขียน HazeNet ใน PyTorch ตั้งแต่ต้น → ทุก operation ถูก track ใน computation graph → .backward() ทะลุผ่านฟิสิกส์ (Semi-Lagrangian, diffusion) และ neural network ได้ในครั้งเดียว

5. Adjoint Method — คณิตศาสตร์ (ที่ PyTorch ทำให้เราไม่ต้องเขียน)

ส่วนนี้อธิบายว่า "adjoint" คืออะไรในทางคณิตศาสตร์ เพื่อให้เข้าใจว่า .backward() ทำอะไร — ไม่ต้องจำสูตร แค่เข้าใจแนวคิด

5.1 Forward Model เป็น Chain

HazeNet rollout 48 ชั่วโมงคือการทำซ้ำ:

C(0) ──F──→ C(1) ──F──→ C(2) ──F──→ … ──F──→ C(T)
  ↑                                              ↓
emission Q                               y = M·C(T)
                                         loss = ||y - y_obs||²

5.2 Adjoint = Backward ผ่าน Chain

ต้องการหา ∂loss/∂Q ซึ่งต้องใช้ chain rule ผ่าน T ขั้น:

∂loss/∂Q = ∂loss/∂C(T) · ∂C(T)/∂C(T-1) · … · ∂C(1)/∂C(0) · ∂C(0)/∂Q
               ↑              ↑                      ↑              ↑
           เริ่มที่นี่       adjoint                adjoint       ผลลัพธ์
                           variable λ(T-1)         variable λ(0)

ใน classical method ต้องกำหนด adjoint variable λ(t) และแก้สมการ adjoint ย้อนเวลา:

λ(T) = M^T · (M·C(T) - y_obs)       ← เริ่มจาก loss gradient
λ(t) = F^T(t+1) · λ(t+1)            ← propagate ย้อนเวลา
∂loss/∂Q = G^T · λ(0)               ← sensitivity ต่อ emission

5.3 ทำไม F^T (Transpose) คือปัญหา

ความยากของ Classical Adjoint:
ถ้า F คือ Semi-Lagrangian advection (2.2) + AFNO neural operator (3.2) + chemistry
→ F^T ต้องคำนวณ transpose ของแต่ละ operation ทั้งหมด → เขียนใหม่ทั้งหมดจากศูนย์
→ ทีม ECMWF ใช้เวลา 7 ปี เขียน adjoint ของ IFS (Integrated Forecast System)

6. ความลับ — Adjoint = .backward() ทุกประการ

Insight สำคัญที่สุดในบทนี้:

Adjoint method ที่นักวิทยาศาสตร์บรรยากาศพัฒนาทศวรรษ คือสิ่งเดียวกัน กับ backpropagation ที่ deep learning community พัฒนาแยกกัน

ทั้งคู่คือ chain rule ย้อนกลับผ่าน computation graph

λ(t) = F^T · λ(t+1) ← classical adjoint
loss.backward() ← PyTorch autograd

คณิตศาสตร์เหมือนกัน 100% — ต่างกันแค่ใครเขียนโค้ด

7. Source Inversion = "Train Q ไม่ใช่ Train W"

7.1 เทียบกับการ Train Neural Network (0.2)

จำจาก 0.2: training loop คือการปรับ weight W ให้ loss ลด
Source Inversion คือการปรับ emission Q ให้ loss ลด — กลไกเหมือนกันทุกอย่าง:

Neural Network Training (0.2) 
# W คือสิ่งที่ optimize
model = TinyNet()
opt = Adam(model.parameters(), lr=1e-3)

for epoch in range(1000):
    y_pred = model(x_train)     # forward
    loss = criterion(y_pred, y_train)
    opt.zero_grad()
    loss.backward()            # ∂loss/∂W
    opt.step()                  # update W

→ W เปลี่ยน · model(x) → y_train ได้ดีขึ้น

Source Inversion (3.7) 
# Q คือสิ่งที่ optimize, W frozen
Q = torch.zeros(H, W, requires_grad=True)
opt = Adam([Q], lr=1e-2)

for iteration in range(1000):
    C_pred = hazenet(Q, met)    # forward
    loss = obs_loss(C_pred, y_obs)
    opt.zero_grad()
    loss.backward()            # ∂loss/∂Q  ← adjoint!
    opt.step()                  # update Q

→ Q เปลี่ยน · hazenet(Q) → y_obs ได้ดีขึ้น

สิ่งที่ต่างกันเพียงอย่างเดียว:
ใน training: requires_grad=True ที่ model.parameters()
ใน inversion: requires_grad=True ที่ Q (input)

PyTorch ไม่สนว่า gradient ไหลไปหาอะไร — มันแค่ทำ chain rule ย้อนกลับ

8. โค้ดเต็มของ Source Inversion Loop

8.1 Observation Operator M

import torch
import torch.nn as nn

def observation_operator(C_field, station_coords):
    """
    C_field: (H, W) — concentration field ทั้ง domain
    station_coords: list of (lat_idx, lon_idx) — ตำแหน่งสถานีวัด
    returns: (n_stations,) — ค่าที่สถานีแต่ละแห่ง
    """
    values = []
    for (i, j) in station_coords:
        values.append(C_field[i, j])
    return torch.stack(values)

# ตำแหน่งสถานีวัด 8 แห่งในภาคเหนือไทย (row, col บน 128×128 grid)
stations = [(45, 32), (48, 35), (52, 30), (55, 38),
            (43, 41), (60, 33), (50, 27), (46, 44)]

8.2 Loss Function รวม Regularization

def inversion_loss(Q, hazenet, met_sequence, y_obs, lambda_reg=1e-3):
    """
    Q:            (H, W) — emission field ที่ optimize
    hazenet:      frozen model
    met_sequence: (T, n_met, H, W) — อุตุนิยมแต่ละชั่วโมง
    y_obs:        (T, n_stations) — ค่าวัดจริง
    lambda_reg:   น้ำหนัก regularization
    """
    T = met_sequence.shape[0]
    C = Q.clamp(min=0)  # physical constraint: emission ≥ 0

    total_obs_loss = torch.tensor(0.0)

    # Rollout forward model
    for t in range(T):
        C = hazenet(C, met_sequence[t])
        y_pred = observation_operator(C, stations)  # (n_stations,)
        total_obs_loss += nn.functional.mse_loss(y_pred, y_obs[t])

    # Regularization: L2 → smooth emission field
    reg_loss = lambda_reg * (Q ** 2).sum()

    return total_obs_loss + reg_loss, total_obs_loss.item()

8.3 Inversion Loop หลัก

# === Setup ===
H, W = 128, 128

# === Inversion Loop ===
history_loss = []
for iteration in range(500):
    optimizer.zero_grad()

    loss, obs_loss = inversion_loss(Q, hazenet, met_seq, y_obs, lambda_reg=1e-3)

    loss.backward()    # ← ADJOINT: ∂loss/∂Q ผ่าน HazeNet ทั้งหมด
    optimizer.step()   # ← update Q ให้ fit observation ดีขึ้น
    scheduler.step()

    if iteration % 50 == 0:
        print(f"iter {iteration:4d} | loss={loss.item():.4f} | obs={obs_loss:.4f} | max_Q={Q.max().item():.2f}")
        history_loss.append(obs_loss)

# Q* = estimated emission field
Q_star = Q.detach().clamp(min=0)
print(f"Inversion done. Max emission: {Q_star.max():.2f} μg/m²")

9. Regularization — ควบคุม Ill-posedness

Regularization คือการเพิ่ม "ข้อสมมติเกี่ยวกับ Q" เข้าไปใน loss เพื่อให้ solution สมเหตุสมผล:

9.1 ประเภทของ Regularization

ชื่อสูตรผลที่ได้ใช้เมื่อ
L2 / Tikhonovλ·||Q||²Q ราบเรียบ ค่าเล็กไม่รู้ว่า fire กระจุกหรือกระจาย
L1 / LASSOλ·||Q||₁Q sparse (เกือบ 0 ทุกที่ ยกเว้นจุดไฟจริง)รู้ว่าไฟมีน้อยจุด concentrated
TV (Total Variation)λ·Σ|∇Q|Q ราบเรียบภายใน cluster ขอบคมชัดไฟเป็น patch ต่อเนื่อง
Non-negativityQ ≥ 0ไม่มีไฟ "ดูดควันกลับ"เสมอ (physical constraint)
Satellite priorλ·||Q - Q_FRP||²ใกล้ FRP ดาวเทียมมี FRP บางส่วน (ไม่มีเมฆ)

9.2 โค้ด Regularization Options

def regularization(Q, Q_prior=None, mode='l2', lam=1e-3):
    Q_pos = Q.clamp(min=0)  # non-negativity

    if mode == 'l2':
        return lam * (Q_pos**2).sum()

    elif mode == 'l1':
        return lam * Q_pos.abs().sum()  # sparse fire map

    elif mode == 'tv':
        # TV = sum of gradient magnitudes
        dy = (Q_pos[1:, :] - Q_pos[:-1, :]).abs().sum()
        dx = (Q_pos[:, 1:] - Q_pos[:, :-1]).abs().sum()
        return lam * (dy + dx)

    elif mode == 'prior' and Q_prior is not None:
        return lam * ((Q_pos - Q_prior)**2).sum()  # pull toward FRP

# ใช้งาน: ใน HazeNet เราแนะนำ L1 + prior (ถ้ามีดาวเทียม)
loss_reg = regularization(Q, Q_prior=Q_frp, mode='prior', lam=1e-3) \
         + regularization(Q, mode='l1', lam=5e-4)

10. Sensitivity Map — ∂(PM2.5)/∂(fire)

10.1 คืออะไร

นอกจาก source inversion แล้ว adjoint ยังให้ Sensitivity Map:
"ถ้าเพิ่มไฟที่พิกัด (i,j) ขึ้น 1 หน่วย PM2.5 ที่เชียงใหม่จะเปลี่ยนเท่าไหร่?"

# คำนวณ Sensitivity Map — โค้ดสั้นมาก!
Q_test = torch.zeros(H, W, requires_grad=True)

# Forward: rollout ทั้ง 48 ชั่วโมง
C = Q_test
for t in range(48):
    C = hazenet(C, met[t])

# ฝุ่นสะสมที่สถานีเชียงใหม่ (เลือก pixel เดียว)
cm_lat, cm_lon = 48, 35
pm25_CM = C[cm_lat, cm_lon]  # scalar

# Backward: adjoint ผ่านทั้ง 48 timestep
pm25_CM.backward()

# Q_test.grad[i,j] = ∂(PM2.5 ที่เชียงใหม่)/∂(fire ที่ pixel (i,j))
sensitivity_map = Q_test.grad  # (H, W)
print(f"Sensitivity at Myanmar border: {sensitivity_map[55,28]:.4f}")
# → 0.0382  (ไฟ 1 μg/m² ที่พม่าชายแดน → เชียงใหม่เพิ่ม 0.038 μg/m²)

10.2 ผลที่ได้ใช้งานได้ทันที

ประโยชน์วิธีใช้ sensitivity_map
ระบุพิกัดเสี่ยงสูงจุดที่ sensitivity สูง = ไฟที่นั่นมีผลต่อเชียงใหม่มาก → ลาดตระเวนก่อน
ลำดับความสำคัญดับไฟsort จุดไฟ (FRP) × sensitivity → ดับไฟที่ impact × probability สูงสุดก่อน
ออกแบบสถานีวัดตั้งสถานีใหม่ที่ sensitivity หลาย source ล้ำกัน = ได้ข้อมูลมีค่าสูงสุด
Counterfactual เร็วΔPM2.5 ≈ sensitivity × ΔQ (แทนที่จะ run forward ใหม่)

10.3 Sensitivity ตาม Lead Time

# sensitivity เปลี่ยนตาม lead time ที่ต้องการ
sensitivity_by_lead = {}
for lead in [6, 12, 24, 48]:  # ชั่วโมง
    Q_test = torch.zeros(H, W, requires_grad=True)
    C = Q_test
    for t in range(lead):
        C = hazenet(C, met[t])
    C[cm_lat, cm_lon].backward()
    sensitivity_by_lead[lead] = Q_test.grad.clone()

# t+6h: sensitivity กระจุกใกล้เชียงใหม่ (ลมยังไม่พาไปไกล)
# t+48h: sensitivity กว้างกว่า ครอบคลุมพม่า/ลาวด้วย

12. ข้อจำกัดซื่อสัตย์

12.1 Model Error Propagates

ปัญหา: ถ้า HazeNet ทำนาย C ผิด (model error) → inversion จะหา Q* ที่ "ชดเชย error" แทนที่จะหาแหล่งจริง

ตัวอย่าง: HazeNet underestimate dispersion ทิศเหนือ → inversion จะเพิ่ม Q ทางเหนือเพื่อชดเชย → เราเห็น "phantom fire" ที่ไม่มีจริง

ลดผลกระทบ: validate HazeNet ก่อน (Module 5) + ใช้ observation ที่หลากหลาย (หลาย station หลาย level)

12.2 Local Minima

ปัญหา: J(Q) อาจมีหลาย local minimum — optimizer อาจ stuck ที่ผิด

วิธีแก้:
  • ใช้ FRP ดาวเทียมเป็น warm start (initial guess ดีกว่า zeros)
  • Annealing: ลด regularization ช้าๆ ตาม iteration
  • Run หลาย inversion จาก initial guess ต่างกัน → เลือก best

12.3 Observation Sparsity

ปัญหา: สถานีวัด 8 แห่ง vs 16,384 pixel unknowns → severely underdetermined แม้มี regularization

ข้อมูลเสริมที่ช่วยได้:
  • AOD จาก MODIS/MAIAC (spatial coverage ดีกว่า แต่ noise)
  • Sentinel-5P (tropomi): NO₂/CO column → proxy ของ fire
  • เพิ่มจำนวนสถานีวัดจาก crowdsourcing sensor

12.4 Memory ของ Backprop ผ่านเวลา

ปัญหา: backprop ผ่าน 48 timestep ต้องเก็บ computation graph ทุก timestep
Memory ∝ T × (H × W × n_channels) → อาจ OOM บน A100 ถ้า T ใหญ่

แก้ด้วย:
  • Gradient checkpointing: เก็บแค่บาง timestep, recompute ที่เหลือระหว่าง backward
  • ลด T: ทำ inversion ทีละ 12h แทน 48h
import torch.utils.checkpoint as checkpoint

# แทนที่: C = hazenet(C, met[t])
C = checkpoint.checkpoint(hazenet, C, met[t])
# → ประหยัด RAM ~10× แลกกับ compute เพิ่ม 30%

13. แบบฝึกหัด

Q1: ทบทวนแนวคิด

Source inversion และ neural network training ต่างกันอย่างไร? เหมือนกันอย่างไร? อธิบายในแง่ของ "อะไรที่ requires_grad=True"

ดูเฉลย

เหมือนกัน: ทั้งคู่ใช้ optimizer.zero_grad() → forward → loss.backward() → optimizer.step() — โครงสร้าง 5 ขั้นตอนเดิม (จาก 0.2) ทุกประการ

ต่างกัน:
- Training: model.parameters() มี requires_grad=True, Q (input) fixed
- Inversion: Q มี requires_grad=True, model.parameters() frozen (requires_grad=False)
กล่าวคือ "สิ่งที่ optimize" สลับกัน — PyTorch ไม่สนใจ แค่ backprop ตาม graph

Q2: Regularization Choice

สถานการณ์: ดาวเทียม FIRMS ตรวจพบ 12 จุดไฟใน domain แต่เมฆปกคลุมบางพื้นที่ — ควรใช้ regularization แบบไหน และทำไม?

ดูเฉลย

แนะนำ: L1 + FRP prior
- L1: รู้ว่าไฟมีน้อยจุด (sparse) → L1 ดัน Q เป็น 0 ที่ส่วนใหญ่ ยกเว้นจุดไฟจริง
- FRP prior: ส่วนที่ดาวเทียมเห็นได้ (ไม่มีเมฆ) ใช้เป็น anchor → λ·||Q - Q_FRP||²
- Non-negativity: เสมอ (physical)
ไม่ใช้ L2 เพียงอย่างเดียวเพราะ L2 ทำให้ Q กระจายทั่วไปแทนที่จะ concentrated ที่จุดไฟ

Q3: Memory Budget

Inversion loop ผ่าน T=72 timestep บน grid 256×256 ถ้าไม่ใช้ gradient checkpointing คร่าวๆ computation graph ต้องเก็บ activation กี่ค่า (สมมติแต่ละ timestep เก็บ 1 field C)?

ดูเฉลย

72 timestep × 256 × 256 × 4 bytes = 72 × 262,144 × 4 = 75,497,472 bytes ≈ 72 MB สำหรับ C เพียงอย่างเดียว
แต่ในความเป็นจริง HazeNet เก็บ intermediate activations ทุก layer ด้วย → อาจ ×10–100 → ~720 MB – 7.2 GB
บน A100 (40 GB) ยังพอ แต่ถ้า batch หลาย trajectory พร้อมกันต้องใช้ checkpointing

Q4: Sensitivity Map

ถ้า sensitivity_map[30, 50] = 0.12 และ sensitivity_map[80, 90] = 0.003 หมายความว่าอะไร? และถ้าดาวเทียมพบไฟ 50 MW ที่ทั้งสองตำแหน่ง ตำแหน่งไหนควรดับก่อน?

ดูเฉลย

- sensitivity[30,50] = 0.12: ไฟ 1 μg/m² ที่ pixel (30,50) ทำให้ PM2.5 ที่เชียงใหม่เพิ่ม 0.12 μg/m³
- sensitivity[80,90] = 0.003: ไฟที่ (80,90) มีผลน้อยกว่า 40× เพราะอาจอยู่คนละทิศลมหรือไกลกว่า
ควรดับ (30,50) ก่อน เพราะ impact สูงกว่ามาก แม้ fire intensity (50 MW) เท่ากัน
Impact estimate: 50 MW × conversion × sensitivity ≈ ΔPM2.5 ที่เชียงใหม่

Q5: Adjoint คือ Backprop

ทีม WRF-Chem ใช้เวลา 5 ปีพัฒนา adjoint code ในขณะที่ HazeNet ได้ฟรี — จงอธิบายว่าทำไม และมีข้อแลกเปลี่ยนอะไรบ้าง?

ดูเฉลย

ทำไมฟรี: HazeNet เขียนใน PyTorch ซึ่งมี automatic differentiation — ทุก operation ถูก record ใน computation graph ระหว่าง forward pass → .backward() traverse graph ย้อนกลับโดยอัตโนมัติ (chain rule) = adjoint โดยไม่ต้องเขียนสูตรเพิ่ม

ข้อแลกเปลี่ยน:
- HazeNet ต้องเขียนใน PyTorch ทั้งหมด (ใช้ CTM เดิมโดยตรงไม่ได้)
- Physics accuracy ต่ำกว่า WRF (เป็น emulator ไม่ใช่ first-principles)
- Memory เพิ่มขึ้นตาม T (gradient checkpointing ช่วยได้)
- แต่ได้ speed และ development time ที่ดีกว่ามาก
