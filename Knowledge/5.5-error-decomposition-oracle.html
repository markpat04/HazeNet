5.5 — Error Decomposition / Oracle Substitution ⭐ CORE 🔴 ยาก

เวลาโมเดลผิด — ผิดเพราะอะไร? แยก error รวมว่ามาจาก "ข้อมูล / ลม / โมเดลเรา" ส่วนไหน

เสาหลักของ rigor (ยกงานเป็นระดับวิจัย A*) · ตรงข้ามกับ 5.4 (ablation=พื้น / oracle=เพดาน) · โยง FastCTM modules · 1.3 (ลม) · 4.1 (emission) · 3.4 (rollout) · 6.1 (attribution error bound)

โจทย์ที่ตัวเลขเดียวตอบไม่ได้: "RMSE = 12 μg/m³" บอกแค่ว่า ผิดเท่าไร ไม่ได้บอกว่า ผิดเพราะอะไร · ควรไปแก้ข้อมูลไฟ? แก้ลม? หรือแก้โมเดล? — ถ้าไม่รู้ จะเสียเวลา (และงบ A100) ปรับผิดที่

2 เครื่องมือของบทนี้:
① Error decomposition — แตก error รวมเป็นชิ้น ๆ ตาม "ที่มา" (process / พื้นที่ / เวลา / regime)
② Oracle substitution — แทนชิ้นหนึ่งด้วย "คำตอบจริง (perfect)" แล้วดูว่า error ลดเท่าไร → ชิ้นนั้นกำลังทำให้เราเสียเท่านั้น

ผลลัพธ์ที่ทรงพลัง: เปลี่ยน "โมเดลเรา RMSE 12" เป็น "ในนั้น ~5 มาจากความไม่แน่นอนของ emission, ~4 จากลมพยากรณ์ผิด, เหลือ ~3 เป็นของโมเดลเราเอง — แปลว่าโมเดลเกือบถึงเพดานแล้วเมื่อเทียบกับคุณภาพ input ที่มี"

1. ทำไม "RMSE เดียว" ไม่พอ ⭐

RMSE รวมคือ "อาการ" ไม่ใช่ "สาเหตุ" · เหมือนหมอบอก "คุณป่วย" โดยไม่บอกว่าเป็นอะไร · เราต้องวินิจฉัยว่า error ก้อนนั้น มาจากไหน ถึงจะรักษาถูกจุด

ทำไมสำคัญเชิงปฏิบัติ: ถ้า error 60% มาจาก "emission ไฟไม่แม่น" แต่เราดันไปทุ่มเวลาปรับสถาปัตยกรรมโมเดล → เสียงบ A100 ไปกับสิ่งที่ขยับผลได้นิดเดียว · error decomposition คือ "แผนที่บอกว่าควรลงแรงตรงไหน" — มีค่ามหาศาลเมื่อทรัพยากรจำกัด

2. โมเดลเรา = ลูกโซ่ของ process → error เข้าได้ทุกข้อต่อ

HazeNet (เหมือน FastCTM) คือลูกโซ่: emission → transport(ลม) → chemistry → deposition → ค่าออก · error เข้ามาได้ที่ทุกข้อต่อ แล้ว สะสมต่อกัน:

กุญแจของวิธีคิด: ถ้าเรา "เปลี่ยน input ของข้อต่อหนึ่งเป็นคำตอบจริง" แล้ว error รวมลดฮวบ → ข้อต่อนั้นคือตัวการ · ถ้าเปลี่ยนแล้วแทบไม่ขยับ → ข้อต่อนั้นไม่ใช่ปัญหา · นี่คือ logic ของ oracle substitution

3. Oracle Substitution — แทนชิ้นด้วย "คำตอบจริง" ⭐

Oracle = "ผู้รู้คำตอบที่สมบูรณ์แบบ" · วิธี: แทน input/ชิ้นส่วนหนึ่งด้วยค่าจริง (ground truth) แล้วรันโมเดลใหม่ · error ที่ ลดลง = "ส่วนที่ชิ้นนั้นกำลังทำให้เราเสียอยู่ตอนนี้"

คู่กับ ablation (5.4) ได้ภาพครบ:

  • Ablation = เอาชิ้นออก → error พุ่ง → บอก "พื้น" (ขาดชิ้นนี้แย่แค่ไหน)
  • Oracle = ทำชิ้นให้เป๊ะ → error ลด → บอก "เพดาน" (ชิ้นนี้ถ้าสมบูรณ์ดีได้แค่ไหน)
ช่องว่างคือคำตอบ: ระยะจาก "โมเดลจริง" ไป "oracle" = ศักยภาพที่เหลืออยู่ของชิ้นนั้น · ถ้า oracle emission ทำให้ error ลดจาก 12→7 แสดงว่า "ปรับปรุง emission ให้ดีขึ้น = ได้กำไรถึง 5 หน่วย" → คุ้มลงแรง · ถ้า oracle เกือบไม่ขยับ → ชิ้นนั้นไม่ใช่คอขวด อย่าเสียเวลา

4. Oracle ของ HazeNet — ทำจริงได้แบบไหนบ้าง ⭐⭐

Oracleแทนด้วยตอบคำถามถ้า error ลดมาก แปลว่า
Oracle emissionemission จริง/ดีที่สุดที่หาได้ (เช่น FINN ละเอียด) แทน FIRMS ดิบข้อมูลไฟคือคอขวดไหมลงแรงปรับ emission (FINN, 4.1/8)
Oracle meteorologyลม/PBL จริง (ERA5 reanalysis) แทนลม "พยากรณ์"error มาจาก "อากาศทายผิด" เท่าไรขีดจำกัดอยู่ที่ NWP ไม่ใช่โมเดลเรา (ดู §5)
Oracle initial stateสภาพ PM2.5 จริงที่ t=0error มาจาก "จุดเริ่มผิด" หรือ "สะสมระหว่างทาง"ปัญหาอยู่ที่ initialization/assimilation
Oracle boundary inflowค่าควันจริงที่ขอบโดเมน (ต้นทางข้ามแดน)ควันเข้าจากนอกกรอบทำให้พลาดเท่าไรต้องขยายโดเมน/ปรับ boundary (โยง 1.4)
เคล็ดสำคัญเรื่องต้นทุน: oracle หลายตัว ไม่ต้องเทรนใหม่ — แค่ "สลับ input ตอน inference" แล้ววัดผล (เช่น ใส่ลม reanalysis แทนลมพยากรณ์) → ถูกกว่า ablation มาก (ablation ต้องเทรนใหม่) · oracle ที่เปลี่ยนแค่ขา input คือการทดลองที่คุ้มงบ A100 ที่สุดในโมดูล 5

5. แยก "ลมพยากรณ์ผิด" ออกจาก "โมเดลเราผิด" — insight ที่เปลี่ยนเกม ⭐

นี่คือ oracle ที่สำคัญที่สุดของงานพยากรณ์อากาศ/มลพิษ · เวลาเราพยากรณ์ PM2.5 ล่วงหน้า เราต้องใช้ "ลมที่พยากรณ์ไว้" (ซึ่งก็ผิดของมันเอง) · ดังนั้น error รวมของเรา = error ของเรา + error ที่สืบทอดมาจากลมที่ทายผิด

ทำไมนี่เปลี่ยนเกม: สมมุติ error รวม = 12 แต่พอใส่ ลม reanalysis (oracle) error ลดเหลือ 5 · แปลว่า 7 ใน 12 เป็นเพราะ "พยากรณ์ลมผิด" ไม่ใช่เพราะโมเดลเราห่วย · โมเดลเราจริง ๆ ทำได้ดีระดับ error 5 · นี่คือการแก้ต่างที่ทรงพลังต่อกรรมการ: "ความผิดพลาดส่วนใหญ่สืบทอดจากขีดจำกัดของการพยากรณ์อากาศ (NWP) ซึ่งอยู่นอกเหนืองานเรา — โมเดลเราเองเกือบถึงเพดาน"
โยง NeuralGCM/3.4: error ที่สืบทอดจากลมจะ โตขึ้นตามระยะพยากรณ์ (ลมยิ่งทายไกลยิ่งผิด) — สอดคล้องกับเส้น skill-vs-leadtime (5.1) · oracle met ช่วยแยกว่าการตกของ skill เป็นเพราะ "ลมหรือโมเดล" ตามแต่ละ lead time

6. แตก error 4 แบบ — มองหลายมุม

แบบแตกตามเผยอะไร
① by processemission/met/chem/... (oracle ข้างบน)คอขวดอยู่ขั้นไหนของลูกโซ่
② by spaceแผนที่ error รายพื้นที่แย่ตรงไหน — เมือง? แอ่ง? ต้นทางข้ามแดน?
③ by time / leaderror vs ชั่วโมงข้างหน้าการสะสม error ของ rollout (3.4)
④ by regimeวันวิกฤต vs วันปกติพลาดที่ "หาง" หรือไม่ (โยง 5.1)
ตัวอย่างที่ทรงคุณค่า (by space + by regime ร่วมกัน): ทำแผนที่ error เฉพาะวันวิกฤต → อาจพบว่าโมเดลแม่นทั่วประเทศแต่ "แย่เป็นพิเศษในแอ่งภาคเหนือช่วงพีค" → ชี้เป้าตรงว่าต้องปรับ physics การกักในแอ่ง (DEM, inversion) · การแตก error หลายมุมพร้อมกันคือที่มาของ insight ที่ลึกที่สุด

7. Error Budget — งบประมาณความผิดพลาด (ผลพาดหัวระดับวิจัย) ⭐

รวมผล oracle ทั้งหมดเป็น "งบ error" ก้อนเดียว — แสดงว่า error รวมแบ่งเป็นสัดส่วนของแต่ละสาเหตุ:

ทำไมนี่คือสิ่งที่ยกงานเป็น A*: งานนักศึกษาทั่วไปจบที่ "RMSE 12, ดีกว่า baseline" · งานระดับวิจัยจะพูดต่อได้ว่า "ในจำนวนนี้ ~3 เป็นของโมเดลเรา ที่เหลือเป็นขีดจำกัดของ input — งานอนาคตควรลงทุนที่ emission ก่อน" · นี่คือการวิเคราะห์ที่ทำให้ reviewer เชื่อว่าเราเข้าใจระบบจริง ไม่ใช่แค่จูนตัวเลข · และมันคือ ฝาแฝดเชิงประจักษ์ของ "attribution error bound" (ทฤษฎี 6.1)

8. งบ A100 + กับดักที่ต้องระวัง

งบ — oracle ถูกกว่า ablation

oracle ที่เปลี่ยนแค่ขา input (เช่น oracle met/emission/boundary) ไม่ต้องเทรนใหม่ → รัน inference ซ้ำเท่านั้น = ถูกมาก · ทำได้หลายตัวในงบน้อย · oracle ที่ต้องเทรนใหม่ (เช่น oracle ที่เปลี่ยนโครงสร้าง) จัดทีหลัง

กับดัก (ต้องเขียน caveat):
  • "คำตอบจริง" ก็ไม่เป๊ะ: reanalysis ไม่ใช่ความจริงสมบูรณ์ · oracle จึงให้ "เพดานโดยประมาณ" ไม่ใช่ค่าสัมบูรณ์
  • ความไม่สอดคล้อง: ใส่ลมจริงแต่ emission ยังเป็นค่าประมาณ → สร้างสภาพที่ไม่เคยเกิด อาจให้ผลแปลก
  • ไม่ใช่บวกตรง ๆ: error จากแต่ละสาเหตุ ไม่บวกเป็นเส้นตรง (มี interaction) → error budget เป็น "เชิงชี้นำ" ไม่ใช่บัญชีปิดยอดเป๊ะ · ต้องระบุข้อนี้

9. Cheat sheet + แบบฝึกหัด

Cheat sheet

  • RMSE เดียว = อาการ · decomposition = วินิจฉัยสาเหตุ
  • โมเดล = ลูกโซ่ (emission→ลม→chem→depo) · error เข้าทุกข้อ + สะสม
  • Oracle = แทนชิ้นด้วยคำตอบจริง → error ลดเท่าไร = ชิ้นนั้นเสียเท่านั้น
  • ablation=พื้น · oracle=เพดาน · ช่องว่าง = ศักยภาพที่เหลือ
  • oracle met แยก "ลมพยากรณ์ผิด" ออกจาก "โมเดลเราผิด" — แก้ต่างทรงพลัง
  • แตก error 4 แบบ: process / space / time / regime
  • error budget = ผลพาดหัวระดับวิจัย (ชี้ว่าควรลงแรงตรงไหน)
  • oracle ขา input ไม่ต้องเทรนใหม่ = ถูก · "คำตอบจริง" ก็ไม่เป๊ะ (caveat)

❓ แบบฝึกหัด

  1. oracle substitution ต่างจาก ablation (5.4) ยังไง? วัดอะไรคนละด้าน?
  2. oracle met ทำให้ error ลดจาก 12→5 — ตีความว่าอย่างไร และใช้แก้ต่างกรรมการยังไง?
  3. ทำไม oracle หลายตัวถึงถูกกว่า ablation ในแง่งบ A100?
  4. error budget เป็น "บัญชีปิดยอดเป๊ะ" ได้ไหม เพราะอะไร?
  5. ถ้า oracle emission แทบไม่ลด error ควรสรุปอะไร?
เฉลย
  1. ablation=เอาชิ้นออก→error พุ่ง→"พื้น" · oracle=ทำชิ้นเป๊ะ→error ลด→"เพดาน" · ช่องว่างระหว่างกัน=ศักยภาพชิ้นนั้น
  2. 7 ใน 12 มาจากลมพยากรณ์ผิด ไม่ใช่โมเดลเรา · แก้ต่าง: "error ส่วนใหญ่สืบทอดจากขีดจำกัด NWP นอกงานเรา โมเดลเราเองเกือบถึงเพดาน"
  3. oracle ที่เปลี่ยนแค่ input ไม่ต้องเทรนใหม่ — แค่ inference ซ้ำ · ablation ต้องเทรนใหม่ทุก variant
  4. ไม่ได้ — error แต่ละสาเหตุไม่บวกเป็นเส้นตรง (interaction) + "คำตอบจริง" เองก็ไม่เป๊ะ → เป็นเชิงชี้นำ
  5. emission ไม่ใช่คอขวด → อย่าเสียเวลาปรับ FINN/emission ไปลงแรงที่สาเหตุอื่นแทน
