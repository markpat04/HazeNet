1. Loss Landscape — "ภูมิประเทศของความผิด"

นึกภาพว่าพารามิเตอร์ทุกตัวของโมเดลคือ "พิกัด" และ loss คือ "ความสูง" ของพื้นที่ตรงนั้น → เกิดเป็น ภูมิประเทศ (landscape) ที่มีหุบเขา เป้าหมายเราคือ เดินไปจุดต่ำสุด (loss น้อยสุด = โมเดลแม่นสุด)

ของจริงมีพารามิเตอร์เป็นล้านมิติ วาดไม่ได้ — แต่หลักการเดียวกัน: gradient บอกทิศ "ลง" ที่ชันสุด ณ จุดที่ยืนอยู่

2. กฎอัปเดต — หัวใจของ gradient descent ★

ทุกก้าวของการเรียนรู้คือสมการเดียวนี้ (ทำซ้ำเรื่อย ๆ):

θใหม่ = θเก่า − η · ∇θ L θ = พารามิเตอร์ · η (eta) = learning rate (ขนาดก้าว) · ∇L = gradient (ทิศชันขึ้น) · เครื่องหมายลบ = เดินสวนทาง (ลง)

อ่านเป็นภาษาคน: "ยืนอยู่ตรงไหน ดูว่าทางขึ้นชันสุดไปทางไหน (∇L) แล้วเดินถอยหลังจากทางนั้นเป็นก้าวขนาด η" — ทำซ้ำจนถึงก้นหุบเขา

เชื่อมกับ 0.2: ใน Hello World ส่วน 4 ของบท 0.2 คุณเขียน x -= lr * x.grad ด้วยมือ — นั่นคือสมการนี้เป๊ะ · optimizer.step() ก็ทำสมการนี้ให้ทุกพารามิเตอร์พร้อมกัน

3. Learning Rate (η) — ตัวที่พังบ่อยสุด ★

η คือ "ขนาดก้าว" และเป็น hyperparameter สำคัญที่สุด ที่คุณต้องจูน — ตั้งผิดเทรนพังทันที

อาการสาเหตุแก้
loss ลงช้ามาก/เกือบนิ่งη เล็กไปเพิ่ม η (×3 หรือ ×10)
loss กระโดดขึ้น-ลง / กลายเป็น NaNη ใหญ่ไป (exploding)ลด η (÷3 หรือ ÷10)
loss ลงดีแล้วค้างกลางคันη คงที่ใหญ่ตอนใกล้ก้นใช้ schedule ลด η (ส่วนล่าง)
วิธีหา η ในทางปฏิบัติ: ลองเป็น "เลขยกกำลังสิบ" — 1e-2, 1e-3, 1e-4 · กับ Adam เริ่มที่ 1e-3 มักใช้ได้เลย · เทรนสั้น ๆ ดู loss curve 2-3 นาทีก่อน ว่าลงไหม อย่าปล่อยยาวแล้วค่อยรู้ว่า η พัง = เสียงบ GPU ฟรี (สำคัญมากกับ A100 จำกัด 15 ชม.)

4. Batch / SGD / Epoch — คำที่ต้องแยกให้ออก

เรามีข้อมูลเยอะมาก (เป็นล้านจุด) คำนวณ gradient จากทั้งหมดทุกก้าวจะช้า/แรมไม่พอ → แบ่งเป็นก้อนเล็ก

คำความหมาย
Batch (mini-batch)ข้อมูลก้อนเล็กที่ใช้คำนวณ gradient 1 ก้าว (เช่น 16 ตัวอย่าง)
Iteration / stepการอัปเดต 1 ครั้ง (= 1 batch)
Epochเห็นข้อมูล "ครบทั้งชุด" 1 รอบ (= หลาย iteration)
SGD (Stochastic GD)gradient descent ที่ใช้ "ทีละ batch" แทนทั้งชุด — เร็วและทำงานได้ดีในทางปฏิบัติ

ตัวอย่าง: ข้อมูล 1,600 ตัวอย่าง · batch size 16 → 1 epoch = 1600/16 = 100 iteration · เทรน 50 epoch = 5,000 ก้าวอัปเดต

เชื่อมกับงานเรา + ข้อจำกัด GPU: batch size ใหญ่ = gradient นิ่งกว่าแต่กินแรม GPU มาก · บน A100 ที่แรมจำกัด เราจะเลือก batch ใหญ่สุดที่ "ไม่ OOM" · ข้อมูล HazeNet เป็นแผนที่ (channel, 85, 105) ก้อนใหญ่ → batch อาจเล็ก (4-16) ต้องลองหา

5. Momentum & Adam — gradient descent รุ่นฉลาด

GD ธรรมดามีปัญหา: ในหุบเขาแคบยาวมันจะ "ส่ายไปมา" ช้า · มีลูกเล่นช่วย:

Momentum — "ลูกบอลกลิ้งสะสมแรง"

แทนที่จะเดินตาม gradient ปัจจุบันล้วน ๆ ให้สะสม "ความเร็ว" จากก้าวก่อน ๆ ด้วย → ทะลุจุดสะดุดเล็ก ๆ และเร่งในทิศที่ไปทางเดิมซ้ำ ๆ

v ← β·v + ∇L   ;   θ ← θ − η·v v = ความเร็วสะสม · β ≈ 0.9 = ความเฉื่อย

Adam — รุ่นมาตรฐานที่ใช้จริง

Adam = Momentum + ปรับ learning rate ให้แต่ละพารามิเตอร์อัตโนมัติ (พารามิเตอร์ที่ gradient ใหญ่ก้าวสั้นลง, ที่เล็กก้าวยาวขึ้น) → ทนทาน จูนง่าย

Optimizerลักษณะใช้เมื่อ
SGDพื้นฐาน ต้องจูน η ดีงานที่อยากได้ generalization สูงสุด (เช่น vision)
SGD + momentumเร็วขึ้น ทะลุสะดุดทางเลือกคลาสสิก
Adamปรับ lr อัตโนมัติ จูนง่ายค่าเริ่มต้นที่แนะนำ (รวมถึง HazeNet)
AdamWAdam + weight decay ที่ถูกต้องโมเดลใหญ่ (transformer/AFNO) — น่าจะใช้ตัวนี้
เชื่อมกับงานเรา: AFNO/transformer มักเทรนด้วย AdamW + learning rate schedule (warmup แล้ว decay) — เราจะใช้สูตรนี้ · เริ่ม AdamW(lr=1e-3, weight_decay=1e-4) เป็นจุดตั้งต้น แล้วจูนจาก loss curve

6. เลือก Loss Function — MSE / MAE / Huber

loss คือ "นิยามของความผิด" ที่เราสั่งให้โมเดลลด — เลือกต่างกัน โมเดลให้ความสำคัญต่างกัน

Lossสูตรนิสัยข้อเสีย
MSE (L2)mean((pred−true)²)ลงโทษ error ใหญ่หนักมาก → เน้นจับค่าพีค · gradient เรียบไวต่อ outlier/noise มาก
MAE (L1)mean(|pred−true|)ทนทานต่อ outlier · ทุก error น้ำหนักเท่ากันgradient ไม่เรียบที่ 0
Huber (SmoothL1)MSE ใกล้ 0, L1 ไกลดีที่สุดของทั้งคู่ — เรียบ + ทนทานมี hyperparam δ ต้องตั้ง
★ การตัดสินใจจริงสำหรับ HazeNet (สำคัญ):
  • ฝุ่น PM2.5 มี ค่าพีคสุดขั้ว ในวันวิกฤต — ซึ่งคือสิ่งที่เราต้องจับให้ได้ที่สุด (เชื่อมกับ metric CSI/POD วันวิกฤต หัวข้อ 5.1)
  • MSE จะ "เพ่ง" ค่าใหญ่ → ดีต่อการจับพีค แต่ ก็ไวต่อ noise ของสถานีที่ข้อมูลกระจาย
  • ทางที่ดี: ลอง Huber (ทนทาน+เน้นพีคได้) หรือเทรนใน log-space (log PM2.5) เพื่อไม่ให้ค่าพีคกลบค่าปกติ · หรือ weighted loss ถ่วงน้ำหนักวันความเข้มข้นสูง
  • นี่คือ "การออกแบบ loss" ที่จะเป็นหนึ่งใน ablation ของเรา (E ในแผน) — ไม่ใช่เลือกมั่ว ต้องทดลองเทียบ

7. อ่าน Loss Curve — ทักษะที่ใช้ทุกวัน ★

กราฟ loss ต่อ epoch บอกสุขภาพการเทรนได้แทบทุกอย่าง เราพล็อต 2 เส้น: train loss + validation loss

คุณเห็นแปลว่าทำอะไร
train↓ + val↓ แล้วแบนชิดกันสุขภาพดีปล่อยจนนิ่ง / หยุด
train↓ แต่ val↑overfitting (จำข้อมูลเทรน)เพิ่มข้อมูล/regularization/dropout, early stopping
ทั้งคู่นิ่งสูงunderfitting / lr เล็ก / โมเดลอ่อนเพิ่ม lr, เพิ่ม capacity, เทรนนานขึ้น
loss เด้ง/พุ่ง/NaNlr ใหญ่ไป / gradient ระเบิดลด lr, gradient clipping
เชื่อมกับงานเรา: "val loss" ของเราต้องวัดบน ปี/สถานีที่กันไว้ (split กันรั่ว จากหัวข้อ 5.2) ไม่งั้นกราฟสวยหลอก ๆ · ช่องว่าง train-val ที่กว้าง = สัญญาณว่าโมเดล generalize ไม่ดี ซึ่งเป็นจุดที่ physics prior (ฟิสิกส์) ช่วยลดได้

8. หลุมพราง: local minima & plateau (อย่ากลัวเกินไป)

  • Local minimum = หุบเขาที่ไม่ใช่ลึกสุด — ในทางทฤษฎีน่ากลัว แต่ในมิติสูง (พารามิเตอร์ล้านตัว) พบน้อยกว่าที่คิด และมักให้ผลใกล้เคียงพอใช้
  • Saddle point (จุดอานม้า) = ราบในบางทิศ ชันในบางทิศ → momentum/Adam ช่วยหลุดได้
  • Plateau = ที่ราบยาว loss ลงช้ามาก → learning rate schedule หรือ warmup ช่วย
อย่าหมกมุ่นกับ "global minimum": เป้าหมายจริงไม่ใช่ loss ต่ำสุดบนข้อมูลเทรน แต่คือ generalize ดีบนข้อมูลใหม่ (val/test) — โมเดลที่ loss เทรนต่ำสุดอาจ overfit · ดู val เป็นหลักเสมอ

Learning rate schedule (โบนัส)

ในทางปฏิบัติเรามัก "ลด η ระหว่างทาง": เริ่มก้าวใหญ่ลงเร็ว แล้วก้าวเล็กลงตอนใกล้ก้นเพื่อไม่กระเด้ง · รูปแบบนิยม: warmup (ค่อย ๆ เพิ่ม η ช่วงแรกกัน NaN) แล้ว cosine decay (ค่อย ๆ ลด) — มาตรฐานของการเทรน transformer/AFNO

9. เชื่อมกับ HazeNet — loss ที่ "รู้ฟิสิกส์"

loss ของเราไม่ใช่แค่ MSE เปล่า ๆ แต่ออกแบบให้สะท้อนสิ่งที่เราแคร์:

L = Ldata(Cpred, Ctrue)  +  λ1·Lrollout  +  λ2·Lpeak L_data = Huber/MSE บนฝุ่น · L_rollout = error สะสมหลาย step (จาก 3.4) · L_peak = ถ่วงน้ำหนักวันวิกฤต
  • ไม่ต้องมี loss บังคับ non-negativity / mass เพราะเราทำเป็นโครงสร้างแล้ว (หัวข้อ 2.4, 3.5) — เป็นจุดแข็งของ design เรา
  • rollout loss สำคัญ: ถ้าเทรน t+1 อย่างเดียว error จะสะสมตอนพยากรณ์ยาว → ต้องเทรน multi-step (เชื่อม curriculum 3.4)
  • การเลือก optimizer/lr/loss ทั้งหมดนี้จะถูกบันทึกเป็น systematic experiment (แผน master) ไม่ใช่จูนมั่ว

10. Checklist — "เทรนไม่ลง" ทำไงดี (เก็บไว้ใช้จริง)

  1. ลด/เพิ่ม learning rate (ลองก่อนเสมอ — สาเหตุอันดับ 1) ลองเป็นเลข ×10/÷10
  2. ลืม zero_grad() ไหม? (จาก 0.2 — gradient สะสมทำเทรนเพี้ยน)
  3. normalize ข้อมูลหรือยัง? input/output ควรสเกลใกล้ ๆ กัน (เช่น mean 0 std 1) ไม่งั้น loss ระเบิด
  4. shape ตรงไหม? broadcasting เงียบ ๆ (จาก 0.1) ทำ loss ผิดโดยไม่ error
  5. loss/ข้อมูลมี NaN ไหม? ตรวจ input · ใส่ gradient clipping
  6. overfit (val↑)? เพิ่มข้อมูล/regularization/early stopping
  7. underfit (นิ่งสูง)? เพิ่ม capacity/เทรนนานขึ้น/เพิ่ม lr
  8. ทดสอบกับข้อมูลจิ๋ว: ลองให้โมเดล "overfit ข้อมูล 1 batch" ให้ได้ก่อน — ถ้าทำไม่ได้แสดงว่ามี bug ในโค้ด ไม่ใช่เรื่อง hyperparameter
