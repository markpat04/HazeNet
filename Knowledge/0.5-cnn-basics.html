ทำไม CNN สำคัญกับเรา: ข้อมูล HazeNet คือ "ภาพหลายชั้น" — แผนที่ฝุ่น + แผนที่ลม + แผนที่อุณหภูมิ ซ้อนกัน · CNN คือเครื่องมือที่ออกแบบมาเพื่อหา pattern เชิงพื้นที่บนภาพแบบนี้โดยเฉพาะ · และที่สำคัญ — "ข้อจำกัด" ของ CNN (มองได้แค่ใกล้ ๆ) คือเหตุผลโดยตรงที่งานเราต้องเพิ่ม AFNO (มองทั่วทั้งภาพ) และ advection (ฟิสิกส์ขนส่งระยะไกล) เข้าใจ CNN = เข้าใจว่าทำไม design เราเป็นแบบนั้น

1. ทำไมไม่ใช้ Fully-Connected (FC) กับภาพ

จาก 0.2 เรารู้จัก nn.Linear (fully-connected) ที่เชื่อมทุก input เข้ากับทุก neuron · ลองใช้กับแผนที่ของเรา:

แผนที่ 85×105 = 8,925 จุด ต่อ 1 ตัวแปร · ถ้าเชื่อมเข้าชั้นซ่อน 1,000 neuron แบบ FC:

8,925 × 1,000 ≈ 9 ล้าน พารามิเตอร์ — แค่ชั้นเดียว ตัวแปรเดียว!

ปัญหา 2 ข้อ:

  • พารามิเตอร์ระเบิด → เปลือง memory + overfit ง่าย (ข้อมูลไม่พอเทรน)
  • ทิ้งโครงสร้างพื้นที่ → FC มองทุกจุดเป็นอิสระ ไม่รู้ว่า "จุดข้าง ๆ กันเกี่ยวข้องกัน" ทั้งที่ฝุ่นที่จุดหนึ่งสัมพันธ์กับเพื่อนบ้านโดยตรง
CNN แก้ทั้ง 2 ข้อ ด้วย 2 ไอเดีย: (1) local — มองทีละหน้าต่างเล็ก ๆ (เพื่อนบ้าน) · (2) weight sharing — ใช้ตัวกรอง (filter) ชุดเดียวกันเลื่อนไปทั้งภาพ → พารามิเตอร์น้อยลงมหาศาล + เคารพโครงสร้างพื้นที่

2. Convolution คืออะไร — หัวใจของ CNN ★

convolution = เอา "ตัวกรอง" (kernel/filter) ก้อนเล็ก ๆ เลื่อนไปทั่วภาพ แต่ละตำแหน่งทำ "คูณตรงช่องแล้วบวกรวม" ได้ออกมา 1 ตัวเลข

แล้ว เลื่อน kernel ไปขวา/ลง ทำซ้ำทุกตำแหน่ง → ได้ output map ใหม่ (เรียก feature map) · input 4×4 กับ kernel 3×3 (ไม่ pad) → output 2×2

output[i,j] = Σ (patch ที่ตำแหน่ง i,j × kernel) เลื่อน kernel ทุกตำแหน่ง คำนวณซ้ำ — "kernel ชุดเดียว ใช้ทั้งภาพ" (weight sharing)
จุดสำคัญ: kernel เล็ก (เช่น 3×3 = 9 ตัวเลข) แต่ใช้กับทั้งภาพ → พารามิเตอร์น้อยมากเทียบ FC · และเพราะมันดู "หน้าต่างรอบ ๆ จุด" มันจึงจับความสัมพันธ์เชิงพื้นที่ได้

3. Filter เรียนรู้อะไร — จากขอบสู่ pattern

ค่าใน kernel ไม่ได้ตั้งเอง — โมเดลเรียนรู้เอง ผ่าน gradient descent (0.3) · filter ต่าง ๆ จะเรียนตรวจจับสิ่งต่างกัน:

  • ชั้นต้น: ตรวจจับของพื้นฐาน — ขอบ, เส้น, การไล่ระดับ (เช่น "ฝุ่นเปลี่ยนเข้มฉับพลันตรงไหน")
  • ชั้นลึก: ประกอบของพื้นฐานเป็น pattern ซับซ้อน — "กลุ่มควันรูปลิ้น", "แนวควันตามหุบเขา"

เรามี หลาย filter ต่อชั้น (เช่น 64 อัน) แต่ละอันจับ pattern คนละแบบ → ได้ feature map 64 ชั้น

4. Hyperparameters: kernel size / stride / padding

พารามิเตอร์คืออะไรผล
kernel sizeขนาดหน้าต่าง (3×3, 5×5)ใหญ่ = เห็นบริบทกว้างขึ้นต่อชั้น แต่พารามิเตอร์มากขึ้น
strideเลื่อนทีละกี่ช่องstride 2 = กระโดดทีละ 2 → output เล็กลงครึ่ง (ย่อภาพ)
paddingเติมขอบ (มักเติม 0)"same" = output ขนาดเท่า input · "valid" = ไม่เติม output เล็กลง
output size = (W + 2·padding − kernel) / stride + 1 เช่น W=4, kernel=3, pad=0, stride=1 → (4−3)/1+1 = 2 ✓ (ตรงกับตัวอย่างส่วน 2)
เชื่อมกับงานเรา: เรามักใช้ padding="same" เพื่อให้แผนที่ output ขนาดเท่า input (กริด 85×105 เข้า → 85×105 ออก) เพราะเราพยากรณ์ฝุ่นทุกจุดบนกริดเดิม

5. Channels & Feature Maps — มิติที่ซ่อนอยู่

จาก 0.1: ภาพจริงไม่ใช่แค่ 2D แต่มี channel · รูปสีมี 3 channel (R,G,B) · convolution ทำงานบนทุก channel พร้อมกัน

Conv2d: (batch, C_in, H, W) → (batch, C_out, H', W') C_in = channel เข้า · C_out = จำนวน filter = channel ออก · (รูปแบบ tensor จาก 0.1)
★ Channels ใน HazeNet คืออะไร: input channel ของเราคือ ตัวแปรต่าง ๆ ที่ซ้อนเป็นแผนที่: 
x.shape = (batch, 6, 85, 105)
          #  6 channel = [PM2.5, wind_u, wind_v, temp, humidity, AOD]
convolution จะมองทุกตัวแปรพร้อมกันที่แต่ละจุด → เรียนรู้ว่า "ฝุ่น+ลมแรง+อุณหภูมิแบบนี้ → ชั่วโมงหน้าฝุ่นจะเป็นยังไง" · output channel คือ feature ที่โมเดลสกัดได้

6. Receptive Field — "นิวรอนหนึ่งตัวมองได้ไกลแค่ไหน" ★★ สำคัญสุดของบท

receptive field (RF) = พื้นที่บน input ที่มีผลต่อ output 1 จุด · kernel 3×3 ชั้นเดียว → RF = 3×3 (เห็นแค่เพื่อนบ้านติดกัน) · แต่พอซ้อนหลายชั้น RF จะโตขึ้นเรื่อย ๆ

กฎคร่าว ๆ: ซ้อน conv 3×3 จำนวน L ชั้น → RF ≈ 2L+1 · จะเห็นไกล 50 ช่อง ต้องซ้อน ~25 ชั้น (ถ้าไม่ใช้ pooling) = ลึกและแพง

★★ นี่คือจุดเชื่อมที่สำคัญที่สุดของทั้งบท:
ควันข้ามแดนเดินทาง หลายร้อยกิโลเมตร — บนกริด 11 กม. = หลายสิบช่อง · ฝุ่นที่เชียงใหม่วันนี้ขึ้นกับไฟที่เมียนมาไกลออกไปมาก · CNN ธรรมดาต้องซ้อนชั้นมหาศาลเพื่อ "มองไกล" พอ = แพงและเทรนยาก

นี่คือเหตุผลโดยตรงที่ design เราเป็นแบบนั้น:
  • AFNO (หัวข้อ 3.2) ทำงานในโดเมนความถี่ → มี receptive field ทั่วทั้งภาพในชั้นเดียว (global) ไม่ต้องซ้อนเยอะ → แก้ปัญหา RF ของ CNN
  • Advection (หัวข้อ 2.2) = ฟิสิกส์ขนส่งฝุ่นระยะไกลตามลมโดยตรง → ไม่ต้องให้เน็ตเรียนการเคลื่อนที่ไกล ๆ เอง
สรุป: ข้อจำกัด receptive field ของ CNN = แรงผลักดันให้เราผสม Fourier + ฟิสิกส์ · ตอนนี้คุณเข้าใจ "ทำไม" ของสถาปัตยกรรมเราแล้ว

7. Pooling — ย่อภาพเพื่อมองกว้างขึ้น (และประหยัด)

pooling = ยุบหน้าต่าง (เช่น 2×2) ให้เหลือค่าเดียว → ภาพเล็กลงครึ่ง · ช่วยให้ RF โตเร็วขึ้นและลดการคำนวณ

Max pooling เก็บค่าเด่นสุดในหน้าต่าง · Average pooling เฉลี่ย · ทั้งคู่ทำให้ภาพเล็กลง → ชั้นถัด ๆ ไป "เห็นกว้าง" ขึ้นต่อ kernel เดิม

8. CNN Stack ทั่วไป + โค้ด PyTorch

CNN จริงคือการซ้อน block conv → activation → (pool) หลาย ๆ ชั้น

import torch.nn as nn

class SmallCNN(nn.Module):
    def __init__(self):
        super().__init__()
        # Conv2d(in_channels, out_channels, kernel_size, padding)
        self.conv1 = nn.Conv2d(6, 32, 3, padding=1)   # 6 ตัวแปร → 32 feature
        self.conv2 = nn.Conv2d(32, 32, 3, padding=1)
        self.out   = nn.Conv2d(32, 1, 3, padding=1)    # → แผนที่ฝุ่น 1 channel

    def forward(self, x):                 # x: (batch, 6, 85, 105)
        x = torch.relu(self.conv1(x))      # (batch, 32, 85, 105)
        x = torch.relu(self.conv2(x))      # padding=1 → ขนาดคงเดิม
        x = self.out(x)                    # (batch, 1, 85, 105)
        return x

สังเกต padding=1 กับ kernel 3 → output ขนาดเท่าเดิม (same) · เทรนด้วยวงจร 5 ขั้นจาก 0.2 ได้เลย

เชื่อมกับงานเรา: นี่คือ "baseline CNN" ที่เราจะเทียบในการทดลอง (E1 ในแผน master) · และ U-Net (หัวข้อ 3.3) ก็สร้างจาก conv block แบบนี้ แค่เพิ่มโครง encoder-decoder + skip connection

9. ★ สรุปเชื่อม HazeNet — ทำไม CNN อย่างเดียวไม่พอ

คุณสมบัติ CNNดีต่อเรายังไงข้อจำกัด → เราแก้ด้วย
local + weight sharingจับ pattern เชิงพื้นที่ พารามิเตอร์น้อย—
translation-equivariant
(เลื่อนภาพ output เลื่อนตาม)		เหมาะกับฟิสิกส์ที่เหมือนกันทุกที่—
receptive field จำกัด—AFNO (global RF) + advection (ขนส่งไกล)
มองทุกอย่างเป็นภาพรวมหลายตัวแปรเป็น channel ได้เพิ่ม physics เพื่อไม่ต้องเรียนการเคลื่อนที่จากศูนย์
ภาพใหญ่ที่ควรจำออกจากบทนี้: CNN คือ "เครื่องมือพื้นฐานเห็นภาพ" ที่ดีและเป็นฐานของ U-Net · แต่ปัญหาของเรา (ควันเดินทางไกล) เกินกำลัง CNN ล้วน → เราจึงผสม Fourier (AFNO) + ฟิสิกส์ (advection) เข้าไป · การเข้าใจ CNN ทำให้คุณเข้าใจว่าแต่ละชิ้นใน design เราแก้ปัญหาอะไร ไม่ใช่ใส่มั่ว

แบบฝึกหัด

  1. input 32×32 · kernel 5×5 · padding 0 · stride 1 → output ขนาดเท่าไร?
  2. ทำไม CNN ใช้พารามิเตอร์น้อยกว่า FC มาก?
  3. nn.Conv2d(6, 64, 3) — ตัวเลข 6 กับ 64 หมายถึงอะไร?
  4. "receptive field" คืออะไร และทำไมมันเป็นปัญหากับควันข้ามแดน?
  5. เราแก้ปัญหา receptive field ของ CNN ด้วย 2 อย่างอะไรใน HazeNet?
▶ เฉลย
  1. (32−5)/1+1 = 28 → output 28×28
  2. weight sharing (kernel ชุดเดียวใช้ทั้งภาพ) + local (มองหน้าต่างเล็ก ไม่เชื่อมทุกจุด)
  3. 6 = input channels (ตัวแปรเข้า) · 64 = output channels (จำนวน filter = feature ที่สกัด)
  4. พื้นที่บน input ที่มีผลต่อ output 1 จุด · ควันข้ามแดนเดินไกลหลายร้อย กม. CNN ต้องซ้อนชั้นมหาศาลถึงจะ "เห็น" ต้นทางได้ = แพง
  5. AFNO (receptive field ทั่วภาพในชั้นเดียว) + advection (ฟิสิกส์ขนส่งระยะไกลโดยตรง)
