1. ภาพใหญ่ — FastCTM เกิดมาแก้ปัญหาอะไร

1.1 ปัญหา: แบบจำลองเคมีดั้งเดิม (CMAQ) แม่นแต่ "ช้าโหด"

CMAQ (Community Multiscale Air Quality) คือแบบจำลองคุณภาพอากาศมาตรฐานที่นักวิทยาศาสตร์ใช้กันทั่วโลก — มันแก้สมการฟิสิกส์-เคมีของบรรยากาศด้วยวิธีเชิงตัวเลข (numerical methods) ทีละจุดบนกริด. ปัญหาคือ:

  • ต้องใช้ ซูเปอร์คอมพิวเตอร์ (HPC) — แพง เข้าถึงยาก
  • รัน 1 รอบใช้เวลา หลายชั่วโมง — ทำ "ลองหลายสถานการณ์" (what-if) แทบไม่ไหว
  • ทำ source inversion / sensitivity ยิ่งยากมหาศาล (ต้องเขียน adjoint code แยก ใช้เวลาพัฒนาเป็นปี ๆ)

1.2 ทางออกเดิม ๆ และจุดอ่อนของมัน

1.3 ไอเดียของ FastCTM: "Principle-Informed"

FastCTM อยู่ตรงกลาง — เร็วแบบ NN แต่มีฟิสิกส์แบบ CMAQ. วิธีคือ ไม่ปล่อยให้ NN เดามั่ว แต่ "ออกแบบโครงสร้าง NN ให้เดินตามสมการฟิสิกส์-เคมีจริง". แทนที่จะมี NN ก้อนเดียวที่ทำทุกอย่างปนกัน เขาแยกเป็น 5 โมดูล ตาม 5 กระบวนการฟิสิกส์:

① Emissionปล่อยฝุ่นใหม่
+
② Transportลมพัดพา
+
③ Diffusionกระจายตัว
+
④ Chemistryปฏิกิริยาเคมี
+
⑤ Depositionตกสู่พื้น
ทำไมการ "แยกโมดูล" ถึงทรงพลัง: เพราะแต่ละโมดูลมีหน้าที่ฟิสิกส์ชัดเจน → (1) ตีความได้ เช่น "วันนี้ฝุ่นขึ้นเพราะ transport เป็นหลัก" (2) หา error ได้ ว่าโมดูลไหนเพี้ยน (3) ใส่ความรู้ฟิสิกส์ลงไปได้ เช่น บังคับอนุรักษ์มวลในโมดูล transport. นี่คือความหมายของ "principle-informed".

2. ทฤษฎีรากฐาน — สมการ Governing ของบรรยากาศ

ทุกอย่างใน FastCTM (และ HazeNet) เริ่มจากสมการเดียวนี้ — "สมการการพา-การแพร่-ปฏิกิริยา" (Advection–Diffusion–Reaction equation). มันบอกว่า "ความเข้มข้นฝุ่นในกล่องอากาศหนึ่ง เปลี่ยนไปตามเวลาอย่างไร":

∂Ci/∂t = −∇·uCi + ∇K∇Ci + Ri + Ei + Di

แปลเป็นภาษาคน: "อัตราการเปลี่ยนแปลงความเข้มข้นฝุ่น (∂C/∂t) = ผลรวมของ 5 กระบวนการ". มาดูทีละเทอม (สีตรงกับ 5 โมดูล):

เทอมชื่อความหมายแบบเห็นภาพ
EiEmission (ปล่อย)แหล่งเติมฝุ่นใหม่เข้ามา — ไฟไหม้ป่า, ไอเสียรถ, โรงงาน. "ก๊อกน้ำเปิด"
−∇·uCiAdvection / Transport (พา)ลม (u) หอบฝุ่นจากที่หนึ่งไปอีกที่ — เหมือนใบไม้ลอยตามกระแสน้ำ. นี่คือหัวใจ "หมอกควันข้ามแดน"
∇K∇CiDiffusion (แพร่)ความปั่นป่วนของอากาศ (turbulence) ทำให้ฝุ่นกระจายจากที่เข้มไปที่จาง — เหมือนหยดสีในน้ำค่อย ๆ จางออก. K = ค่าความแรงการแพร่
RiReaction (ปฏิกิริยา)ฝุ่นเกิดใหม่หรือสลายจากปฏิกิริยาเคมี — เช่น SO₂ + ความชื้น → ซัลเฟต (ฝุ่นทุติยภูมิ)
DiDeposition (ตกสะสม)ฝุ่นออกจากอากาศ — ตกพื้น, ติดใบไม้, ฝนชะลงมา. "ก๊อกน้ำระบายออก"

สัญลักษณ์: Ci = ความเข้มข้นสาร i, u = ความเร็วลม, K = eddy diffusivity (ค่าการแพร่จากความปั่นป่วน), ∇ = ตัวดำเนินการเชิงพื้นที่ (gradient/divergence).

🎯 เชื่อมกับ HazeNet
สมการนี้คือ "รัฐธรรมนูญ" ของทั้ง FastCTM และ HazeNet. สิ่งที่เราเรียนในบท 2.x (superposition) และ 3.5 (conditionally-linear operator) คือการวิเคราะห์คุณสมบัติทางคณิตศาสตร์ของสมการนี้. จุดสำคัญที่ต้องจำ: 4 ใน 5 เทอมเป็น "เชิงเส้นใน C" (E ไม่ขึ้นกับ C เลย; transport, diffusion, deposition แปรผันตรงกับ C) — มีแค่ เทอมเคมี R เท่านั้นที่อาจไม่เชิงเส้น. จำประโยคนี้ไว้ เดี๋ยวมันจะกลับมาสำคัญมากตอนทำ tagged-tracer (ดู §10).

3. ทฤษฎี Operator Splitting — ทำไม "แยกแล้วบวกกัน" ถึงทำได้

คำถามที่ควรสงสัย: "ในเมื่อ 5 กระบวนการเกิดพร้อมกันในธรรมชาติ ทำไมถึงแยกคำนวณทีละอันแล้วเอามาบวกกันได้?" คำตอบคือเทคนิคคลาสสิกชื่อ Operator Splitting (CMAQ เองก็ใช้วิธีนี้ FastCTM แค่เดินตาม):

หลักการ: ถ้าช่วงเวลา Δt สั้นพอ เราประมาณได้ว่า "ทำกระบวนการทีละอันต่อกัน" ≈ "ทำพร้อมกัน". เหมือนทำกับข้าว:
→ ในความจริงทุกอย่างเกิดในกระทะพร้อมกัน แต่ถ้าเราหั่น→ผัด→ปรุง ทีละสเต็ปเร็ว ๆ ผลลัพธ์แทบเหมือนกัน.

ทางคณิตศาสตร์: แทนที่จะแก้สมการรวมทีเดียว เราอัปเดตทีละเทอม: 
C → (+ Emission) → (+ Transport) → (+ Diffusion) → (+ Chemistry) → (+ Deposition) → C ใหม่
ในโค้ด FastCTM ทำง่ายกว่านั้นอีก คือคำนวณ "การเปลี่ยนแปลง (ΔC)" ของแต่ละกระบวนการแล้วบวกรวม:
ΔC = E + Transport + Diffusion + Chemistry + Deposition  ,  C(t+1) = C(t) + ΔC
เหตุผลที่ "ต้อง" แยก ไม่ใช่แค่ "ทำได้":
  • ใส่ฟิสิกส์เฉพาะทางได้ — โมดูล transport ใช้สูตร upwind ที่อนุรักษ์มวล; โมดูลเคมีใช้สูตร kinetics. ถ้าเป็น NN ก้อนเดียวทำไม่ได้
  • ตีความได้ — รู้ว่า ΔC แต่ละก้อนมาจากกระบวนการไหน → process analysis
  • ดีบักได้ — โมดูลไหนเพี้ยนก็แก้เฉพาะตัว
🎯 เชื่อมกับ HazeNet
Operator splitting คือ "เหตุผลเชิงทฤษฎี" ที่ทำให้ HazeNet ออกแบบเป็นโมดูลได้ และ เป็นเหตุผลที่ superposition (บท 2.5) ทำงาน: เพราะแต่ละ operator เชิงเส้น (ยกเว้นเคมี) การแยก tracer แล้วรันผ่านโมดูลเดียวกันแล้วบวกกลับ จึงได้ผลเท่าเดิม. FastCTM ยืนยันว่าโครงนี้ "ใช้ได้จริงใน NN" ไม่ใช่แค่ทฤษฎี. (หมายเหตุซื่อสัตย์: เปเปอร์ไม่ได้เอ่ยคำว่า "operator splitting" ตรง ๆ แต่โครงสร้างเขาคือสิ่งนี้เป๊ะ)

4. สถาปัตยกรรม 5 โมดูล — ลงลึกทีละตัว

ตอนนี้เจาะแต่ละโมดูล: ฟิสิกส์ → สูตรคณิต → NN ที่ใช้ → input/output → ทำไมออกแบบงี้. สังเกตว่าบางโมดูลไม่มี NN เลย (ใช้สูตรฟิสิกส์ตรง ๆ) — นั่นคือความฉลาดของ principle-informed: ใช้ NN เฉพาะส่วนที่ "ไม่รู้สูตรแน่ชัด" เท่านั้น.

1
Emission — การปล่อยฝุ่นใหม่ สูตรล้วน · ไม่มี NN

ฟิสิกส์: สมมติว่าฝุ่นที่ปล่อยออกมา คลุกเคล้าทันทีทั่วชั้น PBL (Planetary Boundary Layer = ชั้นอากาศติดพื้นที่อากาศปั่นป่วนผสมกันดี). ความเข้มข้นที่เพิ่ม = มวลที่ปล่อย ÷ ปริมาตรกล่องอากาศ.

E = 1000 × PE / (PBLH × dx × dy)
Input: PE = อัตราการปล่อย (g/s), PBLH = ความสูงชั้น PBL, dx·dy = ขนาดเซลล์ (12×12 กม.)  |  Output: ΔC เพิ่ม (µg/m³)

ทำไมไม่ใช้ NN? เพราะสูตรนี้คือฟิสิกส์ตรงไปตรงมา (มวล÷ปริมาตร) — รู้แน่ชัดอยู่แล้ว ไม่ต้องให้ NN เดา. ใช้ NN เปลืองเปล่า ๆ และเสี่ยงผิด.

ทฤษฎี PBL: ตอนกลางวันแดดร้อน อากาศลอยตัว → PBL สูง (1-2 กม.) → ฝุ่นเจือจางในปริมาตรใหญ่. ตอนกลางคืน PBL ต่ำ (ไม่กี่ร้อยเมตร) → ฝุ่นอัดในปริมาตรเล็ก → เข้มข้นพุ่ง. นี่คือเหตุผลที่ฝุ่นมักหนักตอนเช้ามืด.

🎯 ใช้กับเรา
HazeNet ทำเหมือนกันได้เลย — โมดูล emission ของเราคือ "ฉีด" emission จากไฟ (FINNv2.5/FIRMS) เข้าระบบด้วยสูตร PBL นี้. และตรงนี้แหละที่เราจะ "ติดป้าย tracer" — ฝุ่นจากเมียนมา vs จากไทย แยกกันตั้งแต่จุดปล่อย (บท 3.6).
2
Transport / Advection — ลมพัดพา upwind CNN · ไม่มีพารามิเตอร์เรียนรู้

ฟิสิกส์: ลมหอบฝุ่นไปตามทิศ. เขียนเป็น divergence ของ (ลม × ความเข้มข้น) แยกเป็นแกน x, y:

∇·uC = ∂(CU)/∂x + ∂(CV)/∂y

คณิต — วิธี Finite Difference + Upwind Scheme: คอมพิวเตอร์คำนวณอนุพันธ์ (∂/∂x) ไม่ได้ตรง ๆ ต้องประมาณด้วย "ผลต่างของเซลล์ข้างเคียง" (finite difference). แต่มีกลเม็ด — ต้องเลือกข้างให้ถูก เรียก upwind scheme: "เอาค่าจากเซลล์ต้นลม (ที่ลมพัดมา)".

Input: C, ลม U (ตะวันออก-ตก) และ V (เหนือ-ใต้) ที่ 10 ม.  |  Output: ΔC จากการพัดพา

ทำไม upwind? ถ้าใช้ผลต่างแบบสมมาตร (central difference) การจำลองจะ "สั่น" และระเบิด (numerical instability). upwind เสถียรกว่าและ อนุรักษ์มวล — ฝุ่นที่ออกจากเซลล์หนึ่งต้องไปโผล่เซลล์ข้าง ๆ พอดี ไม่หายไปเฉย ๆ.

จุดสำคัญมาก (เปเปอร์พูดเอง): "โมดูล transport อนุรักษ์มวลในตัวเอง แต่ FastCTM ทั้งระบบไม่ได้อนุรักษ์มวล". จำประโยคนี้ไว้ให้ดี — มันคือจุดที่ HazeNet ต้องทำให้ดีกว่า (ดู §10).
🎯 ใช้กับเรา
ลอกได้เลย 100%. upwind kernel เป็น convolution ที่ "ฟิกซ์ค่าไว้" (ไม่มีพารามิเตอร์ให้เรียน) → เบา, เสถียร, อนุรักษ์มวล, และ หาอนุพันธ์ได้ (differentiable) → backprop ผ่านได้ → ใช้ทำ source inversion (3.7) ได้. นี่คือหัวใจ "หมอกควันข้ามแดน" ของเรา: ลมจากเมียนมาพัดฝุ่นเข้าไทย = เทอมนี้ทำงาน.
3
Turbulent Diffusion — การกระจายจากความปั่นป่วน UNet++ ทำนาย K + FD kernel

ฟิสิกส์: นอกจากลมพัดเป็นทิศ อากาศยัง "ปั่นป่วน" (turbulent) ทำให้ฝุ่นกระจายจากที่เข้มไปจาง. ความแรงการกระจายวัดด้วย eddy diffusivity K.

∇K∇C = ∂/∂x(K ∂C/∂x) + ∂/∂y(K ∂C/∂y)

ทฤษฎี K-theory: ฝุ่นไหลจากที่เข้มข้นสูง→ต่ำ (เหมือนความร้อนไหลจากร้อน→เย็น) ด้วยอัตราที่แปรผันกับ K. ปัญหาคือ K ไม่มีสูตรตายตัว — มันขึ้นกับสภาพอากาศ (ยิ่งแดดร้อน/ลมแรง ยิ่งปั่นป่วน K สูง). นี่แหละที่ต้องใช้ NN!

K = EncoderK(T, RH, PRS, PBLH)   ← UNet++ เรียนรู้ K จากอากาศ
Input ให้ UNet++: อุณหภูมิ T, ความชื้น RH, ความดัน PRS, ความสูง PBL  |  Output: field ของ K → เอาไปเข้า FD kernel เดิม (Wx, Wy) ได้ ΔC

ออกแบบฉลาดตรงไหน: เขาไม่ให้ NN เดา ΔC ตรง ๆ. แต่ให้ NN เดาแค่ K (ค่าฟิสิกส์ที่ไม่รู้) แล้วเอา K ไปเข้าสูตรฟิสิกส์ที่รู้แน่ (finite difference). → ฟิสิกส์คุมโครง, NN เติมเฉพาะช่องว่าง. นี่คือหัวใจ principle-informed อีกครั้ง.

🎯 ใช้กับเรา
รูปแบบ "NN เดาสัมประสิทธิ์ฟิสิกส์ → เสียบเข้าสูตร" คือเทมเพลตที่ HazeNet ใช้ซ้ำได้ทุกที่. และเพราะ K มาจาก met เท่านั้น (ไม่ขึ้นกับ tracer ตัวไหน) → diffusion เป็น "conditionally-linear" (บท 3.5) → ปลอดภัยสำหรับ tagged-tracer: ทุก tracer ใช้ K ตัวเดียวกัน แยกแล้วบวกกลับได้.
4
Chemistry — ปฏิกิริยาเคมี UNet++ ทำนาย k + 1×1 conv

ฟิสิกส์: สารเกิด/สลายจากปฏิกิริยา. ใช้รูปแบบ kinetics ของ Atkinson: อัตรา = ค่าคงที่ปฏิกิริยา × ฟังก์ชันของความเข้มข้นสารตั้งต้น.

Ri = ki × f(Cj)

ทฤษฎี Arrhenius / Atkinson kinetics: ค่าคงที่ k ขึ้นกับอุณหภูมิ (ยิ่งร้อนปฏิกิริยายิ่งเร็ว — กฎ Arrhenius). ส่วน f(Cj) คือ "ยิ่งสารตั้งต้นเยอะ ปฏิกิริยายิ่งเร็ว" (อาจไม่เชิงเส้น เช่นปฏิกิริยาอันดับสอง ∝ C²).

FastCTM แยกเป็น 2 ส่วน:

  • k = Encoderk(T, RH, PRS, WS, RN, CFRAC) — UNet++ เรียน "ค่าคงที่ปฏิกิริยา" จากสภาพอากาศ (อุณหภูมิ, ความชื้น, เมฆ, ฝน)
  • f(Cj) — เครือข่าย conv 1×1 เล็ก ๆ เรียน "ความเข้มข้นมีผลแบบไหน" (จับความไม่เชิงเส้น)
⚠️ จุดวิกฤตสำหรับเรา: เทอมเคมีคือ เทอมเดียวที่อาจไม่เชิงเส้นใน C (เพราะ f(C) อาจเป็น C² หรือสารหลายตัวคูณกัน). 4 โมดูลอื่นเชิงเส้นหมด แต่ตัวนี้ไม่. นี่คือจุดที่ superposition / tagged-tracer (3.6) จะเริ่มมี error — ฝุ่นทุติยภูมิ (ซัลเฟต, ไนเตรต, SOA) ที่เกิดจากเคมี แยกแหล่งได้ไม่เป๊ะ 100%.
🎯 ใช้กับเรา
โครงสร้าง R = k(met) × f(C) ตรงกับ "conditionally-linear operator" (บท 3.5) เป๊ะ: ถ้ามองว่า k คือ "เงื่อนไข" (condition) ที่มาจาก met และ f เชิงเส้น → ทั้งโอเปอเรเตอร์เป็น conditionally-linear. ถ้า f ไม่เชิงเส้น → ตรงนี้คือ caveat ที่เราต้องเขียนใน limitation ของ HazeNet อย่างซื่อสัตย์ (ตรงกับที่เขียนไว้แล้วในบท 3.6).
5
Deposition — การตกสะสม UNet++ ทำนาย d

ฟิสิกส์: ฝุ่นออกจากอากาศ 2 ทาง — แห้ง (dry): ตกพื้น/ติดใบไม้, และ เปียก (wet): ฝนชะลงมา. โมเดลเป็นเชิงเส้น "ยิ่งฝุ่นเยอะ ยิ่งตกเยอะ":

Di = di × Ci   ,   d = Encoderd(WS, RH, RN, HGT, LULC)
Input ให้ UNet++: ความเร็วลม WS, ความชื้น RH, ฝน RN, ความสูงภูมิประเทศ HGT, การใช้ที่ดิน LULC (เมือง/ป่า)  |  Output: ค่า d (deposition velocity) → ΔC ลดลง

ทฤษฎี deposition velocity: ค่า d ขึ้นกับ (1) อากาศ — ลมแรงพัดฝุ่นชนผิวบ่อย d สูง, ฝนตก d พุ่ง; (2) พื้นผิว — ป่าไม้เป็น "ตัวดูดฝุ่น" ชั้นดี (ใบไม้พื้นผิวเยอะ) ส่วนเมือง/น้ำ d ต่ำ. นี่คือเหตุผลที่ต้องใส่ LULC.

🎯 ใช้กับเรา
D = d×C เป็นเชิงเส้นใน C → ปลอดภัยกับ tagged-tracer. และการใส่ ภูมิประเทศ (DEM/HGT) + การใช้ที่ดิน เป็น input ตรงกับเหตุผลที่งานเรา "ต้องใช้ GeoAI จริง" — ภูมิประเทศภาคเหนือ (แอ่งเชียงใหม่) มีผลต่อทั้งการสะสมและการขังของฝุ่น.

5. ทฤษฎี UNet++ — โครงข่ายที่ใช้ทำนาย "field"

สังเกตว่า 3 โมดูล (diffusion, chemistry, deposition) ใช้ UNet++ เหมือนกันหมด เพื่อทำนายค่าฟิสิกส์ (K, k, d) ทั้งแผนที่จากสภาพอากาศ. UNet++ คืออะไร ทำไมเหมาะ?

หลักการ UNet: มี 2 ครึ่ง — Encoder ย่อภาพลงเรื่อย ๆ (เห็น "บริบทกว้าง" เช่นระบบความกดอากาศ) แล้ว Decoder ขยายกลับเป็นแผนที่ความละเอียดเดิม. Skip connection = สายส่งตรงจาก encoder ไป decoder ชั้นเดียวกัน เพื่อไม่ให้รายละเอียดเล็ก ๆ หาย.

UNet++ ต่างจาก UNet: เพิ่ม skip connection แบบ "ซ้อนหนาแน่น" (nested/dense) ระหว่างชั้น → เชื่อม feature หลายสเกลได้ดีขึ้น → ทำนาย field ที่เนียนและแม่นขึ้น. (FastCTM ใช้ 5 ชั้น: 16, 32, 64, 128, 256 filters)

ทำไม UNet++ เหมาะกับงานนี้: การทำนาย K/k/d คือการแปลง "แผนที่อากาศ → แผนที่สัมประสิทธิ์ฟิสิกส์" (image-to-image). UNet เกิดมาเพื่องานนี้พอดี — มันเห็นทั้งบริบทกว้าง (ระบบอากาศ) และรายละเอียดเฉพาะจุด (ภูเขา/เมือง) พร้อมกัน.
🎯 ใช้กับเรา
HazeNet ใช้ UNet/UNet++ เป็น "encoder" ทำนายสัมประสิทธิ์ฟิสิกส์ได้เลย — เป็นสถาปัตยกรรมเบา เทรนได้ในงบ A100 จำกัด. หรือถ้าจะประหยัดกว่านั้น อาจ fine-tune encoder จาก foundation model (Prithvi/Aurora) ตามที่คุยใน related-work.

6. การเดินเวลาแบบ Recursive + การทดลองที่เฉียบที่สุดในเปเปอร์

6.1 Recursive time-stepping

FastCTM เดินทีละ 1 ชั่วโมง (ภายในซอยเป็น timestep ย่อย 60 วินาที เพื่อความเสถียร). แต่ละชั่วโมง: รัน 5 โมดูล → ได้ ΔC → บวกเข้า C → เอา output เป็น input ของชั่วโมงถัดไป วนไปจนครบ 24 ชม.

6.2 🏆 การทดลอง "Zero-Input" — พิสูจน์ว่ามันเรียน "ฟิสิกส์" จริง ไม่ใช่ "จำสถิติ"

นี่คือการทดลองที่ฉลาดที่สุดในเปเปอร์ และเป็นบทเรียนล้ำค่าสำหรับเรา:

เขาทำอะไร: ลองป้อน "ค่าฝุ่นเริ่มต้น = 0" (โกหกโมเดลว่าตอนเริ่มอากาศสะอาดหมด) แล้วดูว่าพยากรณ์ออกมาเป็นยังไง.

ผล: พอเวลาผ่านไปหลายชั่วโมง พยากรณ์จากการเริ่มที่ 0 เกือบเหมือนกันเป๊ะกับการเริ่มด้วยค่าจริง!

แปลว่าอะไร: โมเดลไม่ได้แค่ "จำว่าเมื่อวานเท่าไหร่พรุ่งนี้ก็ประมาณนั้น" (spatiotemporal autocorrelation — ทางลัดที่โมเดล AI ชอบโกง). แต่มันสร้างฝุ่นขึ้นมาใหม่จาก emission + ฟิสิกส์จริง จนลู่เข้าค่าที่ถูกต้องได้เอง → พิสูจน์ว่ามันเรียน "กลไก" ไม่ใช่ "ความจำ".

🎯 ใช้กับเรา — สำคัญมาก
HazeNet ต้องทำการทดลองนี้เป๊ะ ๆ. เหตุผล: งานเราจะอ้างว่า "ทำ source attribution/inversion ได้" — ซึ่งจะเชื่อถือได้ก็ต่อเมื่อโมเดลตอบสนองต่อ emission จริง ไม่ใช่จำสถิติ. การทดสอบ zero-input คือ "หลักฐานชิ้นเอก" ที่กรรมการจะเชื่อว่าโมเดลเราเข้าใจฟิสิกส์ของแหล่งกำเนิดจริง. ใส่ไว้ใน validation plan ของเราเลย.

เพราะแยกโมดูล FastCTM ทำ process analysis ได้: ในเหตุการณ์ฝุ่นเดือนมกราคม 2023 มันบอกได้ว่า "transport คือตัวขับหลัก ที่ทำให้ PM2.5 พุ่ง ส่วน deposition เป็นตัวลดรอง" — ซึ่งตรงกับผล process analysis ของ CMAQ จริง.

และทำ emission sensitivity ได้: ลองเพิ่ม NOₓ/SO₂ เป็น 2 เท่า → ดู PM2.5 เพิ่มที่ไหนบ้าง (สอดคล้อง CMAQ ที่ R≈0.9).

🎯 ใช้กับเรา — นี่คือ "ตัวอ่อน" ของจุดขายเรา
สังเกตให้ดี: "process analysis" (แยกว่ากระบวนการไหนทำฝุ่นขึ้น) กับ "emission sensitivity" (เพิ่ม emission แล้วดูผล) คือเวอร์ชันตั้งต้นของ attribution + inversion ที่เราจะทำ! FastCTM แตะไว้แค่ผิว ๆ (แยกตาม "กระบวนการ"). HazeNet ไปไกลกว่า: เราแยกตาม "แหล่งกำเนิดเชิงพื้นที่" (เมียนมา vs ลาว vs ไทย) ด้วย tagged-tracer (3.6) และกลับทิศหา emission ด้วย autodiff (3.7). → ช่องว่างตรงนี้คือที่ยืนของเรา.

🔴 บทเรียนวิกฤต #1 — เรื่องอนุรักษ์มวล: FastCTM ยอมให้ระบบ "ไม่อนุรักษ์มวลเป๊ะ" ได้ เพราะเป้าเขาคือแค่ "พยากรณ์ให้แม่น". แต่ HazeNet ทำแบบนั้นไม่ได้! เพราะ tagged-tracer attribution (3.6) ต้องการให้ "ผลรวมของทุก tracer = ความเข้มข้นรวม" เป๊ะ ๆ ไม่งั้น "% การมีส่วนร่วม" จะไม่รวมเป็น 100%. → เราต้องเข้มกว่า FastCTM: ใช้เทคนิค hard-constraint/projection (งาน Tier 2 #5 ใน related-work) บังคับอนุรักษ์มวลในทุกโมดูล.
🔴 บทเรียนวิกฤต #2 — เรื่องตอบสนอง emission (สำคัญสุด): จุดอ่อนที่ใหญ่ที่สุดของ FastCTM คือ "ตอบสนองต่อการเปลี่ยน emission ได้ไม่ดี เพราะเทรนบน emission คงที่". แต่ความสามารถนี้คือ "หัวใจ" ของ source inversion ที่ HazeNet ต้องทำ! (เรากลับทิศหา emission จากค่าที่วัด — ต้องอาศัยว่าโมเดลเข้าใจ "emission เปลี่ยน → ฝุ่นเปลี่ยนยังไง" อย่างถูกต้อง).

→ สิ่งที่เราต้องทำต่างจาก FastCTM: ตั้งใจเทรนด้วย emission ที่หลากหลาย/ถูกรบกวน (perturbed emissions) โดยเฉพาะ เพื่อให้โมเดลเรียน "ความไว (sensitivity) ต่อแหล่ง" อย่างแม่นยำ. นี่คือ insight ที่มีค่าที่สุดจากการอ่านเปเปอร์นี้ — จุดอ่อนของเขา = ความสามารถที่เราขาดไม่ได้.

10. ⭐ FastCTM ↔ HazeNet — เทียบทุกจุด

ส่วนนี้คือหัวใจของบท — สรุปว่า เราลอกอะไร เติมอะไร และทำต่างตรงไหน:

10.1 สิ่งที่เรา "ลอกได้เลย" (พิสูจน์แล้วว่าเวิร์ก)

10.2 สิ่งที่เรา "เติม" (จุดขาย — FastCTM ไม่มี)

  • 🏷️ Tagged-tracer attribution (3.6): แยก C เป็น C⁽ˢ⁾ ตามแหล่ง (เมียนมา/ลาว/ไทย/...) รันผ่าน 5 โมดูลเดียวกัน → บอก "% ฝุ่นมาจากไหน". FastCTM แยกได้แค่ตาม "กระบวนการ" เราแยกตาม "แหล่งเชิงพื้นที่".
  • 🔄 Source inversion ผ่าน autodiff (3.7): เพราะทุกโมดูล differentiable → .backward() ให้ ∂loss/∂Q → กลับทิศหา emission จากค่าที่วัดได้. FastCTM ทำ sensitivity แบบหยาบ ๆ เท่านั้น.
  • 🎲 Calibrated uncertainty (3.9): quantile regression + CRPS → ทำนายพร้อมช่วงความเชื่อมั่น. FastCTM ให้ค่าเดียว.
  • 🌏 โฟกัสหมอกควันข้ามแดน SEA: โดเมน + ข้อมูลเปิด (FINN/Himawari/Air4Thai) + งบเล็ก. FastCTM ทำจีน.

11. สรุป + เช็กลิสต์นำไปใช้

FastCTM ในหนึ่งประโยค: "NN ที่แยกเป็น 5 โมดูลตามสมการ advection-diffusion-reaction โดยใช้สูตรฟิสิกส์ฟิกซ์ (emission, transport) ผสม UNet++ ที่เดาสัมประสิทธิ์ที่ไม่รู้ (K, k, d) แล้วเดินเวลาแบบ recursive — เร็วแบบ AI แต่ตีความได้แบบ CTM."

✅ เช็กลิสต์ที่ดึงมาจากเปเปอร์นี้ไปใส่ HazeNet:

  • ☐ ออกแบบ core เป็น 5 โมดูลตามสมการ governing (ลอกโครง)
  • ☐ ใช้ upwind kernel ฟิกซ์สำหรับ transport (อนุรักษ์มวล + differentiable)
  • ☐ ใช้ UNet++/encoder เดา K, k, d จาก met
  • ☐ บังคับอนุรักษ์มวลทุกโมดูล (เข้มกว่า FastCTM — เพื่อ attribution)
  • ☐ เทรนด้วย perturbed emissions (เพื่อ source inversion — แก้จุดอ่อนเขา)
  • ☐ ทำการทดสอบ zero-input เป็นหลักฐานว่าเรียนฟิสิกส์จริง
  • ☐ ใช้ random patch sampling
  • ☐ เพิ่มชั้น tagged-tracer + autodiff inversion + quantile/CRPS (จุดขายเรา)
  • ☐ วางงบ compute ให้พอดี ~15 A100-hr (โดเมนเล็ก/fine-tune/mixed precision)
🎯 ประโยคพูดกับกรรมการ
"HazeNet ต่อยอดสถาปัตยกรรม principle-informed ที่ FastCTM (GMD 2025) พิสูจน์แล้วว่าใช้ได้ — แต่เราเติม 3 ความสามารถที่เขาไม่มี: source attribution เชิงพื้นที่ด้วย tagged-tracer, source inversion ด้วย differentiable programming, และ uncertainty ที่ calibrate ได้ — พร้อมแก้จุดอ่อน 2 ข้อของเขา (อนุรักษ์มวล + การตอบสนอง emission) ซึ่งจำเป็นต่อความถูกต้องของ attribution โดยเฉพาะกับปัญหาหมอกควันข้ามแดนอาเซียน"
