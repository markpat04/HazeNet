2. แนวคิดหลัก — "ทำลายแล้วสร้างใหม่"

2.1 อุปมาง่ายๆ

จินตนาการรูปภาพสวยงาม แล้วค่อยๆ ราดทรายขาวทับจนมองไม่เห็นอะไร:

ภาพต้นฉบับ x₀
→ noise →
x₂₅₀ (noise ปาน กลาง)
→ noise →
x₁₀₀₀ = pure noise

Forward process: ภาพ → noise ทีละขั้น (T ขั้น, T ≈ 1000)
Backward process: noise → ภาพ ทีละขั้น (เรียนจาก neural network)

Key insight: ถ้าเรียนรู้ที่จะ ย้อนกลับ process นี้ได้ → สามารถสร้างภาพใหม่ได้โดยเริ่มจาก noise สุ่ม

2.2 ทำไมชื่อ "Diffusion"

ชื่อมาจาก Diffusion ในทางฟิสิกส์ — กระบวนการที่อนุภาคกระจายตัวจากบริเวณความเข้มข้นสูงสู่ต่ำจนสม่ำเสมอ ซึ่งคล้ายกับการเพิ่ม noise จนข้อมูลกระจายสม่ำเสมอ (Gaussian) เช่นเดียวกัน

ไม่เกี่ยวกับ Advection-Diffusion ในฟิสิกส์บรรยากาศ (1.2) เลย — แม้จะใช้คำเดียวกัน ชื่อนี้บังเอิญซ้ำกัน เป็นคนละ concept ทั้งหมด

3. Forward Process — เพิ่ม Noise ทีละขั้น

3.1 สูตร

ที่ขั้น t เพิ่ม Gaussian noise เข้าไปในภาพ:

-- Forward process: q(xₜ | xₜ₋₁) --
xₜ = √(1−βₜ) · xₜ₋₁  +  √βₜ · ε
     ↑                    ↑
  ภาพเดิม (หรี่ลง)    noise ใหม่ที่เพิ่ม

เมื่อ β ≡ noise schedule: β₁ < β₂ < ... < β_T
                          เพิ่ม noise มากขึ้นเรื่อยๆ ในแต่ละขั้น
ε ~ N(0, I)  ← random Gaussian noise

3.2 Shortcut: กระโดดตรงไป xₜ

ข้อดีของสูตรนี้คือสามารถคำนวณ xₜ จาก x₀ ตรงๆ ได้โดยไม่ต้องวนลูป T รอบ:

-- Reparameterization trick: ไปถึง xₜ จาก x₀ ตรงเลย --
ᾱₜ = ∏ᵢ₌₁ᵗ (1−βᵢ)   ← product ของ (1−β) ทุกขั้นถึง t

xₜ = √ᾱₜ · x₀  +  √(1−ᾱₜ) · ε     (ε ~ N(0,I))
     ↑                ↑
  ภาพต้นฉบับ        noise รวม
  (จางลงตาม t)

เมื่อ t→T: ᾱₜ→0 → xₜ ≈ pure noise (ข้อมูลต้นฉบับหายไปหมด)

ตัวอย่างค่า ᾱₜ:

ขั้น tᾱₜความหมาย
01.00ภาพต้นฉบับ 100%
2500.63ยังเห็นโครงร่างภาพ
5000.11เริ่มไม่ชัด
7500.006แทบไม่เหลือ signal
1000~0Pure Gaussian noise

4. Backward Process — ลบ Noise ทีละขั้น (สิ่งที่ Neural Network เรียน)

4.1 แนวคิด

ถ้ารู้ distribution ของ p(xₜ₋₁ | xₜ) สามารถย้อนกลับ forward process ได้ — แต่ distribution นี้ขึ้นกับข้อมูลทั้ง dataset → neural network εθ ทำหน้าที่ประมาณค่ามัน

-- Backward step: denoising ทีละขั้น --
p_θ(xₜ₋₁ | xₜ) = N(μ_θ(xₜ, t), Σ_θ)

เมื่อ μ_θ คำนวณจาก noise ที่ network ทำนาย:
μ_θ(xₜ, t) = (1/√αₜ) · [xₜ − (βₜ/√(1−ᾱₜ)) · ε_θ(xₜ, t)]
                                              ↑
                                   Network predict noise

4.2 กระบวนการ Inference

เริ่มจาก pure noise → ลบทีละขั้น T รอบ:

1สุ่ม xT ~ N(0, I) (pure noise)
2for t = T, T−1, ..., 1:
    ε̂ = ε_θ(xₜ, t)   ← network predict noise
    xₜ₋₁ = μ_θ(xₜ, ε̂) + σₜ·z   ← denoising step
3ได้ x₀ = ภาพที่สร้าง

T ≈ 1000 steps (DDPM) หรือ ~50 steps (DDIM เร็วกว่า แต่ยังช้าอยู่)

5. Training — "สอนให้ Predict Noise"

5.1 Loss Function ของ Diffusion

ฟังดูซับซ้อน แต่ training objective ง่ายมาก:

-- Simple Diffusion Loss (DDPM) --
L(θ) = E[||ε − ε_θ(xₜ, t)||²]
          ↑      ↑
    noise จริง   noise ที่ predict

กล่าวคือ: เอาภาพจริง x₀ → เพิ่ม noise รู้ค่า ε ที่เพิ่มเข้าไป
          → ส่ง xₜ ให้ network → network ทาย ε ให้ถูก
          → minimize MSE ของ noise prediction
# ตัวอย่างโค้ด Training (ย่อ)
def diffusion_loss(model, x0, noise_schedule):
    t = torch.randint(1, T, (1,))      # สุ่ม timestep
    eps = torch.randn_like(x0)           # noise จริง
    alpha_bar = noise_schedule[t]
    xt = alpha_bar.sqrt() * x0 + (1 - alpha_bar).sqrt() * eps
    eps_pred = model(xt, t)              # network predict noise
    return F.mse_loss(eps_pred, eps)     # minimize noise prediction error

5.2 Architecture ของ ε_θ

โดยทั่วไปใช้ U-Net (3.3) ที่รับ (xₜ, t) แล้วออก ε̂ — บางโมเดลใช้ Transformer แทน (DiT — Diffusion Transformer ใน Sora/SD3)

ที่น่าสนใจ: U-Net ที่เราใช้ใน 3.3 สำหรับ HazeNet คือ architecture เดียวกับที่ Stable Diffusion ใช้เป็น denoising network — แต่ task ต่างกันโดยสิ้นเชิง

7. 5 เหตุผลที่ HazeNet ไม่ใช้ Diffusion Model

นี่คือส่วนหลักของบท — ท่องจำให้ขึ้นใจ 5 ข้อนี้:

❌ เหตุผล 1: ช้าเกินไปสำหรับ Operational Forecast

Diffusion inference ต้องทำ T = 50–1000 denoising steps ต่อ 1 sample

สถานการณ์Forward passes ที่ต้องทำเวลาโดยประมาณ
HazeNet rollout 48h48 steps × 1 = 48~10 วินาที (A100)
Diffusion rollout 48h (DDIM)48 steps × 50 = 2,400~8 นาที
Diffusion ensemble 50 members2,400 × 50 = 120,000~7 ชั่วโมง

ระบบพยากรณ์ฝุ่นต้องอัปเดตทุก 1–6 ชั่วโมง → Diffusion rollout เต็มๆ ไม่ทันรอบอัปเดตถัดไป

❌ เหตุผล 2: ทำลาย Superposition → Attribution พัง

Diffusion model เป็น non-linear อย่างมาก ในทุก denoising step

-- ต้องการ (3.6 Tagged-Tracer): --
F(C⁽¹⁾ + C⁽²⁾) = F(C⁽¹⁾) + F(C⁽²⁾)   ← linearity in C

-- Diffusion Decoder: --
DiffDecode(C⁽¹⁾ + C⁽²⁾) ≠ DiffDecode(C⁽¹⁾) + DiffDecode(C⁽²⁾)
→ Superposition พัง → Tagged-Tracer ใช้ไม่ได้ → Attribution ไม่ถูกต้อง

นี่คือเหตุผลหลักทาง scientific ว่า diffusion ขัดกับ design goal หลัก ของ HazeNet

❌ เหตุผล 3: Source Inversion (3.7) เป็นไปไม่ได้จริง

Source inversion ต้องการ ∂(PM2.5)/∂(Q) ผ่านทั้ง forward model
กับ Diffusion: gradient ต้องไหลผ่าน T = 50–1000 stochastic denoising steps

-- Gradient ผ่าน diffusion chain: --
∂loss/∂Q = (∂loss/∂x₀) · (∂x₀/∂x₁) · (∂x₁/∂x₂) · ... · (∂x_T/∂Q)
                T = 1000 steps → vanishing gradient รุนแรงมาก

-- Memory ที่ต้องการ backprop ผ่าน T steps: --
mem = T × H × W × channels × 4 bytes
    = 1000 × 128 × 128 × 256 × 4  ≈  17 GB  ← เฉพาะ diffusion chain เดียว

ทั้งช้า ทั้ง RAM หมด ทั้ง vanishing gradient — adjoint ผ่าน diffusion chain ไม่ practical

❌ เหตุผล 4: ไม่รักษา Conservation Laws

HazeNet ออกแบบให้ อนุรักษ์มวลฝุ่น (2.4) — ∫C dV ลดได้แต่ไม่เพิ่มลอยๆ

Diffusion model สร้างภาพที่ "ดูสมจริง" แต่ไม่มีกลไกบังคับ conservation law:

# Diffusion อาจสร้างภาพที่
total_pm25_before = C.sum()
C_diffused = diffusion_decode(C)
total_pm25_after = C_diffused.sum()

assert total_pm25_after ≈ total_pm25_before   # ← FAILS
# ค่า PM2.5 รวมทั้ง domain เปลี่ยน → ผิดฟิสิกส์

สำหรับงานวิจัยที่ เคลมว่าพิสูจน์ได้ทางฟิสิกส์ นี่คือปัญหาร้ายแรง

❌ เหตุผล 5: Stochastic ขัดกับ Operational Determinism

การพยากรณ์ฝุ่นสำหรับ policy-making ต้องการ deterministic baseline ที่ชัดเจน:
"วันพรุ่งนี้ PM2.5 ที่เชียงใหม่จะเป็น X μg/m³ ± Y" ไม่ใช่ "อาจเป็น 80 หรือ 180 ก็ได้"

Diffusion sample ทุกครั้งต่างกัน → run ใหม่อีกรอบได้ผลต่างออกไป → ยากต่อการ:

  • ออกประกาศเตือน (ต้องการค่าที่ reproducible)
  • เปรียบเทียบกับ WRF-Chem (ต้องการ deterministic prediction)
  • Debug หรือ validate ผล (แต่ละ run ต่างกัน)
เรายังต้องการ uncertainty ใน HazeNet แต่เราได้มันจาก ensemble + quantile regression (3.9) แทน ซึ่ง deterministic และ reproducible

8. เปรียบเทียบโดยตรง — HazeNet vs Diffusion สำหรับ Task ของเรา

เกณฑ์HazeNet (เลือก)Diffusion Model (ไม่เลือก)

สรุป: จาก 8 เกณฑ์ HazeNet ชนะ 5, แพ้ 1 (visual realism ซึ่งไม่ใช่ goal ของเรา)
Diffusion ชนะ uncertainty ที่เราแก้ด้วย 3.9 ได้อยู่แล้ว → ไม่มีเหตุผลเพียงพอที่จะใช้ Diffusion

9. ที่ Diffusion ใช้ได้ดีใน Weather/Climate — ซื่อสัตย์

ก่อนตอบกรรมการต้องรู้ว่า Diffusion ไม่ใช่ "ไม่ดี" — แค่ไม่เหมาะกับ task ของเรา

9.1 CorrDiff (NVIDIA, 2023) — Diffusion ที่ใช้ถูกที่

Task: รับ forecast คร่าวๆ 25 km resolution → สร้าง 3 km resolution
ทำไม Diffusion เหมาะ:

  • ไม่ต้อง autoregressive rollout — แค่ step เดียว (low-res → high-res)
  • ไม่ต้องการ attribution หรือ inversion
  • Speed ไม่วิกฤต (offline post-processing)
  • Visual realism สำคัญ (ต้องการ sharp spatial detail)

→ นี่คือ use case ที่ diffusion ชนะ approach อื่น

9.2 GenCast (Google DeepMind, 2024) — Ensemble ด้วย Diffusion

Task: Global weather ensemble 15 วัน
ทำไม Diffusion เหมาะ:

  • Ensemble diversity คือ goal หลัก — diffusion natural ตรงนี้
  • Global scale — attribution ไม่ใช่ feature หลัก
  • Speed ยอมรับได้ (15 วัน forecast ไม่ต้องทำทุก 6 ชั่วโมง)

→ scale และ objective ต่างจาก HazeNet โดยสิ้นเชิง

9.3 Diffusion อาจช่วย HazeNet ได้อย่างไร (ถ้าจะใช้)

ถ้าวันหนึ่งต้องการ Diffusion ใน HazeNet ควรใช้เป็น post-processing เสริม ไม่ใช่ core model:

วิธีใช้บทบาทข้อดี
HazeNet → Diffusion refinerเพิ่ม spatial detail ให้ forecastresolution ดีขึ้น ไม่แตะ core
Diffusion สร้าง synthetic training dataเพิ่ม diversity ของ training scenariosdata augmentation
Diffusion posterior samplinguncertainty estimation เสริมensemble ด้วย sampling

แต่ทั้งหมดนี้คือ "add-on" ไม่ใช่ "แทนที่" core model

10. สคริปต์ตอบกรรมการ

จำสคริปต์นี้ไว้ — ใช้ตอบทั้งภาษาไทยและอังกฤษได้เลย

คำถามกรรมการ (ประมาณ):
"ทำไมไม่ใช้ diffusion model สำหรับ uncertainty? มันกำลัง popular มากในวงการ weather"

คำตอบแนะนำ (ไทย):

"เราพิจารณา diffusion model อย่างจริงจังครับ และทราบว่ามีผลงานที่น่าสนใจอย่าง GenCast (DeepMind) และ CorrDiff (NVIDIA)

อย่างไรก็ตาม HazeNet มี design constraint 3 ข้อที่ขัดกับ diffusion โดยตรง:

หนึ่ง — Attribution: เราต้องการ superposition property (C = ΣC⁽ˢ⁾) สำหรับ source attribution ซึ่ง diffusion decoder ที่ non-linear ทำลายคุณสมบัตินี้

สอง — Source inversion: เราต้องการ differentiable path สำหรับ adjoint-based inversion ซึ่งการ backprop ผ่าน 1000 stochastic denoising steps ไม่ practical

สาม — Speed: ระบบพยากรณ์ฝุ่นต้องอัปเดตทุก 1–6 ชั่วโมง diffusion rollout 48 ชั่วโมงด้วย 50 steps = 2400 forward passes ต่อ sample ซึ่งช้าเกินไปสำหรับ operational use

สำหรับ uncertainty เราใช้ quantile regression และ ensemble ซึ่งให้ calibrated uncertainty estimates โดยไม่เสียคุณสมบัติสองข้อแรก"

คำตอบแนะนำ (อังกฤษ — สำหรับ international reviewer):

"We carefully considered diffusion models, particularly given strong results from GenCast (DeepMind) and CorrDiff (NVIDIA).

However, HazeNet has three hard design constraints that are fundamentally incompatible with diffusion:

First, attribution: our source attribution via tagged-tracers requires superposition — that is, F(ΣC⁽ˢ⁾) = ΣF(C⁽ˢ⁾). The highly non-linear diffusion decoder violates this.

Second, source inversion: we need a differentiable forward model for adjoint-based emission inversion. Backpropagating through 1000 stochastic denoising steps is computationally prohibitive.

Third, operational speed: our system targets 1–6 hour update cycles. A 48-hour diffusion rollout with 50 DDIM steps requires 2400 forward passes per sample, making real-time ensemble generation infeasible on our compute budget.

We handle uncertainty through quantile regression and light ensembles (3.9), which provide calibrated prediction intervals without sacrificing the properties above."

10.1 Follow-up คำถามที่อาจถามต่อ

คำถาม follow-upคำตอบสั้น
"แต่ DDIM ช้าแค่ 50 steps จริงไหม?"ใช่ แต่ 50 steps × 48h rollout = 2400 passes · ถ้าต้องการ 50-member ensemble = 120,000 passes · HazeNet = 48 passes รวม
"Diffusion รักษา mass conservation ไม่ได้จริงหรือ?"ใช่ ไม่มี explicit mechanism · CorrDiff แก้บางส่วนด้วย conditioning แต่ไม่ใช่ hard constraint · งานเราต้องการ hard constraint (2.4)
"ทำไมไม่ใช้ diffusion แค่สำหรับ downscaling?"อาจเป็น future work ที่น่าสนใจ แต่ขอบเขต 2 เดือนและ A100 จำกัด · สำหรับ core task ที่ submit ไม่จำเป็น
"GenCast ดีกว่า ECMWF ENS แล้วไง?"GenCast เก่งมากสำหรับ global medium-range weather · แต่ task เราคือ regional PM2.5 attribution ที่ต้องการ linearity · คนละ objective

11. แบบฝึกหัด

Q1: ทบทวน Forward Process

ถ้า β = 0.02 (ค่า noise ที่เพิ่มแต่ละขั้น) และเริ่มจาก x₀ = ภาพจริง
หลังจาก 500 steps ค่า ᾱ₅₀₀ ≈ (1−0.02)⁵⁰⁰ = ? และ xₜ = √ᾱₜ · x₀ + √(1−ᾱₜ) · ε แปลว่าอะไร?

ดูเฉลย

ᾱ₅₀₀ = (0.98)⁵⁰⁰ ≈ (e^−0.02)⁵⁰⁰ = e^−10 ≈ 0.0000454
√ᾱₜ ≈ 0.0067 (ภาพต้นฉบับเหลือแค่ 0.7%) · √(1−ᾱₜ) ≈ 0.9999 (noise เกือบ 100%)
→ x₅₀₀ แทบเป็น pure noise แล้ว — signal ต้นฉบับหายไปหมด

Q2: Speed Calculation

ถ้า diffusion inference 1 step = 0.02 วินาที (A100)
และต้องการ 48-step rollout × 50 DDIM steps × 100-member ensemble
ทั้งหมดใช้เวลากี่นาที? เทียบกับ HazeNet 48h rollout ที่ใช้เวลา 8 วินาที

ดูเฉลย

Diffusion: 48 × 50 × 100 × 0.02 = 4,800 วินาที = 80 นาที
HazeNet: 8 วินาที (รวม ensemble ถ้า 100 members ≈ 800 วินาที = 13 นาที)
→ Diffusion ช้ากว่า ~6×–100× ขึ้นกับ ensemble size
→ ระบบที่ update ทุก 6 ชั่วโมงไม่สามารถรอ 80 นาทีต่อ forecast ได้

Q3: Superposition Test

สมมติมี denoising network f(x, t) ที่มี non-linear activation (relu, softmax)
พิสูจน์หรืออธิบายว่าทำไม f(C⁽¹⁾ + C⁽²⁾, t) ≠ f(C⁽¹⁾, t) + f(C⁽²⁾, t)

ดูเฉลย

Counter-example ง่าย: สมมติ f(x) = relu(x)
C⁽¹⁾ = −2, C⁽²⁾ = 3
f(C⁽¹⁾ + C⁽²⁾) = relu(1) = 1
f(C⁽¹⁾) + f(C⁽²⁾) = relu(−2) + relu(3) = 0 + 3 = 3
1 ≠ 3 → superposition พัง แม้แค่ activation เดียว
Diffusion network มี relu/attention หลายร้อยชั้น → superposition พังอย่างรุนแรง

Q4: เหตุผลที่ถูกต้อง

เลือกเหตุผลที่ ถูกต้องที่สุด ว่าทำไม HazeNet ไม่ใช้ diffusion (เลือกได้หลายข้อ):
A) Diffusion เป็นเทคนิคเก่าแล้ว ไม่ทันสมัย
B) Diffusion ทำลาย superposition property ที่จำเป็นสำหรับ attribution
C) Diffusion ช้าเกินไปสำหรับ operational 6-hourly update
D) Diffusion ไม่มี conservation law enforcement
E) Diffusion ยากเกินไปที่จะ implement

ดูเฉลย

คำตอบที่ถูก: B, C, D
A: ผิด — Diffusion เป็นเทคนิคที่กำลัง popular และใช้ใน GenCast/CorrDiff (2023–2024)
B: ถูก — non-linearity ทำลาย superposition → tagged-tracer พัง
C: ถูก — speed เป็นข้อจำกัดจริงสำหรับ operational system
D: ถูก — ไม่มี hard constraint → ผิดฟิสิกส์ได้
E: ผิด — เหตุผลทางเทคนิค ไม่ใช่เรื่องความยากของ implementation

Q5: กรรมการถาม

กรรมการถาม: "CorrDiff ของ NVIDIA ใช้ diffusion สำหรับ weather ได้ดีมาก ทำไมคุณถึงไม่ทำแบบนั้น?"
ให้ตอบใน 3–4 ประโยค โดยแสดงว่าเข้าใจทั้ง CorrDiff และ HazeNet

ดูเฉลย

"CorrDiff เป็นงานที่น่าสนใจมากครับ และเป็นตัวอย่างที่ดีว่า diffusion เหมาะกับ task ไหน CorrDiff ทำ downscaling — แปลง forecast คร่าวๆ เป็น high-resolution ในขั้นเดียว ไม่ต้อง rollout หลาย timestep และไม่ต้องการ attribution หรือ source inversion HazeNet ต้องการทั้งสองอย่างนั้นเป็นหัวใจ — tagged-tracer attribution ต้องการ linearity ใน C และ adjoint inversion ต้องการ differentiable forward model ที่ fast ดังนั้น design space ของเราต่างจาก CorrDiff โดยพื้นฐาน"
