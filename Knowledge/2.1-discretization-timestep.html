2.1 — Discretization + Timestep

แปลงสมการต่อเนื่องให้คอมพิวเตอร์คำนวณได้ — กริด (ที่ว่าง) + ก้าวเวลา

เริ่ม Module 2 (แกนคณิตของโมเดล) · บทนี้คือ "สะพาน" จากสมการ ADR (1.2) ไปสู่โค้ด tensor (0.1) · ผูกกับ HazeNet ตลอด

ทำไมบทนี้มาก่อน: สมการ ADR (1.2) เป็นคณิตศาสตร์ต่อเนื่อง — ∂C/∂t สมมติว่าเวลาไหลลื่นไม่ขาดตอน และ ∇ สมมติว่าที่ว่างละเอียดอนันต์ · แต่คอมพิวเตอร์ทำแบบนั้นไม่ได้ มันรู้จักแค่ "ตัวเลขในช่องตาราง" · Discretization = ศิลปะของการ "หั่น" ที่ว่างและเวลาเป็นช่อง ๆ เพื่อให้สมการกลายเป็นการบวกลบคูณบน tensor · เข้าใจบทนี้ = เข้าใจว่าทำไม HazeNet ถึงเป็น tensor รูปร่าง [เวลา, ช่อง, lat, lon] ได้

1. ต่อเนื่อง (continuous) vs ไม่ต่อเนื่อง (discrete)

โลกจริงต่อเนื่อง — ความเข้มข้นฝุ่น C มีค่าที่ทุกจุดในอากาศ และที่ทุกขณะเวลา · แต่คอมพิวเตอร์เก็บได้แค่ตัวเลขจำนวนจำกัด · เราจึง "สุ่มเก็บ" ค่าที่จุดเป็นระยะ ๆ

"การหั่นต่อเนื่องให้เป็นจุด ๆ" นี่แหละคือ discretization · เราหั่น 2 อย่าง: ที่ว่าง (เป็นกริด) และ เวลา (เป็น step)

2. หั่นที่ว่าง = กริด (grid spacing Δx) ★

เราวางตารางทับแผนที่ภาคเหนือ แล้วเก็บค่า PM2.5 หนึ่งค่าต่อ "ช่อง" (grid cell) · ระยะห่างระหว่างช่องเรียก Δx (delta-x)

  • Δx เล็ก = กริดละเอียด = เห็นรายละเอียดมาก แต่ช่องเยอะ → คำนวณหนัก/ช้า/กิน GPU
  • Δx ใหญ่ = กริดหยาบ = เร็ว แต่หยาบ มองไม่เห็นรายละเอียดเล็ก (เช่น หุบเขาแคบ)
เชื่อม HazeNet: เราเลือกกริด 0.1° (~11 กม.) · นี่คือ "Δx" ของเรา · ค่านี้คือ trade-off: ละเอียดพอเห็นความต่างระหว่างจังหวัด แต่หยาบพอให้รันบน 1 GPU ใน 2 เดือนได้ (ข้อจำกัดโครงการ) · จำได้ไหมว่า ERA5 หยาบ 0.25° (1.3)? เราจึงต้อง regrid มันลงมา 0.1° (หัวข้อ 4.3)

3. หั่นเวลา = timestep (Δt) ★

เวลาก็หั่นเหมือนกัน · แทนที่จะรู้ C ทุกขณะ เรารู้ C เป็น "เฟรม" ห่างกัน Δt · เหมือนหนังที่จริง ๆ เป็นภาพนิ่งหลายเฟรมต่อวินาที

เราเขียนค่าที่ step ต่าง ๆ ว่า:

Ct → Ct+1 → Ct+2 → …
แต่ละลูกศรห่างกัน Δt · โมเดล "เดินหน้า" ทีละ step

นี่คือที่มาของ "update equation" ของ HazeNet ที่เขียนว่า Ct+1 = (ทำอะไรบางอย่างกับ) Ct — มันคือกฎที่บอกว่า "จากเฟรมนี้ ไปเฟรมหน้ายังไง"

เชื่อม update equation หลัก: สมการแกนของเรา Ct+1 = 𝒜(u)Ct + g·E − D⊙Ct ก็คือ "กฎเดินหนึ่ง timestep" นั่นเอง · มันบอกว่า "เอาควันเฟรมนี้ เลื่อนตามลม (𝒜) บวกควันที่ปล่อยใหม่ (E) ลบที่หายไป (D) = ควันเฟรมหน้า"

4. อนุพันธ์ → ผลต่าง (finite difference)

ปัญหา: สมการ ADR มี ∂C/∂t และ ∇ ซึ่งเป็นอนุพันธ์ (อัตราการเปลี่ยน ณ จุดเดียว) · คอมพิวเตอร์คำนวณ "ณ จุดเดียว" ไม่ได้ แต่คำนวณ "ผลต่างระหว่างช่อง" ได้

กุญแจ: อนุพันธ์ ≈ ผลต่างหารระยะ

∂C/∂t  ≈  (Ct+1 − Ct) / Δt

∂C/∂x  ≈  (Ci+1 − Ci) / Δx

= "อัตราการเปลี่ยนตามเวลา" ≈ "ค่าต่างระหว่างเฟรมนี้กับเฟรมหน้า หารด้วยช่วงเวลา" · นี่เรียก finite difference (ผลต่างจำกัด) — เป็นวิธีพื้นฐานสุดในการแปลงสมการให้คอมพิวเตอร์แก้ได้

ทำไมเราไม่ต้อง implement finite difference เอง (ส่วนใหญ่): HazeNet ฉลาดตรงที่ไม่ได้แก้ finite difference ทื่อ ๆ สำหรับ advection (มันไม่เสถียร — ดูข้อ 5) · เราใช้ semi-Lagrangian (2.2) ที่ดีกว่า · แต่คุณต้องเข้าใจ finite difference ก่อน เพราะมันคือ "ภาษาฐาน" ที่อธิบายว่าทำไมวิธีอื่นถึงจำเป็น

5. กฎเหล็ก: เงื่อนไข CFL ★★ สำคัญมาก

นี่คือกฎที่ต้องจำ · มันโยง Δx, Δt, และความเร็วลม u เข้าด้วยกัน:

CFL = u · Δt / Δx ≤ 1
"ใน 1 timestep ควันต้องเคลื่อนไม่เกิน 1 ช่องกริด"

เข้าใจด้วยสามัญสำนึก

ถ้าลมเร็วมาก (u ใหญ่) และ timestep ยาว (Δt ใหญ่) → ควันในหนึ่ง step กระโดดข้ามหลายช่อง · วิธีเชิงตัวเลขแบบ finite difference จะ "ตามไม่ทัน" → ตัวเลขระเบิด (เป็น infinity / NaN) หรือเด้งไปมาไม่สมจริง

ดังนั้นเลือก Δt ให้เล็กพอเทียบกับ Δx และลมแรงสุดที่เป็นไปได้ · นี่คือเหตุผลหลักที่ CTM (1.5) ต้องใช้ timestep เล็ก ๆ → จำนวน step เยอะ → ช้า

เชื่อม HazeNet (สำคัญ): semi-Lagrangian advection (2.2) ที่เราใช้ ไม่ติดข้อจำกัด CFL รุนแรง — มันตามรอยควัน "ย้อนกลับ" ไปหาว่ามาจากไหน จึงเลื่อนข้ามหลายช่องได้โดยไม่ระเบิด · นี่เป็นเหตุผลหนึ่งที่เราเลือกมัน และทำให้ใช้ Δt ใหญ่กว่า (= step น้อยกว่า = เร็วกว่า CTM) · แต่คุณต้องรู้จัก CFL ก่อน ถึงจะเข้าใจว่าทำไม semi-Lagrangian ถึงเจ๋ง

6. ความละเอียดของ HazeNet (สรุปตัวเลขจริง)

มิติค่าของเราความหมาย
Δx (กริดพื้นที่)~0.1° (~11 กม.)หนึ่งช่อง ≈ ขนาดอำเภอใหญ่
โดเมนภาคเหนือ + ประเทศเพื่อนบ้านต้องครอบคลุมแหล่งข้ามแดน (1.4)
Δt (ก้าวเวลา)รายชั่วโมง (ตามอินพุต)ตรงกับความถี่ ERA5/Himawari
ขอบฟ้าพยากรณ์1–3 วันล่วงหน้า= เดินหลายสิบ step (autoregressive, ข้อ 7)
ทำไมต้องครอบเพื่อนบ้านด้วย: ถ้ากริดครอบแค่ภาคเหนือไทย ควันจากเมียนมาจะ "โผล่มาจากขอบ" โดยเราไม่รู้ที่มา → แยกแหล่งไม่ได้ · โดเมนต้องใหญ่พอครอบแหล่งต้นทาง นี่เป็นการตัดสินใจ discretization ที่ผูกตรงกับ pain point

7. เชื่อมกับ tensor + autoregressive rollout

เมื่อหั่นที่ว่าง+เวลาแล้ว ข้อมูลเราก็กลายเป็น tensor (จาก 0.1) รูปร่าง:

# รูปร่างข้อมูล HazeNet หลัง discretization
C.shape = (time, channel, lat, lon)
#         เวลา(step) ตัวแปร  แกนเหนือ-ใต้ แกนออก-ตก
#         จาก Δt     PM2.5/ลม/..  จาก Δx     จาก Δx

การ "พยากรณ์ล่วงหน้า" คือการเอา update equation มาเดินซ้ำ ๆ ทีละ step — เรียก autoregressive rollout:

C = C0                          # สภาพเริ่มต้น (วันนี้)
for step in range(48):       # เดิน 48 ชม. ข้างหน้า
    C = model(C, met[step])     # C^t -> C^{t+1} (กฎเดินหนึ่ง step)
# ตอนนี้ C = พยากรณ์ 48 ชม.ล่วงหน้า
error สะสม (ปัญหาสำคัญ): ทุก step เอาเอาต์พุตที่อาจผิดนิดหน่อยมาเป็นอินพุต step ถัดไป → error สะสมทบต้น ยิ่งเดินไกลยิ่งเพี้ยน · นี่คือ ε_rollout ในการแยก error ของเรา (จาก master plan) · เป็นเหตุผลที่เราต้องเทรนแบบ "เดินหลาย step ตอนเทรน" ด้วย ไม่ใช่เทรน 1 step แล้วหวังว่าจะเดินไกลได้

8. error ที่เกิดจาก discretization เอง

การหั่นให้หยาบ "ทิ้งข้อมูล" ไปบ้างเสมอ → เกิด error ที่หลีกเลี่ยงไม่ได้ 2 แบบ:

  • Truncation error (จากหั่นหยาบ): Δx/Δt ใหญ่ → finite difference ประมาณอนุพันธ์หยาบ → คลาดเคลื่อน · ทำให้เกิด "numerical diffusion" (ภาพเบลอเกินจริง เหมือนควันกระจายเองทั้งที่ไม่ควร)
  • Representation error (จากกริดหยาบ): หุบเขาแคบกว่า 11 กม. โมเดลมองไม่เห็น → ฝุ่นในหุบเขาเล็กถูกเฉลี่ยหายไป
เชื่อม error analysis (Module 5): นี่คือส่วนหนึ่งของ ε_structure (error จากโครงสร้าง/การออกแบบ) ในการแยก error ของเรา · เรายอมรับมันอย่างซื่อสัตย์: เลือก 0.1° เพราะ trade-off กับ GPU/เวลา และจะรายงานว่า error ส่วนนี้มาจากความหยาบของกริด ไม่ใช่จากนิวรอลเรียนไม่เก่ง · การแยกแหล่งของ errorให้ออกแบบนี้คือสิ่งที่ทำให้งานเป็นระดับ A*

9. Cheat sheet + แบบฝึกหัด

สรุปจำ:
  • Discretization = หั่นที่ว่าง+เวลาต่อเนื่อง → ช่องตาราง (ให้คอมพิวเตอร์คำนวณได้)
  • Δx = ระยะกริด (เล็ก=ละเอียดแต่ช้า) · HazeNet ใช้ ~0.1° (~11 กม.)
  • Δt = ก้าวเวลา · ทำให้เกิด update equation C^t → C^{t+1}
  • Finite difference: อนุพันธ์ ≈ ผลต่าง/ระยะ (ภาษาฐานของโมเดลเชิงตัวเลข)
  • CFL = u·Δt/Δx ≤ 1 ★★ — ควันเลื่อนไม่เกิน 1 ช่อง/step ไม่งั้นตัวเลขระเบิด
  • semi-Lagrangian (2.2) เลี่ยงข้อจำกัด CFL → ใช้ Δt ใหญ่กว่า → เร็วกว่า CTM
  • tensor: ข้อมูลกลายเป็น (time, channel, lat, lon)
  • autoregressive rollout: เดิน update ซ้ำ → พยากรณ์ไกล · แต่เกิด ε_rollout สะสม
  • discretization error: truncation + representation → ส่วนหนึ่งของ ε_structure

แบบฝึกหัด 1 — กริด Δx = 11 กม., timestep Δt = 1 ชม., ลมแรงสุด u = 40 กม./ชม. · CFL เท่าไร และโมเดลเสถียรไหม (ถ้าใช้ finite difference ธรรมดา)?

เฉลย

CFL = u·Δt/Δx = 40 × 1 / 11 ≈ 3.6 · มากกว่า 1 → ไม่เสถียร ถ้าใช้ finite difference ธรรมดา (ควันกระโดด ~3.6 ช่องต่อ step → ตัวเลขจะระเบิด) · ทางแก้: ลด Δt (เช่นเป็น 15 นาที → CFL≈0.9) หรือ ใช้ semi-Lagrangian ที่ไม่ติดข้อจำกัดนี้ (= สิ่งที่ HazeNet เลือก, 2.2)

แบบฝึกหัด 2 — ทำไม autoregressive rollout ถึงทำให้ error แย่ลงเรื่อย ๆ ตามขอบฟ้าพยากรณ์?

เฉลย

เพราะแต่ละ step เอาเอาต์พุตของ step ก่อน (ที่มี error เล็กน้อย) มาเป็นอินพุตของ step ถัดไป → error ถูกป้อนกลับและขยายทบต้นทุก step · ยิ่งเดินไกล error ยิ่งสะสม = ε_rollout · แก้โดยเทรนให้โมเดลเห็นการเดินหลาย step (multi-step training) ไม่ใช่เทรน 1 step

แบบฝึกหัด 3 — ถ้าเราอยากเห็นรายละเอียดฝุ่นในหุบเขาแคบ 5 กม. กริด 0.1° (11 กม.) ทำได้ไหม? นี่คือ error ชนิดไหน?

เฉลย

ทำไม่ได้ · หุบเขา 5 กม. เล็กกว่า 1 ช่องกริด (11 กม.) → ถูกเฉลี่ยรวมหายไป · นี่คือ representation error (ส่วนหนึ่งของ ε_structure) · ถ้าต้องการต้องลด Δx (กริดละเอียดขึ้น) ซึ่งแลกกับ GPU/เวลามากขึ้น — เป็น trade-off ที่เรายอมรับและรายงานอย่างซื่อสัตย์

จบ 2.1 แล้วคุณควร: เข้าใจว่าทำไมสมการต่อเนื่อง (1.2) ต้องถูก "หั่น" เป็นกริด (Δx) + ก้าวเวลา (Δt) ก่อนคอมพิวเตอร์จะคำนวณได้, รู้จัก finite difference และกฎ CFL (ว่าทำไมตัวเลขระเบิดได้), เห็นว่าข้อมูลกลายเป็น tensor (time,channel,lat,lon) และพยากรณ์ไกลด้วย autoregressive rollout (ที่มาของ ε_rollout)
