4.2 — รูปแบบไฟล์ geospatial 🟡 กลาง

NetCDF · GeoTIFF · Zarr · xarray — ภาษากลางที่ทำให้ข้อมูล 5 แหล่งจาก 4.1 "คุยกันรู้เรื่อง"

ต่อจาก 4.1 (ดึงข้อมูลดิบมาแล้ว) · ส่งต่อ → 4.3 (regrid ให้เป็นกริดเดียว) → 4.4 (ประกอบ datacube) · โยงกลับ 0.1 (tensor thinking)

โจทย์ของบทนี้: 4.1 ดึงข้อมูลดิบมา 5 แหล่ง แต่มันมาคนละรูปแบบ — ERA5 เป็น .nc, DEM เป็น .tif, ไฟ/สถานีเป็น CSV/JSON · จะเอามารวมกันต้องรู้จัก "รูปแบบไฟล์เชิงพื้นที่" และ เครื่องมือเดียวที่อ่านได้ทุกอัน

1. ทำไม CSV / numpy ธรรมดาไม่พอ

ข้อมูลของเราเป็น ก้อนหลายมิติที่มีความหมายเชิงพื้นที่-เวลา เช่น PM2.5 มี 3 แกน: เวลา × ละติจูด × ลองจิจูด · ถ้าเก็บเป็น numpy array เฉย ๆ จะเจอปัญหา 4 ข้อ:

ปัญหาของ numpy/CSV ดิบผลที่ตามมา
แกนไม่มีชื่อ — arr[3, 50, 80] คืออะไร?ต้องจำเองว่าแกน 0 คือเวลา จำสลับเมื่อไรพังเมื่อนั้น
ไม่มี coordinate — index 50 ตรงกับละติจูดเท่าไร?หาค่าที่ "ละติจูด 18.8°N" ไม่ได้โดยตรง ต้องคำนวณ index เอง
ไม่มี metadata — หน่วยอะไร? CRS อะไร?เผลอเอา μg/m³ ไปรวมกับ ppb · ไม่รู้ว่าพิกัดอยู่ระบบไหน
ใหญ่เกินแรม — datacube หลาย GBโหลดทั้งก้อนเข้าแรม → Colab แตก (OOM)
ทางออก: ใช้รูปแบบไฟล์ที่ "อธิบายตัวเองได้" (self-describing: มีชื่อแกน + พิกัด + หน่วย ติดมาในไฟล์) + เครื่องมือที่โหลด "เฉพาะส่วนที่ต้องใช้" ได้ · นั่นคือทั้งหมดที่บทนี้สอน

2. xarray — "numpy ที่มีป้ายชื่อแกน" ★ พระเอก

จาก 0.1 เราคิดข้อมูลเป็น tensor (ก้อนตัวเลขหลายมิติ) · xarray = tensor นั้น + ป้ายชื่อ + พิกัด + metadata · มี 2 ชนิด:

  • DataArray = ตัวแปรเดียว (เช่น PM2.5) พร้อมแกน/พิกัด
  • Dataset = หลายตัวแปรที่ใช้แกนร่วมกัน (เช่น u, v, PBL, PM2.5 บนกริดเดียวกัน) — นี่คือรูปของ datacube เรา

ความสามารถที่เปลี่ยนชีวิต: เลือกข้อมูลด้วย "ชื่อ" ไม่ใช่ตัวเลข index

import xarray as xr
ds = xr.open_dataset("era5_2024_03.nc")

# เลือกด้วย "ป้ายชื่อ" — อ่านรู้เรื่อง ไม่ต้องจำ index
chiangmai = ds["u10"].sel(latitude=18.8, longitude=98.98, method="nearest")
march15  = ds.sel(time="2024-03-15")            # เลือกทั้งวัน
domain   = ds.sel(latitude=slice(25,5), longitude=slice(92,110))

# แปลงความถี่เวลา/เฉลี่ย ได้ในบรรทัดเดียว (เข้าใจปฏิทินเอง)
three_hourly = ds.resample(time="3H").mean()
ทำไม xarray คือหัวใจของ pipeline เรา: ข้อมูลทุกแหล่งใน 4.1 จะถูกเปิดเป็น xarray → ใช้ .sel/.isel/.resample/.interp หั่น+จัดให้ตรงกัน → รวมเป็น Dataset เดียว = datacube (4.4) · คุณจะใช้ xarray เกือบทุกบรรทัดของงานข้อมูล

3. NetCDF — กล่องมาตรฐานวิทยาศาสตร์

NetCDF (.nc) คือฟอร์แมตมาตรฐานสำหรับเก็บอาเรย์หลายมิติ + พิกัด + metadata ในไฟล์เดียว · เป็นรูปแบบที่ ERA5 ส่งมาให้ และเป็นภาษากลางของวงการอุตุ-ภูมิอากาศ

  • self-describing: เปิดมาเห็นชื่อแกน/หน่วย/พิกัดครบ ไม่ต้องเดา
  • CF Conventions: กติกาตั้งชื่อ/หน่วยมาตรฐาน (เช่น standard_name, units) ทำให้เครื่องมือต่าง ๆ เข้าใจตรงกัน
  • สร้างบน HDF5 (netCDF4) · xarray เปิดด้วย xr.open_dataset() ตรง ๆ
ใช้เมื่อไร: เป็นรูปแบบ "อินพุตขาเข้า" จาก ERA5 (และผลิตภัณฑ์ดาวเทียมหลายตัว) · เหมาะกับไฟล์ขนาดกลาง · แต่พอ datacube ใหญ่มากและต้องอ่านขนานบน cloud → เราจะแปลงไป Zarr (ข้อ 5)

4. GeoTIFF / COG — กล่องแรสเตอร์ (ภาพมีพิกัด)

GeoTIFF (.tif) คือไฟล์ภาพแรสเตอร์ที่ "ฝังพิกัดโลก" ไว้ — รู้ว่าแต่ละพิกเซลอยู่ตรงไหนบนโลกผ่าน 2 อย่าง: CRS (ระบบพิกัด เช่น EPSG:4326 = lat/lon) และ geotransform (พิกัดมุม + ขนาดพิกเซล) · เป็นรูปแบบของ DEM และภาพดาวเทียมจำนวนมาก

COG (Cloud Optimized GeoTIFF) = GeoTIFF ที่จัดเรียงภายในเป็น tile + overview → ดึง "เฉพาะหน้าต่างที่ต้องใช้" จาก cloud ได้โดยไม่ต้องโหลดทั้งไฟล์ (Copernicus DEM บน AWS เป็น COG)

import rioxarray            # ส่วนขยายของ xarray สำหรับ raster + CRS
dem = rioxarray.open_rasterio("dem_chiangmai.tif")
print(dem.rio.crs)         # EPSG:4326
print(dem.rio.bounds())    # (west, south, east, north)
เชื่อมกับงานเรา: DEM (4.1) มาเป็น GeoTIFF/COG · เราเปิดด้วย rioxarray ซึ่งทำให้มันกลายเป็น xarray ที่ "รู้ CRS" → พร้อม reproject/regrid ให้เข้ากริด 0.1° ใน 4.3 · เรื่อง CRS/reprojection ลงลึกในบท 4.3

5. Zarr — กล่อง cloud-native สำหรับ datacube ★ ที่เก็บงานเรา

Zarr คือรูปแบบเก็บอาเรย์ขนาดใหญ่แบบ "หั่นเป็นก้อน (chunk) + บีบอัด" · ต่างจาก NetCDF ที่เป็นไฟล์เดียว — Zarr เก็บแต่ละ chunk เป็นไฟล์/ออบเจ็กต์แยก → ทำให้:

  • อ่านเฉพาะ chunk ที่ต้องใช้ ได้ ไม่ต้องโหลดทั้งก้อน → เหมาะกับ datacube หลาย GB บน Colab
  • อ่าน/เขียนขนาน และ ต่อข้อมูล (append) ได้ดี → สร้าง datacube ทีละเดือนแล้วต่อกันได้
  • เป็นมิตรกับ cloud (S3/GCS) — แต่ละ chunk ดึงทีละชิ้น
# บันทึก datacube เป็น Zarr พร้อมหั่น chunk ตามรูปแบบการใช้งาน
cube.chunk({"time": 24, "lat": 180, "lon": 200}).to_zarr("hazenet_cube.zarr")
# เปิดแบบ lazy — ยังไม่โหลดเข้าแรมจนกว่าจะใช้จริง
cube = xr.open_zarr("hazenet_cube.zarr")
การตัดสินใจของโปรเจกต์: เก็บ datacube สุดท้ายเป็น Zarr · ตอนเทรน DataLoader ดึงเฉพาะ chunk/หน้าต่างเวลาที่ batch ต้องใช้ → ไม่ต้องยัดทั้ง 10GB เข้าแรม Colab (เชื่อม 4.5)
เคล็ดลับ chunk: หั่น chunk ให้ตรงกับ "วิธีที่จะอ่าน" — ถ้าเทรนเป็นหน้าต่างเวลา ให้ chunk แกน time ขนาดเล็ก แต่เก็บ lat/lon ทั้งภาพในก้อนเดียว · chunk ผิดทาง = อ่านช้ากว่าเดิมหลายเท่า

6. dask + lazy / chunk — เครื่องมือไม่ให้แรมแตก

dask คือเครื่องยนต์ที่ทำให้ xarray "ขี้เกียจ (lazy)" — เวลาเปิดไฟล์ใหญ่ มันยัง ไม่โหลดข้อมูลจริง แค่จำว่า "จะทำอะไรบ้าง" (สร้างแผนการคำนวณ) · ข้อมูลจะถูกโหลด+คำนวณ ตอนที่เราสั่งเท่านั้น และโหลดทีละ chunk:

ds = xr.open_zarr("hazenet_cube.zarr")   # เร็วมาก ยังไม่โหลดอะไร (lazy)
m  = ds["pm25"].sel(time="2024-03").mean("time")   # ยังเป็นแค่ "แผน"
result = m.compute()   # ← ตรงนี้แหละถึงโหลด+คำนวณจริง (ทีละ chunk)
ทำไมสำคัญกับ Colab: A100 ใน Colab มีแรมจำกัด · lazy + chunk ทำให้เราทำงานกับ datacube ที่ "ใหญ่กว่าแรม" ได้ เพราะไม่เคยมีทั้งก้อนอยู่ในแรมพร้อมกัน · นี่คือกลไกหลักที่กันไม่ให้ OOM (4.5)

7. ข้อมูล "จุด" (CSV/JSON) อยู่ตรงไหนในภาพนี้

จำจาก 4.1: FIRMS (ไฟ) และ Air4Thai (สถานี) เป็น ข้อมูลจุด ไม่ใช่กริด → จึง ยังไม่ใช่ NetCDF/Zarr ตอนแรก · เราจัดการด้วยเครื่องมือคนละชุด แล้วค่อยแปลงเข้ากริด:

ข้อมูลเปิดด้วยแปลงเข้ากริดยังไง
FIRMS (CSV จุดไฟ)pandas / geopandasนับ/รวม FRP ลงช่อง 0.1° → กลายเป็น DataArray การปล่อย E (rasterize, 4.3)
Air4Thai (JSON สถานี)pandasเก็บเป็นค่าที่ "พิกัดสถานี" → ใช้เป็น target แบบจุดตอนคำนวณ loss (ไม่ต้อง rasterize เต็มกริดก็ได้)
ประเด็นสำคัญ: ปลายทางทุกอย่างคือ xarray · กริดมาก็เป็น DataArray อยู่แล้ว · จุดมาก็แปลงเป็น DataArray (สำหรับ E) หรือเก็บคู่พิกัดไว้เทียบ (สำหรับ target) · พอทุกอย่างเป็น xarray บนแกนเดียวกัน → ประกอบเป็น Dataset เดียวได้ใน 4.4

8. ตารางเทียบ + ใช้อันไหนเมื่อไร

สิ่งนี้คืออะไรโครงสร้างใช้ในงานเราตอน
xarrayไลบรารี (ไม่ใช่ไฟล์) — tensor มีป้ายชื่ออยู่ในแรมทุกขั้นตอนของงานข้อมูล
NetCDFไฟล์อาเรย์หลายมิติ + metadataไฟล์เดียวอินพุต ERA5/ดาวเทียม (ขาเข้า)
GeoTIFF/COGภาพแรสเตอร์ + CRS + geotransformไฟล์เดียว (COG=tile)DEM/ภาพดาวเทียม (ขาเข้า)
Zarrอาเรย์หั่น chunk + บีบอัดโฟลเดอร์ของ chunkเก็บ datacube สุดท้าย (ขาออก/เทรน)
pandas/geopandasตาราง/จุดเชิงพื้นที่ตาราง (จุด)FIRMS, Air4Thai (ก่อน rasterize)
กฎจำง่าย: กริดเข้า → NetCDF/GeoTIFF · จุดเข้า → pandas · กลางทาง → xarray · เก็บผลลัพธ์ใหญ่ → Zarr

9. Workflow ของ HazeNet (ร้อยทุกอย่างเข้าด้วยกัน)

import xarray as xr, rioxarray, pandas as pd

# 1) เปิดแต่ละแหล่งให้เป็น xarray
met = xr.open_dataset("era5_2024_03.nc")            # NetCDF → Dataset
dem = rioxarray.open_rasterio("dem.tif").squeeze()    # GeoTIFF → DataArray
fire = pd.read_csv("firms.csv")                       # CSV จุด (rasterize ใน 4.3)

# 2) จัดให้อยู่กริด/เวลาเดียวกัน (รายละเอียด regrid อยู่ 4.3)
met3h = met.resample(time="3H").mean()

# 3) รวมเป็น Dataset เดียว = datacube แล้วเก็บเป็น Zarr
cube = xr.Dataset({"u10": met3h.u10, "v10": met3h.v10, "dem": dem})
cube.chunk({"time":24}).to_zarr("hazenet_cube.zarr", mode="w")

10. Cheat sheet + แบบฝึกหัด

Cheat sheet

  • 🧊 xarray = numpy + ป้ายชื่อ/พิกัด/หน่วย · open_dataset · .sel/.isel/.resample/.interp · ใช้ตลอด
  • 📦 NetCDF (.nc) = กล่องวิทย์ self-describing · รูปของ ERA5 · เปิดด้วย xr.open_dataset
  • 🗺️ GeoTIFF/COG (.tif) = แรสเตอร์ + CRS + geotransform · รูปของ DEM · เปิดด้วย rioxarray
  • ⚡ Zarr = หั่น chunk + บีบอัด · เก็บ datacube · to_zarr/open_zarr · ดีต่อ Colab
  • 🐼 pandas/geopandas = ข้อมูลจุด (FIRMS/Air4Thai) ก่อน rasterize
  • 🦥 dask = lazy + ทีละ chunk → ทำงานก้อนใหญ่กว่าแรมได้ · โหลดจริงตอน .compute()

❓ แบบฝึกหัด

  1. numpy array มีอะไรที่ xarray DataArray มีเพิ่ม 3 อย่าง?
  2. ERA5 มาเป็นฟอร์แมตอะไร? DEM ล่ะ?
  3. ทำไมเก็บ datacube เป็น Zarr ดีกว่า NetCDF ก้อนเดียวสำหรับการเทรนบน Colab?
  4. open_zarr() ไม่โหลดข้อมูลทันที — ข้อมูลถูกโหลดจริงตอนไหน?
  5. FIRMS เป็น CSV จุด — เปิดด้วยอะไร และต้องทำอะไรก่อนเข้า datacube?
เฉลย
  1. ชื่อแกน (dims), พิกัด (coords), และ metadata/หน่วย (attrs) — ทำให้เลือกด้วยชื่อ/ค่าได้ ไม่ต้องจำ index
  2. ERA5 = NetCDF (.nc) · DEM = GeoTIFF/COG (.tif)
  3. Zarr หั่น chunk → อ่านเฉพาะหน้าต่างที่ batch ต้องใช้ + อ่านขนาน + append ได้ → ไม่ต้องยัดทั้งก้อนเข้าแรม (กัน OOM)
  4. ตอนสั่งคำนวณจริง เช่น .compute() / .load() / ตอนเข้าถึงค่า — ก่อนหน้านั้นเป็นแค่ "แผน" (lazy)
  5. เปิดด้วย pandas/geopandas → rasterize (นับ/รวม FRP ลงกริด 0.1°) ให้เป็น DataArray การปล่อย E ก่อน (4.3)

✅ เช็กลิสต์ก่อนไป 4.3

  • ☐ อธิบายได้ว่า xarray ต่างจาก numpy ยังไง (ป้ายชื่อ/พิกัด/metadata)
  • ☐ จับคู่ฟอร์แมต↔แหล่งได้ (ERA5=NetCDF, DEM=GeoTIFF, จุด=CSV/JSON, datacube=Zarr)
  • ☐ ลองเปิด ERA5 ด้วย xr.open_dataset แล้ว .sel ดูค่าที่เชียงใหม่ได้
  • ☐ เข้าใจ lazy/chunk และทำไมจำเป็นบน Colab
