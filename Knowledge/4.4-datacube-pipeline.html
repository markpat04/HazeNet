1. Datacube คืออะไร — กายวิภาคของก้อนสุดท้าย ★

Datacube = xarray Dataset เดียวที่ทุกตัวแปร (channel) อยู่บนแกน (time, lat, lon) เดียวกัน · แบ่ง channel เป็น 3 กลุ่มตามบทบาท (จาก 4.1):

เชื่อมกับ 0.1: สุดท้ายแล้ว datacube คือ tensor ก้อนใหญ่ที่ "มีป้ายชื่อ" · โมเดลกินมันเป็น (batch, channel, lat, lon) ต่อ timestep · ความเข้าใจ tensor (0.1) + xarray (4.2) มาบรรจบตรงนี้

2. สถาปัตยกรรม pipeline — 6 ขั้น A→F

หลักการสำคัญ — ประมวลผล "ทีละก้อนเวลา": อย่าโหลดทุกปีเข้าแรมพร้อมกัน · วน loop ทีละเดือน → ทำ A–B → เขียนต่อ (append) ลง Zarr · นี่คือสาเหตุที่ pipeline ต้องเขียนให้ "ทำทีละชิ้นแล้วต่อกัน" ตั้งแต่ออกแบบ (กัน OOM, โยง 4.5)

3. ขั้น A — ซ้อนทุก channel เป็น Dataset เดียว

หลัง 4.3 ทุกแหล่งเป็น DataArray บนกริดหลักแล้ว · เอามารวมด้วย xr.merge · static (DEM) ต้อง "ขยายซ้ำ" ให้มีแกนเวลาเหมือนตัวอื่น (broadcast):

import xarray as xr
cube = xr.merge([
    u10, v10, pblh, precip, t2m, rh,      # ERA5 (interp แล้ว, 4.3)
    aod,                                   # Himawari (reproject แล้ว)
    emission,                              # FIRMS rasterize → E
    dem_mean, dem_std,                     # DEM aggregate (static)
    pm25,                                  # Air4Thai (target ที่จุด)
])
# static ไม่มีแกน time → broadcast ให้เท่ากันเวลาเข้าโมเดล (หรือใส่ตอน DataLoader ก็ได้)
print(cube)              # เช็ค dims, data_vars, ขนาดให้ครบ
เคล็ด: ตั้งชื่อ channel ให้สื่อความหมาย + ใส่ attrs["units"] ทุกตัว · ตอน debug 3 อาทิตย์ข้างหน้า คุณจะขอบคุณตัวเองที่รู้ว่า precip เป็น m หรือ mm

4. ขั้น B — จัดการ Missing Data ★

4.1/4.3 เตือนแล้วว่า Himawari (กลางคืน/เมฆ) และ FIRMS (เมฆ) มีรู · ปล่อย NaN ไว้เฉย ๆ จะทำให้ loss กลายเป็น NaN แล้วเทรนพัง · ต้องตัดสินใจ 3 อย่าง:

1
สร้าง mask channel: 1=มีข้อมูลจริง, 0=ถูกเติม · เก็บไว้เป็น channel เพื่อให้โมเดล "รู้ว่าตรงไหนเชื่อได้"
2
เติมรู (fill): AOD กลางคืน → เติมด้วยค่ากลางวันล่าสุด / ค่าเฉลี่ย / 0 (แล้วแต่ตัวแปร) · อย่าเติมแบบมั่ว ๆ ให้สอดคล้องฟิสิกส์
3
target ที่หาย: ชั่วโมงที่สถานีไม่มีค่า → ห้ามเติม · ให้คำนวณ loss เฉพาะจุดที่มีค่าจริง (masked loss)
cube["aod_mask"] = cube["aod"].notnull().astype("int8")   # 1=จริง 0=เติม
cube["aod"]      = cube["aod"].ffill("time").fillna(0)         # เติม input
# target: เก็บ NaN ไว้ → ใช้ mask ตอนคิด loss (ไม่เติม!)
caveat ต้องรายงาน: สัดส่วน missing สูงแค่ไหน และเติมด้วยอะไร ต้องเขียนในรายงาน · ถ้า AOD หายไป 50% ตอนกลางคืน แล้วเราเติม 0 — ผลลัพธ์ช่วงกลางคืนจะเชื่อได้น้อยกว่า · mask channel ทำให้ตรวจสอบย้อนได้ และให้โมเดลเรียนรู้ที่จะไม่เชื่อค่าที่เติม

5. ขั้น C — แบ่ง train/val/test ตาม "เวลา" ★ กันโกง

นี่คือจุดที่งาน ML สิ่งแวดล้อมพังบ่อยสุด · ข้อมูลเราเป็น อนุกรมเวลา → ห้ามสุ่มแบ่ง! ต้องแบ่งเป็น บล็อกเวลาต่อเนื่อง (โยง 0.4 และ 5.2):

train = cube.sel(time=slice("2019-12", "2022-04"))
val   = cube.sel(time=slice("2022-12", "2023-04"))
test  = cube.sel(time=slice("2023-12", "2024-04"))
ทำไมสุ่มแบ่งคือโกง: ถ้าสุ่ม ชั่วโมง 14:00 ไปอยู่ train และ 15:00 ของวันเดียวกันไปอยู่ test → โมเดล "เห็นอนาคตใกล้ ๆ" แล้วทำคะแนน test สวยปลอม · พอใช้จริงกับอนาคตที่ไม่เคยเห็นเลยจะแป้ก · แบ่งเป็นบล็อกปี/ฤดู เท่านั้นถึงจะวัด "พยากรณ์อนาคตจริง" ได้ (เต็มเรื่องใน 5.2)

6. ขั้น D — Normalize (กับดัก leakage ที่ซ่อนลึก) ★

channel มีสเกลต่างกันมาก (PBL เป็นพันเมตร, RH เป็น 0–1) · ต้อง normalize ให้อยู่สเกลใกล้กันก่อนเข้าโมเดล (โยง 0.3) · สูตรมาตรฐาน: (x − mean) / std ต่อ channel

กับดักทอง: ต้องคำนวณ mean/std จาก train เท่านั้น แล้วเอาค่าเดียวกันนั้นไปใช้กับ val/test · ห้ามคำนวณสถิติจากทั้งชุด (รวม test) — นั่นคือ data leakage: ข้อมูล test "รั่ว" เข้าสู่การเตรียมข้อมูล ทำให้ผลดีปลอม (โยง 0.4)
import json
# คำนวณสถิติจาก TRAIN เท่านั้น
stats = {v: (float(train[v].mean()), float(train[v].std()))
         for v in ["u10","v10","pblh","t2m","aod","emission","dem_mean"]}
json.dump(stats, open("norm_stats.json","w"))   # เก็บไว้! ต้องใช้ตอน inference ด้วย

def apply_norm(ds):
    for v,(m,s) in stats.items():
        ds[v] = (ds[v] - m) / (s + 1e-6)
    return ds
train, val, test = apply_norm(train), apply_norm(val), apply_norm(test)
3 จุดควรคิด: (1) ตัวแปรเบ้มาก (emission, precip) ลอง log1p ก่อน normalize · (2) target PM2.5 จะ normalize ด้วยหรือไม่ก็ได้ แต่ต้อง "แปลงกลับ" ตอนรายงานผลเป็น μg/m³ · (3) เก็บ norm_stats.json ให้ดี — ตอน deploy/inference ต้องใช้ค่าเดิมเป๊ะ

7. ขั้น E — เก็บเป็น Zarr พร้อม chunk

เขียน 3 ชุดเป็น Zarr · หั่น chunk ให้ตรงกับวิธีอ่านตอนเทรน (จาก 4.2): time เป็นก้อนเล็ก, lat/lon ทั้งภาพในก้อนเดียว:

for name, ds in [("train",train),("val",val),("test",test)]:
    ds.chunk({"time":48, "lat":-1, "lon":-1}) \
      .to_zarr(f"hazenet_{name}.zarr", mode="w")
ทำไมแยก 3 ไฟล์: กันพลาดข้าม split โดยไม่ตั้งใจ + โหลดเฉพาะชุดที่ต้องใช้ · time:48 = 1 chunk ครอบ ~6 วัน (ราย 3 ชม.) พอดีกับการดึง window เทรน · lat/lon:-1 = เก็บทั้งภาพในก้อนเดียว (เพราะโมเดลใช้ทั้งภาพ)

8. ขั้น F — หั่นเป็น "ตัวอย่างเทรน" + DataLoader

โมเดลไม่ได้กินทั้ง datacube ทีเดียว · มันกิน ตัวอย่าง (sample) = หน้าต่างอินพุตอดีต → ทำนายอนาคต · นี่คือที่ที่เชื่อมกับ rollout (3.4):

import torch, xarray as xr
class HazeCube(torch.utils.data.Dataset):
    def __init__(self, path, k=8, H=16):
        self.ds, self.k, self.H = xr.open_zarr(path), k, H   # lazy
        self.ch = ["u10","v10","pblh","t2m","aod","emission","dem_mean","dem_std"]
        self.T = self.ds.sizes["time"]
    def __len__(self): return self.T - self.k - self.H
    def __getitem__(self, i):
        win = self.ds.isel(time=slice(i, i+self.k))            # โหลดเฉพาะ window นี้
        X = win[self.ch].to_array().values                   # (ch, k, lat, lon)
        y = self.ds["pm25"].isel(time=slice(i+self.k, i+self.k+self.H)).values
        return torch.tensor(X).float(), torch.tensor(y).float()

loader = torch.utils.data.DataLoader(HazeCube("hazenet_train.zarr"),
                                     batch_size=4, shuffle=True, num_workers=2)
จุดสวยของดีไซน์นี้: open_zarr เป็น lazy + isel ดึงเฉพาะ window → แต่ละ sample โหลดแค่ไม่กี่ MB ไม่ใช่ทั้ง 10GB · DataLoader หลาย worker ดึงขนานได้ · นี่คือเหตุผลที่เราเก็บ Zarr+chunk มาตั้งแต่ 4.2 (เต็มเรื่อง memory/throughput ใน 4.5)

9. ตรวจสุขภาพ datacube ก่อนเทรน (อย่าข้าม!)

ก่อนเสียเวลา (และงบ A100) เทรน ต้องเช็ค 5 อย่างนี้เสมอ:

เช็คทำยังไงถ้าผิดแปลว่า
ภาพ 1 เฟรมดูสมเหตุผลcube.pm25.isel(time=0).plot()ภาพพลิก/เพี้ยน = lat กลับหัว (4.3)
ช่วงค่าสมจริง.min()/.max() ต่อ channelPM2.5 ติดลบ / PBL = 0 = bug
สัดส่วน NaN.isnull().mean()รูเยอะเกิน → ทบทวนการเติม
แกนตรงกันทุก channelเทียบ lat/lon/timeเหลื่อมช่อง = regrid ไม่เข้า grid เดียว
ไม่มี leakageช่วงเวลา train/val/test ไม่ทับกันทับ = ผลเชื่อไม่ได้
นิสัยที่ดี: เขียน sanity check พวกนี้เป็นฟังก์ชันท้าย build_cube.py ให้รันอัตโนมัติทุกครั้งที่สร้างก้อนใหม่ · เจอ bug ตั้งแต่ตอนสร้าง ดีกว่าไปเจอตอนผลเทรนพิลึกแล้วงงว่าทำไม

10. ทำซ้ำได้ (reproducible) + caveat

กรรมการ/อาจารย์จะถามว่า "สร้างข้อมูลนี้มายังไง" — ต้องตอบได้ด้วย config เดียว + สคริปต์เดียว:

  • config.yaml: โดเมน, ช่วงปี, ความถี่, รายชื่อ channel, ช่วง train/val/test, วิธีเติม missing — ทุกพารามิเตอร์อยู่ในไฟล์เดียว ไม่ฮาร์ดโค้ดกระจาย
  • build_cube.py: อ่าน config → A–E → เซฟ Zarr + norm_stats.json + manifest.json (เวอร์ชันข้อมูล, วันที่ดึง, จำนวนเฟรม)
  • เวอร์ชัน: ตั้งชื่อก้อน hazenet_cube_v1.zarr · เปลี่ยน channel/โดเมน → ขึ้น v2 (อย่าทับของเก่าเงียบ ๆ)
caveat รวมของขั้นข้อมูลทั้งหมด (เขียนในรายงาน):
  1. error สะสม: regrid + เติม missing + normalize ทุกขั้นมี error เล็ก ๆ ที่สะสม → ควร ablation ว่าผลไวต่อ choice แค่ไหน (5.4)
  2. target เบาบาง: PM2.5 มีเฉพาะจุดสถานี — loss คิดได้แค่ไม่กี่พิกเซลต่อเฟรม · ส่วนที่เหลือของภาพโมเดล "เดา" จากฟิสิกส์
  3. ช่วงข้อมูลสั้น: ~5 ฤดู อาจไม่ครอบคลุมปีสุดขั้ว → ระวังการ generalize ไปปีที่อากาศต่างมาก (โยง NeuralGCM: ไม่ extrapolate นอก climate ที่เคยเห็น)

11. Cheat sheet + แบบฝึกหัด

Cheat sheet — pipeline A→F

  • A merge ทุก channel เป็น Dataset (static broadcast)
  • B mask + เติม input · target หาย → masked loss (ห้ามเติม)
  • C แบ่ง train/val/test เป็น บล็อกเวลา (ห้ามสุ่ม)
  • D normalize ด้วยสถิติ จาก train เท่านั้น → เซฟ norm_stats.json
  • E เก็บ Zarr หั่น chunk ตามวิธีอ่าน (time เล็ก, lat/lon เต็ม)
  • F Dataset/DataLoader ดึง window แบบ lazy
  • เสมอ: sanity check + config + manifest (reproducible)

❓ แบบฝึกหัด

  1. DEM เป็น static — ต้องทำอะไรก่อนซ้อนกับ channel ที่มีแกนเวลา?
  2. ทำไม normalize ต้องใช้สถิติจาก train เท่านั้น? เรียกปัญหานี้ว่าอะไร?
  3. ทำไมแบ่ง train/test แบบสุ่มถึงโกงสำหรับข้อมูลอนุกรมเวลา?
  4. target PM2.5 ที่หายไปบางชั่วโมง ควรเติมไหม? จัดการยังไง?
  5. ทำไม DataLoader อ่านจาก Zarr ถึงไม่ทำให้แรมแตกแม้ cube ใหญ่ 10GB?
เฉลย
  1. broadcast = ขยายซ้ำให้มีแกน time เท่าตัวอื่น (หรือใส่เข้าไปตอน DataLoader)
  2. เพราะถ้าใช้สถิติที่รวม test = ข้อมูล test รั่วเข้าการเตรียมข้อมูล → ผลดีปลอม · เรียกว่า data leakage (0.4)
  3. เพราะชั่วโมงข้างเคียงคล้ายกันมาก ถ้ากระจายไปคนละชุด โมเดล "เห็นอนาคตใกล้" → คะแนนสวยปลอม ใช้จริงแป้ก
  4. ห้ามเติม · เก็บ NaN ไว้แล้วคำนวณ loss เฉพาะจุดที่มีค่าจริง (masked loss)
  5. open_zarr เป็น lazy + isel ดึงเฉพาะ window (ไม่กี่ MB) ทีละ sample ผ่าน chunk → ไม่เคยมีทั้งก้อนในแรมพร้อมกัน

  • ☐ mask channel + masked loss สำหรับ target
