5.6 — Statistical Significance: เคลม "ดีกว่า" อย่างมีหลักฐาน 🟡 กลาง

bootstrap CI · paired test · multiple seeds — กันว่าชัยชนะเป็นของจริง ไม่ใช่บังเอิญ

ปิดโมดูล 5 · โยง 5.1 (metric ที่ใส่ CI) · 5.2 (autocorrelation → block bootstrap) · 5.3 (เคลมชนะ baseline) · 5.4 (delta ablation ก็ต้องมีนัยสำคัญ) · 4.5 (seed=งบ A100)

โจทย์: "HazeNet RMSE 11.8 ชนะ ConvLSTM 12.3" — ส่วนต่าง 0.5 นี่ จริง หรือแค่ โชคดีของ test set/seed นี้? · ถ้าตอบไม่ได้ คุณเคลม "ดีกว่า" อย่างซื่อสัตย์ไม่ได้ และ reviewer จะถามทันทีว่า "significant ไหม?"

หัวใจ: ตัวเลขเดียว (RMSE บน test ชุดเดียว เทรนครั้งเดียว) คือ "ตัวอย่างเดียว" จากการกระจายที่มีความบังเอิญปน · เราต้องประเมิน "ขอบเขตความไม่แน่นอน" รอบมัน → รายงานเป็น 11.8 [95% CI: 11.2–12.4] ไม่ใช่ 11.8 เฉย ๆ

2 แหล่งความบังเอิญ: ① test set ที่เราบังเอิญวัด (แก้ด้วย bootstrap) · ② การสุ่มตอนเทรน (seed) (แก้ด้วยหลาย seed)
สารบัญ
1. ทำไมต้องมีนัยสำคัญ ★
2. Confidence Interval — รายงานเป็นช่วง
3. Bootstrap — หา CI โดยไม่ต้องสมมุติ ★
4. Paired test — เทียบแบบจับคู่ ★
5. Block bootstrap — กับ autocorrelation ★
6. Multiple seeds — โชคดีหรือเก่งจริง
7. p-value + effect size (อ่านให้ถูก)
8. งบ A100 + แผนทำจริง

1. ทำไมต้องมีนัยสำคัญ ★

test set ของเราเป็นแค่ "ตัวอย่างหนึ่ง" ของอากาศที่อาจเกิดขึ้น · ถ้าฤดูทดสอบบังเอิญมีวันง่าย ๆ เยอะ โมเดลใด ๆ ก็ดูดีหมด · ส่วนต่างเล็ก ๆ ระหว่าง 2 โมเดลอาจพลิกกลับได้ถ้าเปลี่ยน test ปีอื่น

เชื่อมกับ 5.4: จำได้ไหมว่า ablation delta "+0.3" อาจเป็นแค่ noise? · บทนี้คือเครื่องมือที่ตอบว่า +0.3 นั้นจริงหรือมั่ว · ทุกการเปรียบเทียบในโมดูล 5 (baseline, ablation, oracle) ต้องผ่านบทนี้ก่อนถึงเคลมได้

2. Confidence Interval — รายงานเป็น "ช่วง" ไม่ใช่จุด

95% CI = ช่วงที่เรามั่นใจ ~95% ว่าค่าจริงอยู่ในนั้น · แทนที่จะพูด "RMSE = 11.8" พูดว่า "11.8 [11.2, 12.4]" · ช่วงแคบ = วัดได้แม่น/ข้อมูลเยอะ · ช่วงกว้าง = ไม่แน่นอนสูง

กฎการเทียบด้วยตา (คร่าว ๆ): ถ้า CI ของ 2 โมเดล ไม่ทับกัน → ต่างกันจริงเกือบแน่ · ถ้า ทับกันมาก → อาจไม่ต่าง · ถ้าทับกันนิดหน่อย → ต้องทดสอบให้ชัด (ใช้ paired test, §4) · หมายเหตุ: CI ทับกันเล็กน้อยยัง "ต่างอย่างมีนัยสำคัญ" ได้ ถ้าทดสอบแบบจับคู่ — การดูด้วยตาเป็นแค่ขั้นแรก

3. Bootstrap — หา CI โดยไม่ต้องสมมุติอะไร ★

เรามี test set เดียว แต่อยากรู้ "ถ้าได้ test set อื่นจะต่างแค่ไหน" · bootstrap เลียนแบบสิ่งนี้ด้วยการ สุ่มหยิบซ้ำ (resample with replacement) จาก test ที่มี หลายพันครั้ง → คำนวณ metric ทุกครั้ง → ได้ "การกระจายของ metric" → เอา percentile 2.5/97.5 เป็น CI

import numpy as np
def bootstrap_ci(err, n=2000):       # err = array ของ error ราย sample
    stats = [np.sqrt(np.mean(np.random.choice(err, len(err))**2)) for _ in range(n)]
    return np.percentile(stats, [2.5, 97.5])
ทำไม bootstrap คือเครื่องมือหลัก: ใช้ได้กับ metric อะไรก็ได้ (RMSE, CSI, onset-lead จาก 5.1) โดย ไม่ต้องสมมุติว่าข้อมูลกระจายแบบ normal · และ ไม่ต้องเทรนใหม่ — แค่คำนวณบนผลทำนายที่มีอยู่ → ฟรีในแง่งบ A100 · เป็นวิธีที่เหมาะกับ metric แปลก ๆ อย่าง onset-lead ที่ไม่มีสูตร CI สำเร็จรูป

4. Paired Test — เทียบแบบ "จับคู่" ถูกกว่าและไวกว่า ★

2 โมเดลทำนาย วัน/สถานีเดียวกัน → อย่าเทียบค่าเฉลี่ยแยกกัน · ให้เทียบ ผลต่างราย sample (errorA − errorB ที่จุดเดียวกัน) · เพราะ "วันที่ยากก็ยากสำหรับทั้งคู่" → การจับคู่ หักความแปรปรวนร่วมนั้นทิ้ง → เห็นความต่างจริงชัดขึ้นมาก

เครื่องมือ paired ที่ใช้บ่อย:

  • Paired bootstrap on the difference: resample sample แล้วคิด mean(errorA−errorB) → CI ของ "ผลต่าง" · ถ้า CI ไม่คร่อม 0 → ต่างอย่างมีนัยสำคัญ
  • Permutation test: สุ่มสลับว่าโมเดลไหนได้ค่าไหน หลายพันครั้ง → ดูว่าความต่างขนาดนี้เกิดโดยบังเอิญบ่อยแค่ไหน = p-value
  • Wilcoxon signed-rank: paired test แบบ non-parametric (ไม่สมมุติ normal)
ทำไม paired ดีกว่า unpaired มาก: ความแปรปรวนส่วนใหญ่มาจาก "วันยาก vs วันง่าย" ซึ่งกระทบทั้ง 2 โมเดลเท่า ๆ กัน · พอจับคู่แล้วลบกัน ความแปรปรวนนั้นหายไป เหลือแค่ "ความต่างจริงระหว่างโมเดล" → ทดสอบ มีพลัง (power) สูงกว่า มาก จับความต่างเล็ก ๆ ที่จริงได้

5. Block Bootstrap — แก้ปัญหา autocorrelation (สำคัญต่อข้อมูลเรา) ★

กับดักใหญ่: bootstrap ธรรมดาสมมุติว่า sample อิสระต่อกัน · แต่ข้อมูลเรา autocorrelated (ชั่วโมงติดกัน/สถานีใกล้กันคล้ายกัน — Tobler, 5.2) · ถ้า resample ราย "ชั่วโมง" → จะนับข้อมูลซ้ำเสมือนมีข้อมูลอิสระเยอะกว่าจริง → CI แคบเกินจริง = มั่นใจเกินเหตุ = เคลมนัยสำคัญที่ไม่มีจริง

ทางแก้ — block bootstrap: resample เป็น "ก้อน" ที่อิสระต่อกัน ไม่ใช่รายจุด:

  • resample ทั้ง episode (ช่วงฝุ่นพีคทั้งก้อน) แทนรายชั่วโมง
  • resample ทั้ง สถานี แทนรายพิกเซล
  • resample ทั้ง บล็อกเวลา (เช่นรายสัปดาห์)
หลักเดียวกับ 5.2: เหมือน CV ที่ต้อง "บล็อก" เพื่อกันรั่ว — bootstrap ก็ต้อง "บล็อก" เพื่อไม่ให้ประเมินความมั่นใจสูงเกิน · ทั้งคู่มาจากความจริงเดียว: ข้อมูล geospatial ไม่อิสระ · นี่คือจุดที่งาน AQ จำนวนมากทำพลาด แล้วเคลมนัยสำคัญปลอม

6. Multiple Seeds — แยก "โชคดี" ออกจาก "เก่งจริง"

bootstrap จัดการความบังเอิญจาก test set · แต่ยังมีความบังเอิญจาก การเทรน (สุ่ม init, สุ่มลำดับ batch) · โมเดลเดียวกันเทรน 2 ครั้งด้วย seed ต่างกันได้ผลไม่เท่ากัน

ทางแก้: เทรนหลายครั้งด้วย seed ต่างกัน → รายงาน mean ± std ข้าม seed · ถ้า "ชัยชนะ" ของเราอยู่ภายในความผันผวนของ seed → อาจเป็นแค่ seed ดวงดี ไม่ใช่ไอเดียดี

มาตรฐานทอง vs ความจริงงบเรา: วงการ ML ถือว่า "รายงาน ≥3–5 seeds" คือมาตรฐาน · แต่ แต่ละ seed = เทรนใหม่ = งบ A100 (โยง 4.5/5.2) · ด้วย ~15 ชม. เราทำหลาย seed สำหรับทุก variant ไม่ได้ · ทางออกซื่อสัตย์: ทำ bootstrap (ฟรี) ให้ครบทุกผล + ทำ multi-seed (2–3) เฉพาะ "ผลพาดหัว" (HazeNet vs baseline หลัก) + ระบุชัดว่าความไม่แน่นอนจากการเทรนวัดบนกี่ seed

7. p-value + Effect Size — อ่านให้ถูก

คำหมายความว่าไม่ได้หมายความว่า
p < 0.05ความต่างขนาดนี้เกิดโดยบังเอิญ <5%ไม่ได้แปลว่า "ต่างเยอะ" หรือ "สำคัญเชิงปฏิบัติ"
Effect sizeต่างกัน ขนาดเท่าไร จริง ๆ (เช่น ลด RMSE 4 หน่วย)—
2 กับดักที่ต้องเลี่ยง:
① significant แต่จิ๋ว: ด้วยข้อมูลเยอะ ความต่าง 0.05 μg/m³ อาจ "significant" แต่ ไร้ความหมายเชิงปฏิบัติ · ต้องรายงาน effect size คู่เสมอ — "ต่างอย่างมีนัยสำคัญ และ มากพอจะสำคัญจริง"
② p-hacking: ลองหลาย metric/threshold/subset แล้วรายงานเฉพาะอันที่ p<0.05 = โกง · ตั้ง metric/test ที่จะดูไว้ก่อน แล้วถ้าทดสอบหลายอย่าง ต้องปรับ (เช่น Bonferroni)

8. งบ A100 + แผนทำจริง

เครื่องมือจัดการความบังเอิญจากต้นทุนทำแค่ไหน
Bootstrap CI (block)test setฟรี (ไม่เทรนใหม่)ทุกผล

สรุปแผน: ใช้ block bootstrap + paired test (ฟรีทั้งคู่) เป็นกระดูกสันหลังของการรายงานนัยสำคัญทุกตัวเลข · เก็บ multi-seed ไว้สำหรับ 1–2 การเปรียบเทียบที่สำคัญที่สุด · รายงานทุกตัวเลขเป็น "ค่า [95% CI]" + ระบุว่าใช้ block อะไร, n เท่าไร, กี่ seed — ความโปร่งใสนี้คือสิ่งที่กรรมการมองหา

9. Cheat sheet + แบบฝึกหัด

Cheat sheet

  • ตัวเลขเดียว = ตัวอย่างเดียว · รายงานเป็น ค่า [95% CI] เสมอ
  • Bootstrap = สุ่มหยิบซ้ำ → การกระจายของ metric → percentile CI · ใช้กับ metric อะไรก็ได้ · ฟรี
  • Paired test = เทียบผลต่างราย sample (วันเดียวกัน) → ไวกว่ามาก · CI ของผลต่างไม่คร่อม 0 = ชนะจริง
  • Block bootstrap = resample เป็นก้อน (episode/สถานี) ไม่ใช่รายชั่วโมง — กัน CI แคบเกินจริง (autocorrelation, 5.2)
  • Multiple seeds = แยกโชคดีจากเก่งจริง · แพง → เฉพาะผลพาดหัว
  • p-value ≠ effect size · รายงานคู่กัน · อย่า p-hack

❓ แบบฝึกหัด

  1. ทำไม "RMSE 11.8 ชนะ 12.3" ยังเคลมไม่ได้ถ้าไม่มี CI?
  2. bootstrap หา CI ได้ยังไงโดยไม่ต้องสมมุติการกระจาย?
  3. paired test ดีกว่า unpaired ตรงไหน?
  4. ทำไมข้อมูลเราต้องใช้ block bootstrap ไม่ใช่ bootstrap ธรรมดา?
  5. "significant" แต่ effect size จิ๋ว ควรเคลมว่าดีกว่าไหม?
เฉลย
  1. 0.5 อาจเป็นแค่โชคของ test set/seed นี้ — ถ้า CI ทับกันมากก็อาจไม่ต่างจริง · ต้องวัดความไม่แน่นอนก่อน
  2. สุ่มหยิบซ้ำจาก test หลายพันครั้ง คำนวณ metric ทุกครั้ง → ได้การกระจายจากข้อมูลเอง → เอา percentile 2.5/97.5
  3. หักความแปรปรวนร่วม ("วันยากก็ยากทั้งคู่") ทิ้ง → เหลือความต่างจริงระหว่างโมเดล → power สูงกว่า
  4. ข้อมูล autocorrelated → resample รายชั่วโมงนับซ้ำเสมือนอิสระเยอะเกินจริง → CI แคบเกิน → เคลมนัยสำคัญปลอม · block แก้ได้
  5. ไม่ควรเคลมว่า "ดีกว่าอย่างมีความหมาย" — significant ≠ สำคัญเชิงปฏิบัติ · ต้องดู effect size ด้วย

✅ เช็กลิสต์ปิดโมดูล 5

  • ☐ ทุกตัวเลข metric (5.1) รายงานพร้อม 95% CI (block bootstrap)
  • ☐ ทุกการเคลม "ชนะ" (5.3) ผ่าน paired test (CI ผลต่างไม่คร่อม 0)
  • ☐ delta ของ ablation (5.4) ทดสอบนัยสำคัญแล้ว
  • ☐ ผลพาดหัวมี multi-seed (≥2–3) + ระบุจำนวน seed
  • ☐ รายงาน effect size คู่ p-value · ไม่ p-hack
